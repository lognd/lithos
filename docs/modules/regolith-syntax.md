# regolith-syntax

Lexing/parsing and the extension-string registry (AD-3, AD-14): source
text -> tokens -> a layout pass -> a lossless rowan CST -> a typed AST,
plus the one canonical formatter and the one registry of source-file
extensions. This crate owns L0/L1 only (tokens through syntax-adjacent
static checks); semantic analysis lives in `regolith-sem`. Regolith
reference: `docs/spec/regolith/08` (lexical/layout grammar),
`docs/spec/hematite/02`, `docs/spec/cuprite/07`.

This doc covers one section per source module.

## Module sections

<a id="token"></a>
### `token`

The raw lexical tokens, defined as a `logos` DFA (AD-3): the lexer
*specification* whose patterns define the terminal set. Regolith
reference: `docs/spec/regolith/08` (L0), `docs/spec/hematite/02`,
`docs/spec/cuprite/07`. The lexer emits full-fidelity tokens including
whitespace and comments (rowan needs every byte); tabs in indentation
are an E01xx error (ASCII source, spaces only). The keyword set grows
with the grammar (WO-05).

<a id="layout"></a>
### `layout`

The layout pass: converts leading-whitespace structure into
INDENT/DEDENT/NEWLINE tokens so the parser stays context-free (AD-3,
Python-style off-side rule). Regolith reference: `docs/spec/regolith/08`.
Indentation is spaces only; a tab in indentation is an E01xx
diagnostic (WO-06). Blank lines and comment-only lines emit no layout
tokens.

<a id="syntax-kind"></a>
### `syntax_kind`

`SyntaxKind`: the tag on every rowan CST node and token (AD-3). One
flat `repr(u16)` enum covering terminals (post-layout tokens and
keywords) and non-terminals (grammar nodes), rust-analyzer style.
Regolith reference: `docs/spec/regolith/08`, `docs/spec/hematite/02`,
`docs/spec/cuprite/07`. Adding a construct is: a `SyntaxKind`, a parser
production, a typed AST view, and a grammar.ebnf rule -- all four in
the same change (NO DUPLICATION).

<a id="cst"></a>
### `cst`

The rowan language binding: lossless red/green CST over `SyntaxKind`
(AD-3). Every source byte lives in the tree, so the formatter, precise
spans, and error-resilient trees come for free. This module is the
regolith language marker rowan's generic tree machinery is
parameterized over -- no logic beyond the binding itself.

<a id="parser"></a>
### `parser`

The hand-written, event-based recursive-descent parser with Pratt
expressions and layout-anchored error recovery (AD-3). Regolith
reference: `docs/spec/regolith/08`, `docs/spec/hematite/02`,
`docs/spec/cuprite/07`, and `examples/` (the concrete target corpus).
The parser emits events a builder folds into a rowan tree; error
recovery syncs on INDENT/DEDENT so one bad statement never eats the
whole file (diagnostics stay batch-emitted, regolith/09 sec. 4).

<a id="ast"></a>
### `ast`

Typed AST: zero-cost view structs over the CST (rust-analyzer pattern,
AD-3). In the built system these are GENERATED by an xtask from an
ungrammar file; the hand-written views here are the scaffold seed and
document the shape the codegen must produce. Regolith reference:
`docs/spec/hematite/02`, `docs/spec/cuprite/07`. Each view wraps a
`SyntaxNode` of a specific `SyntaxKind` and exposes typed child
accessors; trivia is skipped at this layer.

`PowerDecl::bus_items`/`load_items` (WO-133 deliverable 2, coordinator
adjudication F-WO133-1, `docs/spec/cuprite/10-power-layer.md` sec. 3a)
parse the `buses:`/`loads:` fields' OPTIONAL per-item properties
(`mainbus(nominal_voltage=480V, phases=3)`) into `PowerListItem`s via
`parse_power_list_items`, a small hand-rolled paren-depth text scan
over the field's lossless value text -- additive, and deliberately
NOT a re-entry into the shared statement/expression grammar (zero
blast radius on any other field/claim shape).

<a id="checks"></a>
### `checks`

L1 static checks: syntax-adjacent semantic rejections that run once
per parse, over the freshly built CST (AD-7: diagnostics are data,
never `Err`). Regolith reference: `docs/spec/regolith/02-quantity-core.md`
sec. 1/3 (dimensional arithmetic, `[a, b]` vs `[i .. j]`) and WO-05
acceptance: dimension-mismatch diagnostics
(`INCOMPATIBLE_QUANTITIES`), `==` on a continuous quantity
(`EQUALITY_ON_CONTINUOUS`), and interval-form checks.

<a id="formatter"></a>
### `formatter`

The canonical formatter: the rowan-based normalizer (AD-3; WO-05
acceptance). Walks the lossless CST token stream and re-emits it with
canonical intra-line spacing, preserving line structure and leading
indentation verbatim, so the transform is meaning-preserving
(`parse(format(x))` matches `parse(x)`'s token stream and tree shape)
and idempotent (`format(format(x)) == format(x)`). Regolith reference:
`docs/spec/hematite/04` (canonical forms). One normalizer; the CLI
`fmt` and the golden pipeline both call it.

<a id="extension"></a>
### `extension`

The ONE registry of source-file extensions and their languages. Ground
rule 6 / AD-14: extension strings live here and nowhere else; every
other layer (including Python, via the FFI) asks this module.
Extensions follow the first-four-letters-of-the-mineral rule (D108):
`.hema` (hematite), `.cupr` (cuprite), `.fluo` (fluorite, D93),
`.calx` (calcite, D133 -- a deliberate four-letter-rule exception,
since calcite's pre-cycle-26 draft used the now-dead `.calc`).

<a id="debug"></a>
### `debug`

Pipeline inspection dumps backing `regolith debug tokens|cst|ast`
(AD-13 / DX contract 5: intermediate states are always inspectable).
Plain-text, deterministic renderings of each stage for goldens and
human debugging; stdout is data (these strings), logs go to stderr.

<a id="walk"></a>
### `walk`

Walk bodies: the elaborated form of the WO-05 opaque profile islands
(WO-11 grammar half). A walk is a pen-path sketch: segments, joins,
holes, regions, constraints, and exports. Regolith reference:
`docs/spec/hematite/02` sec. 5. The constraint vocabulary is the
closed SolveSpace-equivalent set (hematite/07 OPEN-5, D65); no solving
happens here -- this module records structure for the static ledger
(`regolith-sem` `profile`); direction words are recorded as uniqueness
hints only, checked at solve time.
