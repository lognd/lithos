## B. Decisions

**D102 (F103 cut 1, F106). Temporal claim forms are two normative
families.**
- REDUCTIONS -- `peak(expr[, within d after e])`, `rms(expr,
  band=[lo,hi])`, `overshoot(expr[, window])` -- yield a scalar and
  REQUIRE an external comparator: `ClaimForm::{Peak, Rms, Overshoot}`
  gain `op: String, rhs: String` (the `Comparison` shape). A
  reduction with no trailing comparator is a compile diagnostic.
  `peak` remains sense-driven (worst adverse excursion, D93).
- CONTAINMENTS -- `settles(expr, to=<tol>, within d after e)`,
  `stays_within(expr, mask=<ref>[, window])` -- are self-contained
  assertions: their parameters ARE the acceptance; a trailing
  comparator is a compile diagnostic. Their `ClaimForm` variants
  carry typed parameter fields and no op/rhs.
Schema fields ride WO-30's single bump (F100 discipline).

**D103 (F103 cut 2). General comparison claims + expression givens.**
A require-line claim may be `lhs_expr <op> rhs_expr` with the
comparator mid-expression (the Kestrel link budget). Lowering rule:
exactly ONE top-level comparator per claim line (more or zero at top
level = compile diagnostic); each side is an ordinary quantity
expression. Every entity-field reference term (`comms.pa_out`,
`gs_uhf437.sensitivity`) lowers to a GIVEN resolved through the
entity DB with its own `Cause` (INV-21): `Given` gains
`refs: Vec<(String, String)>` (reference path -> resolved value
source text; schema field in the WO-30 bump). dB legality is already
L1 (log-unit views); this closes the last gap between the link
budget claim and the registered `link_budget` pack.

**D104 (F103 cut 3). Contract-IR field identity is the NAME.**
Interface/impl promised bounds match by field name (the WO-12 IR
already keys fields by name at the source level; conformance_windows
stops taking the FIRST comparator-bound field per side and builds a
name-keyed map, erroring constructively on a promised name with no
matching demand side). No new identity scheme -- names are already
unique per interface (existing L1 duplicate check).

**D105 (F103 cuts 4-5). The three API surfaces + sweep-domain claim
lines.**
- (a) Sweep-domain claim lines: `forall <var> in [lo,hi]:` prefixes
  on a require line lower into the EXISTING `Obligation.sweep`
  (SweepDomain) -- the grammar surface exposes what the schema
  already carries; discrete `{a, b}` domains ride the same path
  (D93/D95 alignment). This unblocks the buck efficiency pack.
- (b) Reduced-tier numeric base class (`harness/numeric.py`):
  subclass provides `evaluate_point(inputs: Mapping[str, float]) ->
  float` + declared per-input monotonicity (or none); the base owns
  corner enumeration / grid sweep, eps declaration, D95 coverage
  axes, and the one margin rule via the existing `Model.discharge`.
  First customer: the lumped-thermal reference pack (WO-26 list).
- (c) Planner-model evidence: a plan artifact is a `plan`-kind
  payload on the D96 channel (content-addressed), its lockfile row
  carries `cause: planner(<what>)` -- NO new evidence shape
  (regolith/07 sec. 6 "plan = evidence" now has its transport).
- (d) INV-12 match-set growth: lockfile waiver rows gain
  `match_set: Vec<String>` (sorted entity refs at authorship);
  `regolith build` diffs prior-lockfile match sets and reports any
  GROWTH as a named diagnostic requiring re-acknowledgment (the
  waiver ladder's anti-rot residual). Schema field in the WO-30
  bump; the diff pass is WO-26 remainder implementation.

## E. Same-day addendum 2: F108/D109, the firmware realizer

**F108. The compute track verifies firmware but nothing GENERATES
its design-determined layer.** cuprite/05 sec. 4 already treats the
image as a realized, measured artifact (fit/stack/WCET/boot claims,
`partitions:` maps, `extern` linking) -- the verification half
exists. But the facts the toolchain OWNS after binding (pin-mux
assignments with lockfile causes, peripheral allocations from flow
demands, clock constraints, interrupt-capable-pin facts from
component records, declared events) are re-derived BY HAND in user
firmware (`#define LED_PIN 13`, hand-written init code) -- the exact
staleness failure G42 named for wire lengths: nothing invalidates
the hand copy when the design moves. Owner ask (2026-07-07): a code
artifact generating the C/assembly layer + extern links for
cross-language consumers.

**D109. A firmware realizer (WO-37) generates the DERIVED code
layer; it decides nothing and never generates application logic.**
The regolith/07 sec. 6 backend rule applied to code: generated
firmware SERIALIZES pinned decisions (the planner already made
them), exactly as gerbers serialize the pinned layout.

- **Generates** (v1): the hardware contract header (symbolic
  constants for every pin/net/peripheral assignment, provenance
  comments carrying the lockfile hash -- application code references
  the design symbolically, so a re-plan BREAKS COMPILATION instead
  of silently misbehaving); pin configuration + peripheral init from
  the pin-mux/binding lockfile rows; clock tree setup from the
  declared constraints; ISR vector stubs typed from the event ledger
  (`on <event>` handlers, WO-36's typed surface); the linker memory
  map from the image's declared `partitions:`; a build-system
  fragment. Everything content-addressed and byte-identical from the
  same lockfile.
- **Never generates**: application/control logic (user code, or a
  future `by spec` synthesis question -- existing deferral, not this
  surface); RTOS choice or scheduling policy (the schedule is
  CLAIMED and verified, cuprite/05, not synthesized here).
- **Cross-language rule**: C is the lingua-franca artifact with an
  `extern "C"` contract; other languages consume GENERATED bindings
  derived from the one contract header (Rust `-sys`-shaped first,
  per demand) -- one source of truth, the feldspar OPEN-3 precedent.
- **Vendor mapping is pack content**: abstract peripheral config ->
  vendor HAL/register idiom comes from signed MCU-family packs
  (registry records + templates, trust tiers apply), same shape as
  process capability tables; regolith core stays vendor-neutral.
- **The loop closes on existing machinery**: the image built from
  generated BSP + user code re-enters through `image`/`extern`
  hash-pinning, and the EXISTING fit/stack/WCET/boot claims verify
  it. No new claim vocabulary, no new trust surface.
