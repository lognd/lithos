# Design cycle 22 -- 2026-07-07 (the developer-tooling surface: LSP, editor, lints, docs)

Mandate (owner, 2026-07-07): design everything a regular language
ships for its developers -- a language server, editor integration
with syntax highlighting (VS Code first), linting, and the rest of
the standard toolchain -- as zero-shot work orders. Numbering
continues (F109.., D110..).

## A. Findings

**F109. The tooling inventory: half already exists, by design.**
Present today: a canonicalizing formatter (`regolith fmt`, proptested
idempotent); one diagnostic renderer with codes, spans, severities
(Error/Warning), and machine-applicable fixes; `regolith debug
tokens|cst|ast|ir` pipeline inspection; a lossless rowan CST chosen
EXPLICITLY for "formatter + future-LSP requirements" (AD-3, the
rust-analyzer architecture); L1 name resolution; per-crate purity
that makes the whole front end embeddable (AD-2 "future-LSP/wasm
reusable"); a 3-OS release CI matrix. Missing: the server itself, an
editor extension, advisory (style) lints, doc generation, project
scaffolding, and watch mode.

**F110. The fix seam was pre-built.** `regolith-diag::Fix.replacement`
is documented as "the edit the LSP applies" -- quick fixes are
already data; the server only has to forward them.

**F111. The classic integration bug is named in advance.** regolith
spans are byte offsets; LSP positions are UTF-16 code units in
line/column form. Every LSP integration that ignores this corrupts
ranges on the first non-ASCII character. The repo is ASCII-only, but
USER designs are not guaranteed to be -- the converter must be
correct, not skipped (WO-38 carries a dedicated test).

## B. Decisions

**D110. The language server is a Rust crate (`regolith-ls`) reusing
the compiler crates in-process.** `lsp-server` + `lsp-types`
(rust-analyzer lineage: synchronous, no async runtime) over stdio.
It depends on `regolith-api` and below, NEVER on `regolith-py`, and
is NOT part of the wheel (AD-2's one-wheel promise stays lean);
distribution is per-platform binaries from the existing release
matrix, bundled by the editor extension. Incrementality v1:
reparse-on-change of the edited file (the lossless CST is fast at
corpus scale; SLO: syntax-tier diagnostics < 100ms on the largest
corpus file) plus a debounced workspace lower for semantic tiers.
Salsa-style incremental queries are DEFERRED with a reopen
criterion: the SLO breaks on a real workspace 10x the corpus.

**D111. One diagnostics pipeline; build-tier facts come from
artifacts, not embedded Python.** The server publishes the SAME
`regolith-diag` diagnostics (code, severity, spans, related, fixes
-> LSP quick fixes) that the CLI renders -- no second lint engine,
ever. Facts only the ORCHESTRATOR knows (obligation status, margins,
evidence tier, discharging model, resolved values with causes)
surface in hovers/inlays by READING the schema-versioned build
artifacts (`.regolith/` evidence cache, lockfile, serialized
BuildPayload), refreshed on file events. The server never imports or
spawns Python for its own features; the editor's "build" command
shells to the `regolith` CLI like any terminal.

**D112. Advisory lints are compiler passes in the existing code
registry, configured in `quarry.toml`.** A new Lint code family
(Warning severity by default) joins `regolith-diag`; a `[lints]`
table in `quarry.toml` maps code -> allow|warn|deny (deny promotes
to Error). The physical-claims waive ladder is NOT involved -- style
lints are configuration, not engineering deviations (different
audit surfaces, deliberately). v1 lint set: unused declaration,
unreferenced feature, shadowed name, unused import, dead-name
vocabulary (retired keywords/extensions in prose positions where
detectable), and a todo!/assume! inventory surfaced as advisory
counts. One renderer, one publisher (D111).

**D113 (AD-24). One front end for humans and tools: editor grammar
artifacts are GENERATED from the compiler's own tables.** The
TextMate grammar's keyword/token classes are emitted from
`regolith-syntax`'s lexer tables by a build step and drift-checked
in CI -- an independently-maintained grammar is the two-copies bug
in editor form. Semantic tokens from the server are the accurate
layer on top. Consequences: tree-sitter is DEFERRED (a second
grammar to desync; reopen on concrete demand from an editor that
cannot consume LSP semantic tokens), wasm playground DEFERRED
(reopen: first outreach/docs-site need), DAP-style debugging is NOT
a goal -- the declarative analog is explain/trace (hover + `regolith
debug` + the harness "what would resolve it" diagnostics), already
owned elsewhere. Recorded as AD-24 in `00-architecture.md`.

**D114. The editor extension is `lithos` for VS Code, living
in-monorepo at `editors/vscode/`** (rust-analyzer precedent),
TypeScript, excluded from the wheel: language definitions for
`.hema`/`.cupr`/`.fluo`, the generated TextMate grammars, language
configuration (comments, brackets, indentation), snippets, the LSP
client + bundled server binaries, commands (check/build/fmt/rules
test), a problem matcher, and a status item showing build/evidence
state from the artifacts. Marketplace publishing is owner-gated;
the WO ships a CI-built `.vsix`.

**D115. Docs generation: `regolith doc`, markdown v1.** Renders a
package's public surface -- interfaces (roles, params, flow decls),
parts/blocks/flownets, claims with their obligation status and
margins WHEN build artifacts are present, registry records -- from
the typed CST plus leading `#` comment blocks attached to
declarations (no new doc-comment syntax in v1; the existing comment
convention is the doc text). HTML/site rendering deferred (reopen:
lodestone registry web UI work).

**D116. Scaffolding + watch mode round out the CLI.** `quarry new
<name> --template mech|elec|fluid|system` emits a working project
(quarry.toml, example source with one honest claim, .gitignore, CI
snippet); `regolith check --watch` (watchfiles) re-runs check on
save with the one renderer. Both Python CLI, small.
