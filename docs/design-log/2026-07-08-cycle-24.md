## D128. Realized-domain artifacts are first-class IRs; extraction runs in-pipeline

Owner direction, quoted in substance: there are to be intermediate
representations IN THE LANGUAGE for geometry, PCB layout, and so
forth -- accepting that this updates a large part of the project.

**The decision.** regolith/08's L4 ("REALIZED IR") stops being a
conceptual level realized only by opaque native artifacts (STEP
files, `.kicad_pcb`) held orchestrator-side. Every realized-domain
artifact that any pipeline pass, rule pack, or solver pack reads
becomes a first-class intermediate representation of the toolchain,
exactly like the contract IR (L2) and obligations (L5) already are:

- Rust-sourced, schemars-derived schema in `regolith-oblig` (AD-5);
- content-addressed through the one encoder (AD-18);
- a payload kind on the D96 generalized ref channel and a
  `BuildPayload`/store citizen (WO-30, AD-22);
- inspectable (`regolith debug ir` surfaces realized IRs supplied to
  a build).

The initial IR set: **RealizedGeometry** (mech realized geometry;
WO-22's Python forward contract promoted per the AD-22 rule; payload
kind `geometry.realized`, already in the D96 vocabulary) and
**RealizedLayout** (elec placed/routed board content from WO-24;
NEW payload kind `layout.realized` -- coordinate the kind-vocabulary
addition with feldspar OPEN-2, which pins that list). "And so forth"
is the RULE, not an enumeration: any future realized-domain content
(routed harness geometry rides RealizedGeometry; firmware map data
per regolith/08 L4 is future work) enters the same way. Native files
remain pinned side artifacts and evidence; the semantic content the
toolchain consumes is the IR, parsed from the native form EXACTLY
ONCE, producer-side, by the realizer adapter.

**Consequence for WO-32 (the escalated question).** Permanent
GeomExtract deferral is REJECTED. fluorite/03 sec. 1's ratified
wording -- "Extraction is part of lowering" -- stands literally:
`regolith-lower::extract` runs in-pipeline, over RealizedGeometry
bytes supplied to `lower()` as compile inputs. `EdgeParams::
GeomExtract` survives only as the PRE-REALIZATION placeholder: an
edge whose implementing part has no realized geometry IR yet lowers
with the selector intact and its dependent obligations stay honestly
indeterminate, naming the missing IR; it is never a discharge-time
mechanism.

**AD-17 is preserved, not amended.** The purity rule forbids IO in
the pipeline; it never forbade realized content AS AN INPUT (WO-32
deliverable 2 already said "the record CONTENT is an input"). The
orchestrator resolves digests against the WO-30 store and passes
bytes in; `lower()` stays a pure function -- now of (sources,
realized-IR inputs) instead of sources alone.

**The staged build loop.** In-pipeline extraction over
realizer-produced IRs makes the build loop explicitly staged: lower
-> realize (producing new IRs) -> re-lower with them, to a fixed
point. Termination is by content addressing: unchanged IR inputs
produce a byte-identical re-lower (INV-10), so the loop closes the
first time realization adds nothing new. The orchestrator (which
already owns build tiers and the lazy loop) owns this loop; every
iteration is logged.

**Anti-staleness falls out.** Extracted values are baked into
payloads cited to the IR digest they came from, so a geometry change
changes the payload digest and invalidates exactly the dependent
obligations -- the property G42 demanded for wire runs (WO-34) now
holds for every extraction consumer by construction.

Normative home: AD-25 (`../implementation/00-architecture.md`).
Machinery: NEW WO-42 (realized-domain IRs: schemas, realizer
promotion, the realized-input channel, the staged loop). WO-32
D4a/D4b are UNBLOCKED (see D129 and the WO-32 body's updated
ledger); WO-22/24/25/34 gained amendment notes pointing at WO-42.

## D129. `Obligation.payloads`: the general payload-ref channel on obligations (schema 9 -> 10)

Approved by the owner for the WO-32 D4 resume (recorded here so the
approval is a ledger fact, not a memory-file fact): `Obligation`
gains `payloads: Vec<PayloadRef>` -- a GENERAL vector channel, not a
single-purpose flownet slot -- so obligations can carry
content-addressed refs of any D96 kind (`flownet` first; `field`,
`geometry.realized`, `plan` are known future riders). One
`SCHEMA_VERSION` bump 9 -> 10, `make schema` regeneration, golden
corpus re-keyed (the version folds into every content address;
counts and diagnostics unchanged) -- the same mechanics as every
prior bump.

This closes the second D4 recon blocker: fluorite `require fluids.*`
claims lower to obligations carrying a `kind: flownet` `PayloadRef`
(WO-32 D4a), and `BuildPayload.flownets` emission plus the first
orchestrator `PayloadStore.put` producer (WO-32 D4b) has a channel
to reference.
