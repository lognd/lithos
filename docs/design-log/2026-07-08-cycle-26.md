## D132. quarry and lodestone are RETIRED; the package manager is **magnetite**

**The problem.** Two proper names (quarry the tool, lodestone the
registry) for what is operationally one subsystem: the package
manager's client module already owns fetch, index, trust, vendoring,
manifests, and scaffolding in one place (`python/regolith/quarry/`).
Two brands for one seam is naming surface without technical payoff,
and both names sat outside the "tools live under the regolith
umbrella" rule the owner has now made explicit.

**The decision (owner, 2026-07-08).**

- The package manager is **magnetite**. Poetically exact: lodestone
  *is* naturally magnetized magnetite -- the old registry name
  collapses into the new tool name by mineralogy. It attracts
  packages. One geology theme preserved.
- Manifest file: `magnetite.toml` (was `quarry.toml`).
- Python module: `regolith.magnetite` (was `regolith.quarry`).
- CLI: the `regolith` binary stays the ONE entry point; the
  subcommand group renames to `regolith magnetite ...`
  (e.g. `regolith magnetite new`).
- The registry service has NO proper name: it is "the magnetite
  registry." Everything D77 decided about it (sparse index,
  content-addressed archives, signing-carries-trust, yank-not-delete,
  vendoring) stands unchanged -- this is a rename, not a redesign.
- `regolith.lock` is unchanged (it was never quarry-branded).

**Sweep rules** (same discipline as the D81 rename): code, living
docs, examples, tests, and templates are swept in one change;
`docs/design-log/` stays verbatim (history); `quarry`/`lodestone`
join the retired-names list (hematite/04 sec. 4 registry + the
CLAUDE.md names table). Trust-key prose ("quarry trust key
designations") becomes "magnetite trust key designations" -- the
mechanism (INV-14, INV-22, INV-28) is untouched; no schema or
evidence-key content changes, so goldens must NOT drift (the word
"quarry" never appears in a hash input -- verified during the sweep).

Normative home: this entry; regolith/11 (rewritten in the same
change); the names tables in CLAUDE.md, TODO.md START HERE, and
implementation/README ground rule 6.

## D133. **calcite** is the civil/architectural track (name revived, chartered)

**The decision (owner, 2026-07-08).** The fourth language track is
civil engineering / architecture / building planning, named
**calcite** -- the primary mineral of limestone and the calcined
heart of cement: the mineral you build buildings out of. Files:
`.calx`.

**On the revival of a dead name.** `calcite`/`.calx` was the fluid
track's DRAFT name, killed at ratification (D93) in favor of
fluorite. The draft usage never shipped in any surface, extension
registry, or example; the two usages never coexist. The retired-list
entry is amended: calcite-the-fluid-draft stays dead; calcite-the-
civil-track is a fresh assignment by the same owner who killed the
old one. Grep hygiene: any pre-cycle-26 doc phrase "calcite (dead)"
refers to the fluid draft.

**v1 scope (decided now, in the charter).** The charter is
`docs/calcite/01-charter.md`, written this cycle; it decides the
construct set, the vocabulary direction, the claim families, the
lowering shape, and the non-goals, so the spec-elaboration WO is
zero-shot. Headline: spaces + structural frame + envelope as the
three artifact families; loads and code compliance as PACK CONTENT
(load-combination packs, building-code rule packs on the WO-28
engine -- building codes are DFM-rules-shaped); structural sizing
claims discharge through the same obligation pipeline (feldspar is
the natural FEA/solver home); MEP is explicitly cross-track
composition (fluorite for hydronics/plumbing, cuprite for power),
not new calcite surface.

Machinery: WO-46 (spec elaboration from the charter, fluorite
ratification pattern) -> WO-47 (front end, WO-31 pattern) -> WO-48
(lowering + reference code packs, WO-32 pattern).

## D134. Plugin architecture v1 (AD-26): one entry-point seam, typed kinds

**The problem.** Four plugin-shaped seams grew independently, each
sound, none unified: solver/model packs (WO-20,
`regolith.model_packs` entry points), rule packs (WO-28, in-language
+ engine), MCU family packs (WO-37, `realizer/firmware/packs.py`
seam), and manufacturing backends (WO-25, `regolith.backends`
framework). A fifth (external solver distributions -- feldspar)
rides WO-20's seam. Every future extension re-derives discovery,
identity, versioning, and trust from scratch.

**The decision.** ONE discovery seam, generalizing WO-20's proven
discipline (AD-26 added to 00-architecture.md in this change):

- One entry-point group: `regolith.plugins`. Each entry point yields
  a typed `PluginManifest` (pydantic, frozen): `id` (unique,
  loud-duplicate-rejected), `kind` (closed enum v1: `model_pack`,
  `rule_pack`, `mcu_pack`, `backend`), `version` (folded into
  evidence/cache keys exactly as WO-20 folds pack versions today),
  and the kind-specific registration callable.
- `regolith.model_packs` migrates to `regolith.plugins` with
  `kind=model_pack`; feldspar's entry point moves in the same
  release. Nothing is published, so no compatibility alias is kept.
- Composition stays deterministic (sorted by id), duplicates stay
  loud (`BuildReport.pack_errors` generalizes to `plugin_errors`),
  trust stays signature-carried (INV-14/INV-28: a plugin's evidence
  is signed by ITS key; installation confers nothing).
- CLI: `regolith plugin list` (id, kind, version, source
  distribution).
- Explicit v1 NON-GOAL with reopen criterion: language tracks are
  NOT plugins -- grammars compile into `regolith-syntax` (AD-3, one
  parser stack). Reopen only if a third party actually attempts an
  out-of-tree track and the one-front-end rule (AD-24) can be
  preserved.

Machinery: WO-44. Normative home: AD-26 + this entry.

## D135. The standard library charter: `std.` is reserved, tracks ship batteries

**The decision.** The stdlib is PACKAGES, not compiler surface --
regolith/11's stance stands; this decision fills in the catalog and
the governance:

- The `std.` name prefix is RESERVED in the magnetite namespace;
  only the lithos project publishes under it.
- Development home: `stdlib/` at the repo root, one magnetite
  package per entry, each with its own manifest and record/model/
  rule content; published like any package; the in-repo copies are
  what CI and the corpus consume (no phantom `process=` refs).
- v1 catalog (decided): `std.units` (already effectively built in --
  stays in `regolith-qty`, documented as the one non-package stdlib
  member); `std.mech` (material records, fastener/bearing records,
  weld models -- D65/OPEN-13 content; absorbs `std.mech.weld`);
  `std.sheet_metal` (the WO-28 reference DFM pack, already named);
  `std.elec` (passive/connector records, `jlc_2l` DRC reference
  pack, link/buck harness-model registrations); `std.fluid` (media
  property tables, fitting/loss-coefficient records). Deferred until
  calcite lands: `std.civil` (load combinations, occupancy/egress
  tables, reference code packs -- named in the calcite charter).
- Evidence tiers are honest: transcribed-not-verified records say
  `tier=community` in-file (the registry-records precedent, D58).

Machinery: WO-45 (packaging + the mech/elec/fluid content that
already exists in-tree), WO-28 remainder (rule-pack content), WO-48
(std.civil). Normative home: regolith/11 sec. 8 (rewritten this
cycle).

## D136. The pipeline CLI is completed: `regolith build [--release]`

**The problem.** WO-25's close-out named it plainly: the whole
staged pipeline exists as a Python API (`orchestrator.staged_build`
/ `release_gate`) but no CLI verb drives it, so the canonical demo
(`regolith build --release && regolith ship`) cannot be typed. The
north-star loop ("declarative file in, working artifact out") must
be drivable from the shell.

**The decision.** A `regolith build` verb: T-tier selection
(`--tier`, default per regolith/09), `--release` running the INV-24
release gate, stdout = data (report path/JSON), logs to stderr,
exit code = gate verdict. `regolith ship` keeps its own verb and
consumes the build's outputs (the WO-25 input triple). No new
semantics -- purely surfacing the existing API.

Machinery: WO-43. Normative home: regolith/09 (a short CLI
subsection added when it lands) + this entry.

## D137. Documentation debt is paid by DELETION, not archiving

**The decision.** Git history is the archive; the tree carries only
living text. Applied this cycle:

- `docs/README.md`: the per-cycle revision log (400+ lines
  duplicating design-log content) is replaced by a pointer to
  `docs/design-log/`; the doc holds only the current map and
  reading order.
- `docs/audit/` is DELETED. Every FE-*/BE-* finding is fixed with
  regression tests (TRIAGE.md's own final state); the findings'
  content lives in git history and in the tests that pin them.
- `docs/implementation/design/harness-phase-c.md` is DELETED: a
  status snapshot fully superseded by the code, its tests, and the
  WO-20/26 files.
- `TODO.md` is REWRITTEN as a live queue only: START HERE, the
  dispatch queue, and the remaining-work ledger. Completed-cycle
  narratives and long-done checklists go; the design logs already
  hold every ledger verbatim.
- Feldspar `TODO.md` likewise condenses to current state + pointers
  (M1/WO-01..11 are done; the file was a pure done-ledger).
- Track docs, invariants, architecture, design charters, and
  work-order files are NOT thinned -- they are normative, not
  narrative.

Rule going forward: a status snapshot may live in TODO.md or a WO's
Progress section, never in a standalone doc.
