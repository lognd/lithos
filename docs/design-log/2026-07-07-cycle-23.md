## A. Findings

**F112. The compiler's own pressure inventory lags its consumers'.**
Our golden corpus registers THREE entries (cubesat + two single
files) while feldspar's demand corpus has grown to six projects /
40+ files (dune buggy alone: 27 files, four language tracks). The
stress signal that found G34-G43 -- and calcite/fluorite itself --

**F113. The HDL-superset claim exists but is imprecise and
untested.** cuprite/03's banner is "the HDL superset with continuous
quantities"; nothing says WHICH features of WHICH HDLs, in what
sense "superset" holds, or how it is checked. An unfalsifiable
superset claim is marketing, not spec.

**D120 (F113). The HDL claim, made precise: semantic superset of the
synthesizable design subset; embedding superset of the rest; never a
syntactic superset.** Recorded normatively in
`docs/cuprite/09-hdl-coverage.md` (cuprite 0.12):

1. For the SYNTHESIZABLE DESIGN SUBSET of Verilog-2005, VHDL-2008,
   SystemVerilog-2017, and Veryl, cuprite provides a native semantic
   equivalent for every feature class (the matrix in 09 is the
   normative inventory: modules/entities, hierarchy, parameters/
   generics/generate, processes/always blocks under the
   event-bounded semantics, FSMs via modes, interfaces/modports,
   packages, clock/reset abstraction, typed signals) -- usually
   STRONGER (units on every quantity, static CDC ledgers, contracts
   instead of bare modports).
2. VERIFICATION-LANGUAGE features (SVA properties/sequences,
   covergroups, UVM class machinery, constrained randomization) are
   deliberately NOT reproduced: they map onto the claim/obligation/
   evidence machinery (temporal claim vocabulary, coverage encoding,
   value-source intervals + planner allocation). The matrix records
   each mapping; where the mapping is partial (deep SVA sequence
   algebra), the row says PARTIAL and names the extern route.
3. Anything else crosses via `by extern(ref, format)` -- Verilog
   transparent (elaborated), VHDL/SV opaque-with-contracts in v1,
   Veryl by embedding its emitted SystemVerilog. So the superset
   claim is: NATIVE for the design subset, EMBEDDING for the rest,
   with equivalence obligations either way.
4. TESTABILITY: `examples/hdl/` carries one fixture pair per matrix
   row (the HDL source embedded via extern + the native cuprite
   equivalent, with an equivalence obligation binding them). A
   matrix row with neither a native mechanism nor an extern fixture
   is a FINDING by definition (the G34 catalog-coverage rule applied
   to HDLs). Cuprite never parses HDL text as cuprite -- the
   syntactic-superset reading is explicitly disclaimed.

## D. Handoff addendum (pre-cutoff): the findings-promotion protocol

The four corpus agents (D121/D122/D123) report candidate findings
their READMEs record. Promotion into this ledger is now
RULE-GOVERNED so it needs no design authority:

**D124. Promotion rules for candidate findings.**
1. A finding that only RECORDS DEMAND for something already decided
   or work-ordered (e.g. "needed WO-33 computed fields here",
   "EXPECT-TODO maps to the D117 overbroad-waiver lint") is folded
   by the integrating agent: append it verbatim (with project + file
   attribution) to a "Promoted findings" section at the end of this
   file, and add a one-line demand note to the owning WO's file if
   one exists. Mechanical; no authority needed.
2. A finding that a spec is AMBIGUOUS between readings, or that a
   construct is MISSING with no owning WO/deferral, is a DESIGN
   QUESTION: the integrating agent appends it to the same section
   marked "QUEUED (needs design cycle)" and does NOT resolve it,
   work it around, or edit any spec. The next design cycle (owner or
   a Fable instance) opens with that queue.
3. A finding that a fixture could not be written because the AGENT
   misread a spec is neither: fix the fixture, note nothing.
The integration recipe itself (apply diff -> `git mv` the project
into `examples/systems/` [hdl/ and tracks/fluorite/ and negative/
stay where the agents put them relative to the NEW layout; projects
authored pre-reorg arrive at old top-level paths] -> hand-merge the
two corpus-dict lines to the systems/ paths -> regenerate goldens
via the documented env vars -> `make check` -> one commit per
project -> promote findings per the rules above) is mechanical end
to end.

## E. Addendum (2026-07-08, owner-authorized correction): WO-29 Q4(a) named the wrong opaque form

**D125. `../implementation/design/23-lowering-output-surface.md` Q4(a)
corrected: the promoted opaque form is `then:` claim-scope feature
constructors, not `parts:` per-line orbit constructors.**

The WO-29 dispatch that decided Q4 (D91, cycle 19) named `parts:`
per-line orbit constructors (`n x Thing(...)`) as the promotion
needed to materialize `Hole`/`Bend` entities for deliverable 2. This
does not match the fixtures: every `parts:` orbit line in the corpus
(`examples/systems/cubesat/structure.hema`'s `rails: 4 x Rail`,
`panels: 4 x SidePanel`; `examples/tracks/hematite/gear_reducer.hema`'s
`parts:` block) instantiates sub-parts or sub-assemblies, never hole
or bend geometry. The actual feature constructors deliverable 2 needs
(`Bore(...)`, `CBore(...)`, `Pierce(...)`, `PatternOf<Boss(...)>`,
etc.) all appear inside `then:` claim scopes
(`examples/tracks/hematite/pillow_block.hema`,
`gear_reducer.hema`, `sheet_bracket.hema`, `weldment_frame.hema`,
`torch_igniter.hema`), which `crates/regolith-lower/src/claims.rs`
already partially walks (per claim LINE, not per constructor call)
-- `crates/regolith-lower/src/entities.rs::build_entities` never
descends into any nested block body, `then:` included, today.

WHY this matters beyond a doc typo: acting on the uncorrected Q4(a)
would have spent the parser-promotion budget on the wrong CST
production -- a new `parts:` per-line node that deliverable 2 does
not actually consume -- while leaving the real wall (`then:`
claim-scope constructor calls) undiscovered until deliverable 2's
implementation hit it directly, the same rediscovery tax Q4 was
written to avoid for WO-23/WO-28.

Correction (Q4(a) in design/23, this cycle): the promoted form is
`then:` claim-scope constructor calls. `parts:` orbit lines stay
opaque under this WO (they were never actually blocking anything);
if a consumer later needs them, that is a new, separately scoped
promotion. Also decided as a corollary: deliverables 2/3/5 share ONE
claim-scope walk (NO DUPLICATION, AD-17) rather than growing three
independent traversals of the same CST; see design/23 Q4(a)
corollary for the split (5's `connect` endpoints are a different CST
production, not this walk, but stay in the same module).

No other Q1-Q5 decision is reopened; scope held to this single
factual mismatch per owner authorization.

## G. Addendum (2026-07-08, owner-authorized): WO-29 D4 source construct named + D4 closed

**D126. The binding-requirement bridge (WO-29 deliverable 4) projects
from architecture-resource `promises:` arguments, NOT `budget:`
statements.**

WO-29's own D4 cut note (cycle 19) left the demand vocabulary open:
Q3/D90 decided the Rust/Python split but never named WHICH source
construct a block's `min_capabilities` demand projects from. The
prior agent's nearest structured surface, `ContractGraph::budgets`,
was flagged as possibly wrong because `budget:` reads as a CEILING
the block must not exceed, not a demand ON a supplied component. This
addendum resolves it against the specs.

Resolution: the construct is the `promises:` keyword argument on a
`resources:`/`memories:`/`peripherals:` entry inside an
`architecture for <Computer>:` declaration.

- `docs/cuprite/05-computer-track.md` sec. 2: "Execution resources are
  abstract blocks with promises -- `executor`, `memory`, `mover`,
  `fabric` are stdlib block contracts", each entry a
  `<name>: <contract>(promises: <bound>...)` form
  (`cpu0: executor(promises: >= 4Mops f32 sustained,
  context_switch <= 5us)`). The section's "peripheral matching -- the
  demand vector against candidate resource bundles" IS WO-24's
  component-binding screen; the promise bound is what a supplied part
  must satisfy.
- `docs/regolith/10-domain-binding.md` sec. 1 confirms the vocabulary
  split: the `budget` row ("error budget, power budget, timing
  budget, noise budget" -- closure-arithmetic ceilings) is a DISTINCT
  regolith concept from the `interface promises` row ("drive strength,
  timing, power, protocol conformance") and the `interface demands`
  row ("max load capacitance, edge rate, impedance"). A block's
  capability demand on a supplied component is the promises/demands
  vocabulary, never the budget row. So `budget:` was indeed the wrong
  surface, exactly as the cut note suspected.

WHY the prior agent could not find it structured: the demand lived
inside a call's argument list as a comparison-bearing keyword argument
(`promises: >= 20Mops f32 sustained`), a shape the parser had no
grammar for -- it bailed to an `OpaqueIsland` right after the
`promises` token, leaving the call's `RParen` unclosed. That was a
parser gap, not a missing construct.

D4 as built this cycle (branch `wo29-d4`):
1. Parser promotion: keyword arguments `name: <value>` inside a
   `CallExpr` `ArgList` now structure into a typed `KeywordArg` CST
   node (grammar.ebnf + fuzz seed in lockstep); the kestrel.cupr CST
   golden regenerated, zero new diagnostics.
2. Rust (`regolith-lower::block_requirement`): each architecture
   resource block's `promises:` bound projects into a schema-versioned
   `regolith_ir::BlockRequirement { owner, block, contract, demands }`
   `BuildPayload` field. RAW demands only (spelled comparator + value
   text), mirroring deliverable 3's raw-text discipline; `SCHEMA_VERSION`
   bumped 7 -> 8 (one bump), `make schema` regenerated. Golden corpus
   re-keyed (schema-version folds into every content address; counts
   and diagnostics unchanged).
3. Python (`regolith.realizer.elec.bridge`): screens the raw demands
   into the numeric WO-24 `BlockRequirement` and derives
   `ComponentCandidate`s from quarry `RecordStore` records (typed
   `Record.capabilities` slice added). D90 split honored: Rust emits
   raw demands, Python screens. Scope kept honest -- only `>=`/`>`
   demands (the candidate>=minimum direction WO-24's `_satisfies`
   models) become minimums; `<=`/`==`/`<` ceiling demands are logged
   and skipped, not force-fit.

Scope held to the D4 question under owner authorization; no other
WO-29 decision reopened. Deliverable 4 CLOSES; with D3/D5 already on
master, WO-29 is DONE.
