# Design cycle 23 -- 2026-07-07 (the stress corpus + the HDL coverage claim)

Mandate (owner, 2026-07-07): mirror the feldspar-side demand corpus
on our side -- many more golden files including fluorite, at least
one dune-buggy-scale project (~20 files), plus other large projects
that push the language constructs to the ABSOLUTE limit; and make
the "cuprite is a superset of HDLs (VHDL / Verilog / SystemVerilog /
Veryl)" claim assertable. Numbering continues (F112.., D119..).

## A. Findings

**F112. The compiler's own pressure inventory lags its consumers'.**
Our golden corpus registers THREE entries (cubesat + two single
files) while feldspar's demand corpus has grown to six projects /
40+ files (dune buggy alone: 27 files, four language tracks). The
stress signal that found G34-G43 -- and calcite/fluorite itself --
came from THEIR side. The language repo should be generating that
signal, not just consuming it.

**F113. The HDL-superset claim exists but is imprecise and
untested.** cuprite/03's banner is "the HDL superset with continuous
quantities"; nothing says WHICH features of WHICH HDLs, in what
sense "superset" holds, or how it is checked. An unfalsifiable
superset claim is marketing, not spec.

## B. Decisions

**D119. The stress-corpus program: three large projects + per-file
golden discipline.** Modeled on feldspar's examples/lithos ledger
convention (per-project README: file map, per-file "pressure
applied" column, candidate-findings ledger):

- `examples/cnc_router/` (~20 files, the dune-buggy-scale flagship;
  EVERY construct family): 3-axis CNC router -- generic axis module
  instantiated 3x (monomorphization under real load), machined
  plates with deep stage/setup pipelines, positioning-error budget
  spanning leadscrew/coupling/frame stiffness (tolerance chains +
  budgets), bed-size variants, profile walks (kinematics), spindle
  thermal zones, motion-planner workloads (WCET/schedule), a
  purchased spindle entering as a sealed brownfield import, a
  SUPPLIED G-code plan, deliberate expert-ladder usage (locks,
  hints, one waiver -- the D117 lint fixtures in the wild), wiring
  `harness:` prose (WO-34-shaped), and a flood-coolant `.fluo`
  circuit.
- `examples/espresso_machine/` (~14 files, fluorite-first): dual
  boiler + steam + water circuits (three flownets), thermostat and
  three-way-valve line-ups, pump as imposer, thermosiphon loop,
  boiler zone temperatures as computed-field demand, scale/flow
  degradation claims, cuprite controller commanding valves through
  the one event ledger, pressure-vessel mech claims fed by fluid
  peaks (the cross-track promise chain in both directions).
- `examples/sdr_transceiver/` (~13 files, cuprite/computer to the

  domains + CDC ledger pressure, DDR-class timing budgets on

  core with equivalence obligations, mode statecharts, ADC noise
  floors, power/noise budgets, pin-mux contention by construction.

Rules: every `.hema`/`.cupr` file is PARSE-CLEAN (a poisoned
subject, INV-20, is corpus rot unless it is a deliberate negative
fixture); claim forms the harness cannot yet lower are EXPECTED and
land in the deferral golden -- that is what it is for (the honest
gap inventory, refreshed by real demand). `.fluo` files are spec
pressure tests until WO-31 (the extension registry makes them
invisible to `check`; each project README says so). Golden + deferral
corpus entries are added per project; goldens regenerate ONLY via
the documented update env var and are diff-reviewed. Project
findings are recorded as CANDIDATE entries in the project README and
promoted to this ledger by the coordinating design cycle -- corpus
authors never edit the design log.

**D120 (F113). The HDL claim, made precise: semantic superset of the
synthesizable design subset; embedding superset of the rest; never a
syntactic superset.** Recorded normatively in
`docs/cuprite/09-hdl-coverage.md` (cuprite 0.12):

1. For the SYNTHESIZABLE DESIGN SUBSET of Verilog-2005, VHDL-2008,
   SystemVerilog-2017, and Veryl, cuprite provides a native semantic
   equivalent for every feature class (the matrix in 09 is the
   normative inventory: modules/entities, hierarchy, parameters/
   generics/generate, processes/always blocks under the
   event-bounded semantics, FSMs via modes, interfaces/modports,
   packages, clock/reset abstraction, typed signals) -- usually
   STRONGER (units on every quantity, static CDC ledgers, contracts
   instead of bare modports).
2. VERIFICATION-LANGUAGE features (SVA properties/sequences,
   covergroups, UVM class machinery, constrained randomization) are
   deliberately NOT reproduced: they map onto the claim/obligation/
   evidence machinery (temporal claim vocabulary, coverage encoding,
   value-source intervals + planner allocation). The matrix records
   each mapping; where the mapping is partial (deep SVA sequence
   algebra), the row says PARTIAL and names the extern route.
3. Anything else crosses via `by extern(ref, format)` -- Verilog
   transparent (elaborated), VHDL/SV opaque-with-contracts in v1,
   Veryl by embedding its emitted SystemVerilog. So the superset
   claim is: NATIVE for the design subset, EMBEDDING for the rest,
   with equivalence obligations either way.
4. TESTABILITY: `examples/hdl/` carries one fixture pair per matrix
   row (the HDL source embedded via extern + the native cuprite
   equivalent, with an equivalence obligation binding them). A
   matrix row with neither a native mechanism nor an extern fixture
   is a FINDING by definition (the G34 catalog-coverage rule applied
   to HDLs). Cuprite never parses HDL text as cuprite -- the
   syntactic-superset reading is explicitly disclaimed.

**D121. Corpus authoring is dispatched; the matrix and this ledger
are not.** Three parallel worktree agents author the projects (and
the `examples/hdl/` fixtures ride the sdr agent); the coordinator
owns the HDL matrix, finding promotion, golden diff review, and
integration -- the same coordinator/author split as the WO waves.

**D123. `examples/negative/`: the rule-breaking corpus, graded and
self-calibrating.** Each file breaks EXACTLY ONE rule and declares
it in a header the driver parses:

```
# BREAKS: <one-line rule statement, citing the spec section>
# EXPECT: E0104            (code(s) the compiler MUST emit today)
# EXPECT-TODO: INV-4       (known-uncaught: driver xfails -- this IS
#                           the demand signal for a lint/check)
# EXPECT-WO31              (.fluo: skipped until the extension lands)
```

`tests/golden/test_negative_corpus.py` walks the directory, asserts
every EXPECT code present in that file's diagnostics (and, for
single-subject files, that clean sibling subjects still lower --
INV-20 gating is itself under test), xfails EXPECT-TODO, skips
EXPECT-WO31. Gradation: OBVIOUS (parse poison, unknown unit, 1V+1A,
dBm+dBm, == on continuous, duplicate names) -> MEDIUM (sibling-scope
refer/INV-6, over-match, generic arity/dead-generic E0504/E0503,
budget overflow E0432, capability E0410, boundary subsumption E0407,
single-driver, supply short, unjoined terminal, conformance
narrower-than-promise) -> HIDDEN (asymmetric-givens verify-one
INV-4, post-commit mutation INV-5, region overlap INV-23,
uncancelled subtracted dB reference, overbroad waiver, silent
shadowing, uncontracted sealed import -- several of these are
EXPECTED to be EXPECT-TODO today; each maps to a WO-40/D117 lint or
an invariant, and an EXPECT-TODO with no owning WO is a finding).
The corpus is SELF-CALIBRATING: the author verifies every EXPECT
against actual compiler output; expected-but-not-emitted flips to
EXPECT-TODO plus a candidate finding. The negative corpus is NOT in
`_CORPUS` (its driver is its own gate).

## D. Handoff addendum (pre-cutoff): the findings-promotion protocol

The four corpus agents (D121/D122/D123) report candidate findings
their READMEs record. Promotion into this ledger is now
RULE-GOVERNED so it needs no design authority:

**D124. Promotion rules for candidate findings.**
1. A finding that only RECORDS DEMAND for something already decided
   or work-ordered (e.g. "needed WO-33 computed fields here",
   "EXPECT-TODO maps to the D117 overbroad-waiver lint") is folded
   by the integrating agent: append it verbatim (with project + file
   attribution) to a "Promoted findings" section at the end of this
   file, and add a one-line demand note to the owning WO's file if
   one exists. Mechanical; no authority needed.
2. A finding that a spec is AMBIGUOUS between readings, or that a
   construct is MISSING with no owning WO/deferral, is a DESIGN
   QUESTION: the integrating agent appends it to the same section
   marked "QUEUED (needs design cycle)" and does NOT resolve it,
   work it around, or edit any spec. The next design cycle (owner or
   a Fable instance) opens with that queue.
3. A finding that a fixture could not be written because the AGENT
   misread a spec is neither: fix the fixture, note nothing.
The integration recipe itself (apply diff -> `git mv` the project
into `examples/systems/` [hdl/ and tracks/fluorite/ and negative/
stay where the agents put them relative to the NEW layout; projects
authored pre-reorg arrive at old top-level paths] -> hand-merge the
two corpus-dict lines to the systems/ paths -> regenerate goldens
via the documented env vars -> `make check` -> one commit per
project -> promote findings per the rules above) is mechanical end
to end.
