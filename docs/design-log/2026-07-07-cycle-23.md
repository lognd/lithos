# Design cycle 23 -- 2026-07-07 (the stress corpus + the HDL coverage claim)

Mandate (owner, 2026-07-07): mirror the feldspar-side demand corpus
on our side -- many more golden files including fluorite, at least
one dune-buggy-scale project (~20 files), plus other large projects
that push the language constructs to the ABSOLUTE limit; and make
the "cuprite is a superset of HDLs (VHDL / Verilog / SystemVerilog /
Veryl)" claim assertable. Numbering continues (F112.., D119..).

## A. Findings

**F112. The compiler's own pressure inventory lags its consumers'.**
Our golden corpus registers THREE entries (cubesat + two single
files) while feldspar's demand corpus has grown to six projects /
40+ files (dune buggy alone: 27 files, four language tracks). The
stress signal that found G34-G43 -- and calcite/fluorite itself --
came from THEIR side. The language repo should be generating that
signal, not just consuming it.

**F113. The HDL-superset claim exists but is imprecise and
untested.** cuprite/03's banner is "the HDL superset with continuous
quantities"; nothing says WHICH features of WHICH HDLs, in what
sense "superset" holds, or how it is checked. An unfalsifiable
superset claim is marketing, not spec.

**D120 (F113). The HDL claim, made precise: semantic superset of the
synthesizable design subset; embedding superset of the rest; never a
syntactic superset.** Recorded normatively in
`docs/cuprite/09-hdl-coverage.md` (cuprite 0.12):

1. For the SYNTHESIZABLE DESIGN SUBSET of Verilog-2005, VHDL-2008,
   SystemVerilog-2017, and Veryl, cuprite provides a native semantic
   equivalent for every feature class (the matrix in 09 is the
   normative inventory: modules/entities, hierarchy, parameters/
   generics/generate, processes/always blocks under the
   event-bounded semantics, FSMs via modes, interfaces/modports,
   packages, clock/reset abstraction, typed signals) -- usually
   STRONGER (units on every quantity, static CDC ledgers, contracts
   instead of bare modports).
2. VERIFICATION-LANGUAGE features (SVA properties/sequences,
   covergroups, UVM class machinery, constrained randomization) are
   deliberately NOT reproduced: they map onto the claim/obligation/
   evidence machinery (temporal claim vocabulary, coverage encoding,
   value-source intervals + planner allocation). The matrix records
   each mapping; where the mapping is partial (deep SVA sequence
   algebra), the row says PARTIAL and names the extern route.
3. Anything else crosses via `by extern(ref, format)` -- Verilog
   transparent (elaborated), VHDL/SV opaque-with-contracts in v1,
   Veryl by embedding its emitted SystemVerilog. So the superset
   claim is: NATIVE for the design subset, EMBEDDING for the rest,
   with equivalence obligations either way.
4. TESTABILITY: `examples/hdl/` carries one fixture pair per matrix
   row (the HDL source embedded via extern + the native cuprite
   equivalent, with an equivalence obligation binding them). A
   matrix row with neither a native mechanism nor an extern fixture
   is a FINDING by definition (the G34 catalog-coverage rule applied
   to HDLs). Cuprite never parses HDL text as cuprite -- the
   syntactic-superset reading is explicitly disclaimed.

**D121. Corpus authoring is dispatched; the matrix and this ledger
are not.** Three parallel worktree agents author the projects (and
the `examples/hdl/` fixtures ride the sdr agent); the coordinator
owns the HDL matrix, finding promotion, golden diff review, and
integration -- the same coordinator/author split as the WO waves.
