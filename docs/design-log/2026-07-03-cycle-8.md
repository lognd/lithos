Mandate (user directive): fresh read, resolve ALL open questions, make
the specifications powerful. Method: cold front-to-back read of the
whole tree (substrate 01-13, mech 01-07, elec 01-08), then close every

## A. Decisions

**D62. D54 settled: artifact-typed caller params** (substrate `04`
sec. 1). Settled on D54's own criterion, cheaply: the corpus already
held the second organic use (F88) -- `PayloadPcb<bits: image>` beside
`ObcPcb<fw: image>`, two boards, two image kinds (ELF firmware, FPGA
bitstream), one shape. The alternative (a board importing the
integrating system's artifact) inverts the dependency direction that
contract-first decomposition relies on -- the board's file would
depend on its integrator. Discipline recorded so the feature stays
small: an artifact-typed param is **injection, not templating** --
whole-artifact reference, permanent borrow, the callee may consume
contract + measured DB + content hash and can never modify or
introspect construction. No new machinery.
- *Unambiguous:* binding is by name at instantiation, monomorphic.
- *Intent-based:* "this board carries that image" is intent; an
  upward import would be mechanism leaking into layering.

**D63. OPEN-2 closed: allocation policies are pack-provided budget
math** (mech `03` sec. 5) -- the D49 pattern re-applied. `allocate:`
already takes a named policy; std ships `worst_case` and `rss`;
`statistical(cpk=1.33)` is one pack away. The real dependency is
distribution-valued process capability data, which is ordinary
evidence-tiered registry content: a community-tier Cpk table refusing
a `trust: >= certified` claim group is the system working. The
"reserve a syntax slot" leaning dissolved -- the slot was `allocate:`
all along.

**D64. OPEN-3 closed for v1: kinematics is model packs, not syntax**
(mech `07` sec. 3). The constraint that held it open ("the syntax must
not need to change") is dischargeable by inspection: mechanism
coordinates = config variables + `exposing`; transmission laws =
`couples:`; mobility = the DOF ledger; quantification over motion =
`forall <cfg>`; range/collision/clearance-over-sweep = claim forms +
sweep-capable harness models (L5 content). A motion collision check is
`mech.clearance(a, b) > 2mm forall pivot.theta` today; what is missing
is the model pack that discharges it, and model packs are never spec.
Reopen only on a failing *syntax* example.

**D65. OPEN-5 closed in shape: the profile constraint vocabulary is a
closed v1 set** (mech `02` sec. 5) -- the SolveSpace constraint kinds
(coincident, distance, angle, radius/diameter, tangent, perpendicular,
parallel, horizontal/vertical in the profile frame, equal, symmetric,
midpoint, on_entity). Rationale: the solver target implements exactly
this set, it is the de facto sketch-constraint standard, and the
walk/constraints/exports surface plus the DOF ledger were already
settled. Solver *interaction* (parameterization, convergence
diagnostics) is implementation-owned: WO-11 now, Phase C later.

**D66. OPEN-6 closed: re-import needed no mechanism** (mech `07`
sec. 3). Trace the existing machinery: content drift fails the pin
loudly (INV-22); re-pinning is explicit; queries re-resolve against
the new measured entity DB; T2 conformance re-runs; a binding that no
longer resolves is the ordinary constructive query error with
near-miss listings. The one genuinely new artifact is the re-import
*diff report* (per query: same/moved/gone; per impl: T2 delta) --
`check --explain` rendering of information the build already computes.
Tooling, Phase C.

**D67. OPEN-8 closed: surface state joins the contact-record
coherence key** (mech `07` sec. 3). `contact { A, B }` records may
carry `surface:` qualifiers; resolution picks the unique most-specific
record (both surfaces > one > bare pair) per the trait-coherence
rulebook -- the machinery built for material-class specificity,
unchanged. The finishing stage supplies each side's surface state
(precedent: `sn_curve(..., surface=per_finishing_stage)`).
`lubrication:` keeps selecting fields within the resolved record.

**D68. OPEN-11 closed: `refining` legality is per-(op,
geometry-class), declared in process modules** (mech `07` sec. 3) --
the standing leaning, promoted. It is a capability-table fact checked
at L3; a wrong declaration is a pack bug with provenance (E06xx names
the pack), the same trust posture as every capability table. The
alternative (per-op only) was rejected because it is false to reality:
Ream refines cylindrical geometry and nothing else.

**D69. EOPEN-5 closed: the lazy loop is pre-layout only; layout is
realize-once-verify** (elec `06` sec. 3). Post-layout failures blame
back to pre-layout decisions through ordinary conflict-driven search
(blame sets already cross the realization boundary). Incremental
re-route is an orchestrator optimization deferred with a reopen
criterion: a real design whose post-layout failure no pre-layout
variable can close.
- *User-friendly note:* this keeps `optimize` latency bounded by
  construction -- the expensive realizer runs once per candidate,
  never inside an inner loop.

**D70. EOPEN-6 closed: no workload sublanguage -- a consequence of
D60** (elec `05` sec. 4, `01` sec. 4). A workload body expressive
enough to compile to firmware or RTL is a program, and D60 already
decided programs do not live in design source (local verifiability is
the charter). Deeper content enters the one door everything foreign
enters: `by extern` against a contract, with evidence. The HLS
temptation is thereby answered *permanently for design source*, not
deferred -- one rationale closing two questions is the sign it is the
right rationale.

**D71. EOPEN-8 closed: modes ARE config variables** (elec `08`
sec. 2). Exposers (connection/block/system) + `forall`/`during` +
mode-entry events + sequencing masks cover every observed need; the
two examples most likely to break it (motor drive, Kestrel `op`) did
not. A first-class statechart would add transition-graph semantics --
a second behavioral formalism -- for zero demonstrated demand.
Reopen criterion: a design needing transition-graph *verification*
(unreachable mode, illegal transition sequence) inexpressible as
claims over config domains and events.

**D72. EOPEN-9 closed: conformance depth is pack content, not
language** (elec `08` sec. 2). Structural conformance (roles + timing
templates) is v1. Transaction-level conformance already has a
mechanism: a protocol pack that ships a behavioral `spec:` makes it an
ordinary T3 equivalence obligation with declared coverage. v1 packs do
not ship those specs; `assume!` carries the gap visibly. Nothing was
open except model availability.

**D73. EOPEN-10 closed: HIL results are `by test(ref)`** -- the same
evidence class as mech lab reports and first articles. An HIL run
discharges exactly the obligations whose claims it observed, keyed by
content address. There never was a co-verification "boundary"; now the
ledger says so.

**D74. EOPEN-11 closed: the EMC posture is the honest-deferral
machinery working as designed** (elec `04` sec. 6). Claims writable
now (budgets, masks, boundary `emc:`), discharged by `assume!`/`by
test` until chamber-model packs exist; `--release` refuses the
unacknowledged gap. Chamber models can arrive any time with zero spec
change -- which is the definition of a closed question.

**D75. EOPEN-13 closed: nogoods are per-run; the cross-run soundness
condition is stated once** (elec `08` sec. 2). Substrate `07` already
says nogoods are solver state, never lockfile content. Cross-run reuse
is sound iff the cache key includes every catalog record revision the
nogood's blame set consumed -- INV-1's discipline applied to search
state. Implementing that cache is orchestrator work with no spec
surface; it cannot reopen.

**D76. EOPEN-14 closed: WCET model availability is registry content**
(elec `08` sec. 2). Ship conservative bound models for the Cortex-M
class; everywhere else, `by test` traces with declared error models.
`info.wcet` claims are architecture-agnostic by construction; a new
architecture is a new model pack.

**D77. SOPEN-3 technical half closed: the registry hosting model**
(substrate `11` sec. 10; INV-22 corollary added in the same change per
the ledger house rule). One observation does all the work: content
addressing (INV-22) makes hosting affect *availability, never
meaning*. Consequences, each now normative: sparse append-only index +
dumb content-addressed archive store; sources declared in the manifest
(no ambient state); mirrors trivially safe (a poisoned mirror can only
cause the loud drift error); signing carries trust tiers and hosting
does not (no registry operator can mint certification); yank hides but
never deletes (yesterday's lockfile builds today); `quarry vendor` for
air-gapped builds; computed-semver enforcement is the one server-side
check (it protects consumers from authors). Deliberately absent:
registry-side verdicts, per-registry namespace magic, any mutable
"latest" channel. Naming (the tool, the public registry) rides with
OPEN-10/EOPEN-1 as the owner's call -- candidates in section C.

## C. Naming candidates (OPEN-10 / EOPEN-1 / SOPEN-3 names -- OWNER'S
CALL, not decided here)

Constraints: the pair must read as a family; evocative single words in
the register of `quarry` (which is staying -- it names the tool that
extracts raw material, and the owner likes it); short, ASCII, usable
as file extensions and CLI verbs; low collision with dev tooling and
with in-language vocabulary (mech/04 sec. 1 near-collision rule).

Candidate families, strongest first:

1. **mill / loom** (`.mill`, `mill check`; `.loom`, `loom check`).
   The two great shaping machines: a mill cuts solids; a loom weaves
   threads -- and "wiring loom" is *literally the trade term* for a
   harness, so the elec name is domain jargon in the best way. Family
   story: the quarry feeds the mill. Collisions to weigh: `mill` is a
   Scala build tool (com-lihaoyi/mill; domain-distant but real);
   `loom` is a video tool and a JVM project (both distant);
   in-language, `cnc_mill` process modules and `FaceMill` features
   exist -- the language name never appears in source, but the echo
   is worth a deliberate yes/no.
2. **lathe / etch** (`.lathe`, `.etch`). Both subtractive-process
   words, both nearly collision-free in dev tooling (Debian "etch" is
   ancient history). `etch` is a superb PCB word. `lathe` is narrower
   than milling as a metaphor (turning only) and `cnc_lathe` has the
   same in-language echo as mill.
3. **forge / loom** (`.forge`, `.loom`). Forge is the strongest mech
   metaphor (shaping metal by force + heat) but the most collided
   name in software (SourceForge, MinecraftForge, Atlassian Forge,
   "forge" as the generic term for git hosts).
4. **Keep mill / loom.** Honestly serviceable, zero rename cost,
   initialisms that say exactly what they are; the cost is charm,
   not function. WO ground rule 6 keeps everything rename-safe either
   way (one extension-registry module; CLI placeholder `decl`).

Registry (the public instance quarry resolves against by default):
**lode** is the natural mate for quarry ("the mother lode"; an ore
vein you mine) -- short, warm, unclaimed in this space. Alternatives:
`vein` (fine, slightly clinical), `bedrock` (strong word, strongly
Minecraft), or leaving it "the quarry registry" (zero-name option).
`seam` is disqualified: [SEAM-n] is spec status vocabulary.

Decision protocol unchanged: due at Phase B start; WO ground rule 6
means the rename is one module + docs sweep whenever taken.
