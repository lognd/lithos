# Design cycle 9 -- 2026-07-03 (implementation architecture + naming)

Mandate (user directive): make the large architectural decisions for
the actual project -- Rust where it matters (lexing, parsing, graph
solving, computationally heavy), Python for orchestration and the
outside world; PyO3 + maturin; the most professional cross-platform
choices; optimize FIRST AND FOREMOST for DX; leave no stone unturned;
prepare everything for hand-off to less-capable agents for granular

This cycle produces implementation decisions, not spec changes; spec
versions unchanged (mech 0.13 / elec 0.10). Numbering continues
(D78..).

## A. Decisions

**D78. NAMING SETTLED (owner's decision): mill / loom / quarry.**
The mechanical language is **mill** (`.mill`); the electrical and
computer language is **loom** (`.loom`); the package tool remains
**quarry** (owner-liked, confirmed). Rationale as pitched (cycle 8
log sec. C): the two great shaping machines -- a mill cuts solids, a
loom weaves wires ("wiring loom" is the trade term for a harness) --
and the quarry feeds them. Closes OPEN-10 and EOPEN-1, and SOPEN-3's
tool-name residue. Still open, deliberately: the **umbrella name**
for the shared distribution/CLI/import package (the two languages
share one toolchain -- one wheel, one lockfile, mixed projects like
Kestrel -- so the dist cannot be named after either language);
placeholders stay `decl-eng`/`decl` per implementation ground rule 6.
Candidate offered: **wright** (the maker word -- millwright,
wheelwright, shipwright -- a millwright is literally one who builds
mills). The public registry instance leans **lode** [LEANING].
Transition rule: the docs/examples corpus keeps `mill`/`loom`
spellings until a single mechanical rename sweep (TODO.md); the
extension registry module accepts `.mill`/`.loom` AND legacy
`.mill`/`.loom` until that sweep lands, then drops legacy.

**D79. Implementation architecture adopted:**
`docs/implementation/00-architecture.md` (AD-1..16) is normative.
The load-bearing choices, argued there in full:

- **AD-1 split**: Rust owns source-text -> serialized-obligations
  (quantity core, lexer/CST/parser/formatter, entity DB, queries,
  ownership, monomorphization, symmetry, ledgers, contract IR,
  obligation construction, content addressing, diagnostics); Python
  owns everything that talks to the world (orchestrator, evidence
  cache, lockfile authoring, harness models on numpy/scipy, realizer
  adapters, quarry, CLI). The boundary coincides with the spec's own
  serialization point (obligations are self-contained by design), so
  the FFI is coarse: one crossing per build.
- **AD-2 layout**: monorepo; cargo workspace (`decl-util/diag/qty/
  syntax/sem/ir/oblig/api/py`, strict layering, `decl-py` =
  marshalling only) + maturin mixed layout + uv; ONE wheel carries
  the Python package and the compiled extension -- version skew is
  unrepresentable.
- **AD-3 parser stack**: logos lexer + layout pass (INDENT/DEDENT) +
  rowan lossless CST + hand-written event-based recursive descent
  with Pratt expressions and layout-anchored error recovery; typed
  AST as generated CST views; cargo-fuzz as part of done. The
  rust-analyzer/ruff/biome architecture; parser generators rejected
  (indentation, recovery, CST fidelity). Lossless CST buys the
  formatter and the future LSP for free.
- **AD-4 boundary**: PyO3 abi3-py312; CoreSession/BuildOutput; GIL
  released during compile (rayon inside); panics caught at every
  entry (CoreBug + backtrace); failing builds are DATA (diagnostics/
  verdicts), never exceptions; the `decl/compiler.py` facade is the
  only importer of `decl._core` and converts to typani Result --
  house style holds on both sides of the fence.
- **AD-5 schemas**: single source of truth in Rust; schemars ->
  JSON Schema -> generated pydantic v2 models, committed +
  CI-drift-checked; content addresses = blake3 over versioned
  canonical CBOR (JSON for payloads, CBOR only as hash input).
  NO DUPLICATION made mechanical at the language boundary.
- **AD-6 determinism**: no HashMap order in outputs, ryu float
  formatting, fixed summation order, outward-rounded intervals via
  f64::next_up/next_down, 3-OS golden-hash CI diff; INV-21's
  causeless-values-unrepresentable as a Rust API property.
- **AD-7/8**: thiserror per crate, diagnostics-as-values, ONE
  renderer (annotate-snippets, in Rust; Python prints strings);
  tracing everywhere bridged via pyo3-log into one dictConfig.
- **AD-9 numerics**: rational-exponent dimensions (nV/sqrt(Hz) is
  real), exact-rational unit factors, f64 values, own Interval type.
- **AD-10..13**: uv/pydantic/typani/typer stack; per-layer test
  strategy (insta, proptest, fuzz, criterion on the Kestrel corpus,
  cross-boundary pytest goldens); 3-OS CI + maturin wheels +
  cargo-deny; the DX contract (sec. 0) with the Makefile as the
  interface -- `make install` to working env, `make dev` watch loop,
  `make check` cheapest-first single gate.
- **AD-14 WO impact**: every WO reassigned by language (header lines
  added in place); WO-01 rewritten for the hybrid workspace; NEW
  WO-18 (FFI bridge + schema pipeline + typed facade + drift
  checks), gating WO-14/15; WO-05's lark note superseded by AD-3.
- **AD-15 risk register**: the stones, each with a disposition --
  panic policy, schema drift, platform nondeterminism, GIL, wheel
  skew, hash-order leaks, float text drift, dev-loop speed, renderer
  duplication, logging split-brain, rename mechanics, Windows,
  incrementality upgrade path (salsa adoptable later; content
  addressing already gives artifact-level incrementality), LSP/wasm
  future (CST + logic-free bindings keep the core embeddable).

**D80. Registry named (owner's decision): lodestone.** The public
registry instance quarry resolves against by default is **lodestone**
-- the naturally magnetized stone that points the way (lodestar
lineage), staying in the quarry/mill/loom mineral-and-machine
register while adding the "this is what orients resolution" meaning.
Supersedes the `lode` [LEANING] in D78. SOPEN-3 is now closed in
full: hosting model (D77), tool name quarry (D78), registry name
lodestone (D80).

## B. Findings

**F92. The spec's own architecture dictated the FFI boundary.** The
four-component split (substrate `01` sec. 4) and "obligations are
self-contained and serializable" (substrate `07` sec. 2) were written
before any implementation language was chosen -- and they land
exactly on the Rust/Python seam: compiler and harness are pure
functions with the orchestrator composing them, which is precisely
the coarse-FFI shape PyO3 wants. No spec accommodation was needed.
Design-for-verifiability turned out to be design-for-embedding.
