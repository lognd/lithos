# Design cycle 19 -- 2026-07-06 (WO-29 design pass: the lowering output surface)

Mandate: WO-29 deliverable 1 -- settle Q1-Q5 of
`23-lowering-output-surface.md` before any implementation leaf,
closing the F96/D87 gap (four consumer WOs each proved their
infrastructure against hand-built fixtures because `regolith-lower`
did not yet emit the typed IR their forward contracts assumed).

Numbering continues (F97.., D88..).

## A. Decisions

**D88 (Q1). `EntityKind` gets first-class `Hole` / `Bend` variants;
typed fields ride the existing `Measures` map, not new struct
fields.** `Other(String)` stays the pack-defined escape hatch only --
the two domain kinds two consumers (WO-28's `forall`, and the mech
realizer's geometry) already name in forward contracts get a stable,
exhaustively-matchable discriminant instead of a string comparison.
Well-known measure keys: `position`/`diameter`/`edge_distance` for
`Hole`, `radius`/`angle`/`line` for `Bend`. `Net` stays bare (no
typed-field need is forward-contracted yet).

**D89 (Q2). Feature program lands as a new `BuildPayload` field,
Rust-authored, Python-regenerated.** `feature_programs:
IndexMap<PartName, FeatureProgram>` -- a payload field, not a side
artifact (AD-4: consumers see only the serialized payload; a side
artifact would need its own versioning/determinism story outside
`payload_json()`, INV-10). AD-5 direction of truth: Rust schemars type
is authored, `SCHEMA_VERSION` bumps, the hand-written
`python/regolith/realizer/mech/schema.py::FeatureProgram` becomes the
drift-checked target and is eventually replaced by the generated
model.

**D90 (Q3). Binding-bridge split: Rust emits raw per-block capability
demands; Python derives the candidate table.** `regolith-lower` stays
IO-free (AD-17) -- it projects `BlockRequirement`-shaped demands from
lowered entities/claims into a new payload field. Quarry registry
candidate screening is I/O, so it stays Python/orchestrator-side. The
allocation loop is untouched either way (regolith/07 sec. 7).

**D91 (Q4). Exactly two `OpaqueIsland` items are promoted, no
more.** `parts:` per-line orbit constructors (deliverable 2's `Hole`/
`Bend` materialization) and `connect` arrow-line endpoints
(deliverable 5's `Mating` construction). Everything else in the WO-05
residue list stays opaque, each already owned by a named future item
(WO-24's `flows:`, WO-23's/WO-11's Walk surface, the WO-05-residual
elec behavioral bodies) -- promoting them here would be scope creep
this WO does not need.

**D92 (Q5). AD-22 adopted: one producer, no consumer side
channels.** The four independent WOs that hit this gap and each
recorded a cut instead of inventing a workaround were following a
rule that was, until now, only convention. `00-architecture.md` AD-22
makes it explicit: consumers bind only to schema-versioned lowering
output; a forward-authored consumer contract is a spec to be promoted
into the payload, never a permanent parallel path into internal
compiler state.

## B. What this cycle actually closed

The design pass (D88-D92) is decided and normative
(`23-lowering-output-surface.md` section 5 flips to DECIDED). Of the
implementation deliverables gated on it:

- Deliverable 2 (domain entity structuring): the `EntityKind`
  extension (D88) landed and is query-engine-reachable (`base_selector`
  in `regolith-sem::query` now maps `holes`/`hole` and `bends`/`bend`).
  Populating `Hole`/`Bend` entities from real `parts:` lines (the
  Q4(a) parser promotion, D91) hit the SAME class of upstream wall
  WO-23 and WO-28 each hit before this WO existed: `parts:` bodies are
  today swallowed whole as one `OpaqueIsland` per declaration (one
  entity per top-level `Decl`, not one per constructor line), and
  giving the parser a typed per-line CST node is itself a
  grammar-and-CST change larger than this dispatch's remaining budget
  could respect and still keep `make check` green with real (not
  faked) golden coverage. Recorded as a cut in the WO-29 file rather
  than faked; the `EntityKind` shape decision is not blocked by it and
  is available to whichever dispatch does the `parts:`-line promotion
  next.
- Deliverables 3, 4, 5 (feature-program emission, binding-requirement
  bridge, `connect` -> `Mating`): shapes decided (D89/D90/D91) but not
  implemented this cycle for the same reason -- each needs either the
  `parts:`-line promotion or the `connect`-line promotion (D91) as a
  prerequisite, and those parser-level changes are the actual
  remaining gap, not anything this design pass left ambiguous.

This mirrors the WO-28 spec-cycle precedent exactly: land the shape
decisions as their own reviewable, normative slice; let the next
dispatch build on settled ground instead of re-deciding Q1-Q5 under
implementation pressure.
