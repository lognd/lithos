# dcad — Declarative CAD Language: Overview Specification

> **Draft 0.2** — full rewrite incorporating the scope/commit execution model,
> the ownership system, interfaces and contracts, the obligation/evidence
> boundary, and the tolerance system.
> Status: design phase. Items marked **[OPEN]** need a decision; items marked
> **[SEAM]** are designed on both sides but the joint itself is not yet
> specified.
> Scope note: mechanical track only. Electrical is deferred entirely.

---

## Part I — Foundations

### 1. Vision

A declarative, goal-oriented language for mechanical design, architected like
a modern compiled language: typed source, multiple IRs, lowering passes,
static verification before expensive computation, and reproducible builds.

The central inversion versus traditional CAD:

```
Traditional:  Geometry → (manual analysis) → "does it work?"
dcad:         Claims + Contracts → (solvers, provers) → Geometry + Evidence
```

The designer declares **what a part must do**, **how it will be made**, and
**what it promises to other parts**. Dimensions and tolerances may be left
unresolved; the system allocates them. Every geometric reference has exactly
one valid interpretation, enforced statically. Every physical claim is either
backed by evidence or loudly marked as assumed.

**Why text:** diffable, reviewable, version-controllable — and statically
checkable, which makes design generation (human or LLM) a *locally verifiable*
problem: a part can be written and checked against fixed contracts without
rendering anything.

### 2. System Architecture: Four Components

The system is split so that each component can be built, tested, and
contributed to by someone who understands only that component.

```
┌────────────────────────────────────────────────────────────┐
│ QUANTITY CORE          shared vocabulary (the "ABI")       │
│   namespaces, quantities, units, tensor ranks, intervals   │
└──────────────┬─────────────────────────────┬───────────────┘
               │                             │
┌──────────────▼──────────────┐ ┌────────────▼───────────────┐
│ MODELING LANGUAGE (Part 1)  │ │ VERIFICATION HARNESS (P2)  │
│  what is CLAIMED            │ │  how claims are CHECKED    │
│  geometry, ownership,       │ │  physics models, validity  │
│  contracts, requirements,   │ │  domains, error estimates, │
│  constraint graph, fits     │ │  meshing, solvers, search  │
│                             │ │                            │
│  emits: OBLIGATIONS ────────┼─┼──► returns: EVIDENCE       │
└──────────────┬──────────────┘ └────────────┬───────────────┘
               │                             │
┌──────────────▼─────────────────────────────▼───────────────┐
│ ORCHESTRATOR                                               │
│   optimization loop, caching, lockfile, scheduling,        │
│   the API surface for CLI / CI / future UI (Part 3)        │
└────────────────────────────────────────────────────────────┘
```

- **Quantity core.** Typed physical quantities in namespaces
  (`mech.stress: Pa, tensor(2)`; `thermo.temperature: K, scalar`), units,
  dimensional analysis, and interval-valued quantities. Owned by neither big
  component; both depend on it. This is the system's keystone and changes to
  it are breaking changes everywhere.
- **Modeling language.** Everything semantic: parts, features, scopes,
  ownership, queries, interfaces, fits, budgets, requirements. Performs all
  *non-physical* checks itself (borrow conflicts, cardinality, units, DOF
  ledgers, capability-table lookups, budget arithmetic). Compiles physical
  claims into **obligations**.
- **Harness.** A registry of physics models — analytical (Lamé, joint
  diagrams, beam theory) and numerical (FEA at varying fidelity) — each with a
  validity domain, error model, and cost. Discharges obligations; returns
  **evidence**. The boundary is physics, *not* expense: closed-form physics
  lives here too. Speed tiers are a property of which model discharges a
  claim, not of which component runs the math.
- **Orchestrator.** Owns all state and iteration: the free-variable
  optimization loop, evidence caching, the lockfile, parallel discharge.
  Compiler and harness are pure functions; the orchestrator composes them.

### 3. Design Principles

1. **Name intent, not topology.** No entity indices in source; all references
   are semantic queries.
2. **Ambiguity is a compile error.** Queries with multiple interpretations
   fail with geometric, actionable diagnostics. `.all` and `.any` are explicit
   escape hatches with checked semantics.
3. **Single ownership, explicit joins.** Every entity has one owner; selecting
   across ownership boundaries requires join syntax.
4. **Promises, not actuals.** Assemblies compute only with declared contract
   values. Consuming a computed actual is a named opt-in that records its
   blast radius in the build graph.
5. **The defaults test.** A default behavior is permissible iff it is
   (a) conservative, (b) local in effect, and (c) materialized in the lockfile
   where a diff will surface it. Anything failing the test must be an explicit
   declaration.
6. **Intervals, not points.** Tolerances, fit interferences, preload scatter,
   and environment ranges propagate as intervals; every check is evaluated at
   its own worst-case corner.
7. **Claims need evidence.** Every physical claim is discharged, violated,
   indeterminate, or *explicitly assumed*. "I can't tell" is never conflated
   with "it fails."
8. **Static before geometric before physical.** The cheapest tier that can
   catch an error owns it.

## Part II — The Modeling Language

### 4. Type System

#### 4.1 Quantities and units

All scalars are typed physical quantities from the quantity core. Dimensional
analysis runs at parse time; unit errors never reach a solver. Arithmetic
between incompatible quantities is a type error.

Equality (`==`) on continuous quantities is a **compile error**. Legal forms:
`within(a, b, tol)`, constraint declarations (`coincident`, `equal_within`),
and exact equality on discrete types only.

#### 4.2 Toleranced dimensions

`length` is **nominal + tolerance**, not a bare scalar. The tolerance slot is
usually *unwritten in source* and *allocated by the compiler* (§12), resolved
into the lockfile. Explicit forms exist: `5mm ± 0.1`, `Ø30 H7`,
`position(±0.05, to=frame)`.

#### 4.3 Free variables

A dimension may be a bounded free variable:

```
var wall: length in [1mm, 6mm] minimize
var fillet_r: length = 0.5 * wall          # derived
bore = Pocket(circle(d=var(30mm, 50mm)), ...)
flange = Bend(edge=..., radius=free)        # "DFM decides the legal minimum"
```

- **Domains are hard.** Violations are unsatisfiable, never clamped.
- **Resolution is two-phase** (§14): eager (DFM/closed-form propagation on the
  constraint graph), then lazy (physics-driven, via the orchestrator loop)
  only for variables eager resolution cannot pin and only when verification of
  the eager candidate fails.
- **Topology boundaries are domain constraints.** If a value would change the
  topology class (e.g., a fillet consuming an adjacent face), the feasible
  domain is the declared domain ∩ the topology-preserving region. The
  optimizer never proposes across the boundary. Resolving *pinned against*
  a boundary produces a `note:` naming the boundary and the intentional
  alternative (e.g., `FullRoundFillet`). For **literal** dimensions, crossing
  a topology boundary is a compile error (`E0304`) raised semantically and
  pre-kernel where predictable, with constructive options including
  conversion to a free variable.

#### 4.4 Integer and cardinality variables

`n = var(2, 6)` over integers is **monomorphized**: the bounded domain is
exhaustively enumerated and *every* static check (cardinality, borrow,
indexing, conformance) runs per-instantiation. Source must be valid for the
whole domain or the domain must shrink. Lazy resolution over integer variables
is discrete branch evaluation, matching the language semantics exactly.

Conditional features may condition **only on eagerly-resolvable values**.
Conditioning feature existence on physics-resolved values is forbidden; the
mechanism for genuinely alternative designs is a named `variant`, each
independently checked. **[OPEN-1]** `variant` syntax and cross-variant
sharing rules are unspecified.

### 5. Parts, Processes, Stages

A part binds geometry to a **process pipeline**. Real parts are sequences
(cast → machine; cut → bend → weld → drill), so `stage` is first-class:

```
part Housing:
    material: AlSi10Mg

    stage cast: process=die_cast(...)
        ...features...

    stage machined: process=cnc_mill(axes=3), from=cast
        datum_face = FaceMill(on=cast.body.bottom, depth=0.5mm)
        ...
```

- Each stage has its own process module, capability table (§12.1), and DFM
  rule pack. Features must come from the stage's process vocabulary.
- Stage boundaries are ownership checkpoints: a later stage may modify earlier
  geometry (that is what machining a casting *is*), transferring ownership,
  subject to borrow rules (§9) — notably, interface-bound entities are
  protected across stages.
- Dimensions and tolerances are stage-qualified; capability checks run against
  the stage that *finishes* the controlling surface.

### 6. Execution Model: Scopes and Commit

The execution model is HDL-style: **within a scope, all features are
concurrent**; state advances only at scope boundaries.

```
stage formed: process=sheet_metal(stock=1.5mm)

    scope: blank  = Blank(rect(80mm, 120mm))

    then:  holes  = Pierce(circle(5mm), pattern=grid(2,3,25mm), on=blank.face)
           slot   = Pierce(slot(6mm, 20mm), from=blank.left, offset=15mm)
           flange = Bend(edge=blank.top, angle=90deg, radius=free)
           # text order of these three is irrelevant; all see the same
           # entry snapshot; none can see each other's output

    then:  blend  = Fillet(where=flange.bend_edges, radius=free)
```

Rules:

1. **Snapshot reads.** Every feature in a scope queries the body state at
   scope entry. A feature's exports are visible only in *later* scopes.
   Referencing a sibling's exports is a compile error.
2. **Merge at commit.** Effects merge at the scope boundary. Merge legality:
   - *Same-sign overlap* (two cuts, or two bosses, on the same region):
     geometry commutes; merge is automatic. Ownership of the contested region
     is ambiguous and **must be declared only if some later query references
     it** (lazy ordering: `merge(a over b)` or feature reordering).
   - *Mixed-sign overlap* (an add and a cut contest a region): geometry does
     not commute → **always** a hard error in one scope. Resolve by
     sequencing into separate scopes or an explicit `merge(a before b)`.
3. **Dependency depth is visible.** Scope nesting depth *is* the feature
   dependency depth. Shallow designs (many features off the same snapshot)
   are the path of least resistance, by construction.
4. **Borrow checking is per-scope** (§9): the conflict check is set
   intersection of modified-entity sets against the entry snapshot.

`.current`/`.original` no longer exist: a query means "at my scope's entry
snapshot," full stop. Pre-modification references are served by **datums**
(§10).

### 7. Topology as a Database; Queries

After each commit, the body's topology is an entity database:

```
faces: id | origin_feature | owner | kind | normal | area | tags | sym_orbit
edges: id | origin_feature | owner | kind | dir | len | adj_faces | sym_orbit
```

IDs are internal only. Source-level references are **queries**:

```
shell.edges.where(parallel_to=Z, adjacent_to=shell.top)
edges.where(from_feature=cutout)
body.edges[adjacent=top, dir=Z]                # sugar form
body.top_edges - slot.all_edges                # set algebra
edges.at_intersection(slot, base.top)          # explicit cross-owner join
pattern.instances.nearest(corner_ref)          # instance addressing (§11)
```

- **Static validation:** predicate names, entity kinds, operand types, and —
  via per-feature topology-effect predictions — expected cardinalities are
  checked on the Feature IR, before any geometry kernel runs.
- **Cardinality typing:** a query is `Entity` (exactly one), `Set[Entity; n]`
  (cardinality tied to an integer variable; consumers must be
  cardinality-polymorphic, as fillet/chamfer naturally are), or
  `Set[Entity]` (dynamic; consumers must accept `.all`).
- **Features export named views**, not raw topology: filtered queries over
  entities the feature created/modified (`boss.side`, `boss.base_edge`),
  plus virtual geometry (`boss.axis`).
- **Post-geometry verification** confirms static topology predictions; feature
  classes whose effect is data-dependent must declare so and are verified in
  this pass (`E0304` family on failure).

### 8. Symmetry and `any`

The entity database tracks the body's **symmetry group**, computed
conservatively on the Feature IR from per-feature declared contributions
(a bolt-circle pattern declares Cₙ; a centered rect its mirrors; the body
group is the intersection). Conservative-but-sound: an undetected true
symmetry may cause a spurious `any` error; a false symmetry can never be
asserted.

`pattern.any` — "choose one, without loss of generality" — is legal iff all
candidates lie in **one orbit** of the *current* group. Later features can
break symmetry, splitting orbits; an `any` after the break is `E0502` with
pinning suggestions (`nearest`, `opposite`, explicit index). `any` resolves to
a canonical representative recorded in the lockfile (bit-reproducible builds).

Symmetry state flows into obligations as hints (§16): a Cₙ-symmetric body and
load case licenses sector models with cyclic boundary conditions in the
harness, automatically.

### 9. Ownership and Borrows

1. **Single ownership.** Owner = the feature that most recently created or
   modified the entity (per merge rules §6.2 for contested regions).
2. **Selections are borrows.** A feature consuming a query holds an immutable
   borrow on the resolved entities for the remainder of its stage. A later
   feature (or stage) modifying borrowed entities is a borrow conflict
   (`E0302`) with suggested resolutions (reorder/rescope, narrow the query,
   `rebind()` after the modifier).
3. **Interface bindings are permanent borrows** (§13.2): role-bound entities
   are protected for the part's lifetime, across all stages. "A later op ate
   the bearing seat" is a compile error naming the interface as borrower.
4. **Cross-owner selection requires joins** (`∩`, `at_intersection`).
5. The full borrow check runs on the Feature IR using predicted topology
   effects; the post-geometry pass (§7) closes the prediction gap.

### 10. Datums

Non-topological, immutable reference geometry (planes, axes, points),
explicitly captured:

```
datum left_ref = blank.left_edge.as_datum()    # captured at this commit
flange = Bend(edge=blank.left_edge, ...)        # consumes the edge — fine
hole   = Pierce(circle(5mm), from=left_ref, offset=10mm)   # still valid
```

Datums cannot be modified or consumed → outside the borrow system by
construction. They are the *only* mechanism for referencing
since-consumed geometry, and the natural anchors for tolerance frames and
instance addressing.

### 11. Patterns and Instances

`HolePattern`, `PatternOf` (§13.5) and kin produce instance sets with declared
symmetry. Instance addressing is semantic, never positional-index:
`instances.nearest(datum)`, `instances.where(x > 40mm)`, `instances.any`
(under §8 rules). Per-instance variation is out of scope for v1 — "all holes
5mm except one 6mm" is two features.

Indexing against integer-variable cardinality is checked per-instantiation
(§4.4): `holes[4]` with `n=var(2,6)` is `E0501` unless the domain guarantees
it.

### 12. Tolerances, Fits, Budgets

#### 12.1 Process capability tables

Process modules export what they can hold (`bore(d<20mm): IT9`;
`bore(..., ream): IT7`; `face: ±0.05mm`; `position: ±0.1mm`), per stage.
Capability lookup is a static check.

#### 12.2 Fits

Fits are declared by ISO designation between mating roles (`H7/g6`); the
compiler expands limit deviations from ISO 286 tables and checks process
capability (`E0410` with constructive options: add finishing op, add stage,
loosen fit). Fit interferences/clearances are **intervals** consumed by
connection models (§13.4).

#### 12.3 Allocation: local by default, budgets by declaration

- **Default:** strictly local allocation per connection/stackup — loosest
  (cheapest) process-capable tolerances that close each requirement,
  using only the parts at that connection. No cross-part information flow.
  (Passes the defaults test: conservative, local, lockfile-materialized.)
- **When local allocation cannot close a chain:** `E0432`, naming the chain,
  the worst contributors, and the options. Cross-part rebalancing exists
  *only* inside an explicitly declared, named **budget**:

```
budget mesh_alignment:
    require: misalignment(gear.axis, pinion.axis) < 0.05mm
    members: [housing, shaft]              # explicit blast radius
    allocate: cost_optimal
    pins: shaft.seat_runout: locked(±0.015)
```

  A part's required tolerances change only via budgets it is a member of; the
  lockfile diff names the budget as the cause.
- Stackup math: worst-case in v1; RSS as a declared option per budget.
  **[OPEN-2]** statistical allocation policy (process capability indices,
  Cpk) is future work but the budget syntax should reserve a slot.

## Part III — Interfaces, Connections, Assemblies

### 13. Interfaces

#### 13.1 Definition

An interface = **datum reference frame + geometric roles + tolerances on
roles relative to the frame + a promise set** (loads, stiffness):

```
interface BearingSeat<d: length, depth: length>:
    frame:
        origin: bore.axis ∩ shoulder.plane
        z: bore.axis                      # x,y free → C∞ symmetry, declared
    roles:
        bore:     cylindrical(d=d, depth >= depth), internal
        shoulder: planar, normal=+z, annular(od >= d + 8mm)
    tolerances:
        bore:     cylindricity(0.008mm), tol_class(H7)
        shoulder: perpendicularity(0.01mm, to=z)
    loads:
        radial: rated(5kN)
        axial:  rated(2kN, +z)
    stiffness:
        radial: promised(>= 80kN/mm)
        tilt:   promised(>= 4kN·m/mrad)
```

The frame is the contract's coordinate system: all tolerances, alignment, and
loads are expressed in it; assembly positioning is frame alignment; assembly
stackups are frame-to-frame transform chains.

#### 13.2 Implementation and conformance

```
impl BearingSeat<d=30mm, depth=10mm> for Housing:
    bore     = machined.main_bore.wall
    shoulder = machined.main_bore.floor
```

Conformance checks in three tiers: **static** (role kinds by feature
construction; parameter match — binding may *pin* a free variable;
process-capability vs. tolerance demands, per finishing stage), **geometric**
(post-build predicates: depths, annular material, no role-violating
intrusions), **physical** (survive the promise set — discharged as
obligations, §16). Role binding is a **permanent borrow** (§9.3).

Roles bind to geometry from a named stage; the binding stage must be the one
that finishes the surface. **[SEAM-1]** The precise rule for role binding ×
stages × scopes (e.g., binding to geometry later *refined* by a finishing op
on the same face) is not yet specified — this is the next part-level design
task, together with **[SEAM-2]**, the interaction of `PatternOf` instances
with scope/commit and per-instance role binding.

Promises carry an **evidence clause**: `by analysis` (default → harness
obligation), `by catalog(ref)` (datasheet evidence, hash-pinned), `by
test(report_ref)`. Vendor parts are interface bundles whose obligations are
discharged by catalog evidence — never modeled internally. This makes
designed, purchased, and lab-tested components one verification model, and
defines the stdlib/ecosystem unit: community interface packs (NEMA mounts,
ISO flanges, extrusions, fastener patterns) with **semver from contract
diffs** (widening promises/loosening demands = minor; the reverse = major).

#### 13.3 Composition

- **Bundling:** interfaces contain interfaces
  (`PilotedFlange = Pilot + PatternOf<ThreadedHole<M6>, n, circle(bc)>`),
  adopting a sub-interface's frame.
- **Refinement:** narrow an existing contract (tighter tolerance, higher
  rating); refined impls are substitutable for the base (checked, one-way).
- **No inheritance/overriding.** Substitutability would be unenforceable.

#### 13.4 Matings (connections)

A `mating` is a first-class connector between complementary interfaces,
carrying five contract kinds:

```
mating BoltedFlange<n, bolt: BoltSpec, circle>:
    between: FlangeFace, FlangeFace
    align:   a.frame = b.frame (z anti-parallel, planes in contact)
    fit:     —                                  # (press-fit example: H7/p6)
    dof:     removed=[all 6]
    preload: torque: 25N·m → P0; P0_range=[0.75,1.25]·P0   # method scatter
    model:   joint_diagram(k_bolt, k_members(frustum), Φ)   # → harness node
    state_requirements:
        no_separation: P0_min - (1-Φ)·F_ext_max > 0, SF=1.5
        no_slip:       μ·(P0_min·n - F_ax) > F_shear, SF=1.3
    generates:
        → flanges: clamp_pressure(P0_max·n, over=contact_annulus)
        → bolts:   static(P0_max + Φ·F_ext_max), alternating(Φ·F_alt)
    capability:
        shear: μ·P0_min·n      axial: per joint diagram
```

- **Alignment** is the *only* positioning mechanism in assemblies (no
  free-transform mates). Under-located parts are compile errors.
- **DOF accounting** feeds the assembly ledger (§14).
- **Generated loads** are *declared as model references*
  (`contact_pressure = model_ref(lame, inputs={d, D, δ: interval})`),
  evaluated by the harness, injected into each implementing part's
  verification load set. **Interval discipline:** each check consumes its own
  worst-case end (part stress at δ_max/P0_max; transmitted capability at
  δ_min/P0_min; separation at P0_min).
- **State requirements** verify the connection stays in its declared regime
  (clamped, stuck, seated) over the whole load and environment envelope.
- Connection models live in the harness registry with validity domains;
  `model=fea_contact` is the declared escalation (a `measured`-class opt-in).
- The mating stdlib ships in two coordinated, co-versioned halves:
  declarations (modeling side) and model nodes (harness side).

#### 13.5 `PatternOf`

`PatternOf<I, n, layout>`: n instances of interface `I` in a layout, declaring
its symmetry group (Cₙ for circular layouts) by construction — enabling
`bolts.any` (§8) and sector-symmetric discharge hints.

### 14. Assemblies

```
assembly Gearbox:
    parts:
        housing: impl BearingSeat<30,10> as front_seat, ... 
        shaft:   ...
    connect:
        front: BearingMount(housing.front_seat, brg_front.od)
        rear:  BearingMount(housing.rear_seat,  brg_rear.od, axial=floating)
    environment:
        temperature: [-20°C, 95°C]
    require Loads:
        applied: gear_mesh_force(2.2kN, at=shaft.gear_seat, angle=20deg)
    require Dynamics:
        f1 > 120Hz
```

Assembly-level computation uses **promises only** (Principle 4):

1. **DOF ledger.** Sum removed/free DOF per part: must equal 0 (statics) or a
   *declared* free set (mechanisms). Over-constraint (e.g., double axial fix —
   the fixed/floating bearing rule) and under-constraint are both errors
   (`E0420` family) before any geometry exists.
2. **Rigid statics.** The connection graph + ledger *is* a free-body diagram;
   reactions at every interface are solved by linear algebra. Envelope checks:
   `actual ≤ rated` per interface.
3. **Stiffness network.** Promised stiffnesses + connection models + part
   masses form a spring(-mass) network: deflection chains
   (`require: misalignment < x`) and first-mode estimates
   (`require Dynamics`) — conservative by construction, since promises are
   bounds.
4. **Connection state + generated loads** evaluated at interval corners,
   including **environment corners** (differential expansion shifts δ and P0;
   "the press fit that spins when hot" is a compile-time error at the hot
   corner).
5. **Escalation opt-ins.** `stiffness=measured`, `model=fea_contact` consume
   actuals: recorded in the build graph as cross-part dependency edges
   (part edit → assembly re-verification through that connection). This is
   the system-test tier; promise-backed checks are the unit-test tier.

**Interface-first workflow:** an assembly of *unbound* interfaces +
connections + applied loads is fully checkable (ledger, statics, envelope
budgets, tolerance budgets) before any part exists. Parts then implement
load-annotated contracts independently — the work-breakdown unit for teams
and for LLM generation alike.

**[OPEN-3]** Mechanism support beyond the DOF ledger (kinematic ranges,
`travel_range >= 150mm`, collision over a motion sweep) is unscoped. Likely a
v2 layer over the same connection graph; the `dof: free=[...]` syntax should
not need to change.

### 15. Deferral: `todo!` and `assume!`

- `impl ... = todo!` — defers **conformance**, not contract: parameters and
  the full promise set must be concrete; only proof is deferred. Assembly
  solves run normally on the promises. Carried in the warnings ledger;
  refused by `--release`.
- `assume!` — accepts an **obligation** without evidence (assumed loads,
  assumed stiffness in early design). First-class, ledgered, refused by
  `--release` except by explicit per-item waiver.

Both make early-phase uncertainty *visible state* instead of optimistic lies.

## Part IV — Claims, Obligations, Evidence

### 16. Requirements are claims

`require` blocks speak only the quantity core — no solver names, no `fea.`
prefixes:

```
require Structural:
    mech.max(stress.von_mises) < material.yield / 2.5
    mech.displacement(at=front_seat.frame) < 0.05mm
```

Verification *hints* are droppable annotations
(`@hint(symmetry=C3, regime=linear_elastic)`); symmetry hints flow
automatically from §8.

### 17. Obligations

The compiler lowers every physical claim (requirements, interface promises
`by analysis`, connection state requirements, generated-load stress checks)
into self-contained obligations:

```
obligation housing.front_seat.stiffness_promise:
    claim:    mech.stiffness(frame=front_seat.frame, dir=radial) >= 80 kN/mm
    subject:  geometry_ref(housing, snapshot=#a91f3c)      # content-addressed
    given:
        material: AlSi10Mg @ [-20°C, 95°C]
        loads:    interface_envelope(front_seat)
        backing:  promises_only          # or measured(...) for opt-ins
    hints:    [symmetry=C∞(z)]
```

Self-contained ⇒ serializable ⇒ remote/distributed discharge for free. The
harness never sees source; the compiler never sees physics.

### 18. Discharge: margin-driven model selection

The harness holds a registry of models, each with **inputs/outputs (quantity
core), validity domain, error model, cost**. Discharge rule:

> A model discharges a claim iff the claim holds **after charging the model's
> worst-case error against the margin**: `value + ε_model ≤ limit`, with the
> model inside its validity domain.

Consequences:

- **Accuracy is automatic.** Fat margins discharge via beam theory in
  microseconds; thin margins force refined FEA because nothing cheaper can
  close them. No fidelity knobs. Healthy safety factors make verification
  cheap — the right incentive.
- **Best-path search** over the model graph (quantity namespaces as nodes,
  models as edges) finds the cheapest in-domain path that closes the margin.
- **Evidence** is the only return type:

```
evidence: status: discharged | violated | indeterminate
          value: 94.2 kN/mm (ε ≤ 3.1)   margin: 11.1 after error
          model: shell_fea_v2 (validity: thin_wall ✓, linear ✓)
          cost: 4.2s    hash: #e77c01
```

- **Indeterminate ≠ violated.** "No in-domain model is accurate enough" gets
  its own diagnostic family: which models were considered, why rejected,
  what would resolve it (refinement budget, `measured`, more margin).
- Evidence is content-addressed and cached: unchanged
  (snapshot, contract, registry-version) ⇒ already discharged. Promise-backed
  assembly obligations survive part edits untouched — incremental
  verification falls out of the contract architecture.
- The harness's three implementation tiers (Python prototype → dynamically
  compiled nodes → fused optimized pipelines) are invisible across the
  boundary; the obligation/evidence format never changes as a model
  graduates.

### 19. The orchestrator loop

Only for physics-resolved free variables, and only after the eager candidate
fails verification:

```
1. compiler: DFM-resolved candidate → snapshot + obligations
2. harness:  discharge all → evidence set (+ sensitivities on request)
3. all discharged → lockfile, done
4. else: propose next assignment (adjoint-guided; constrained to
   eager-feasible ∩ topology-preserving domains), goto 1 incrementally
```

Compiler and harness stay pure; iteration, caching, scheduling, lockfile live
here. The orchestrator API is what CLI, CI, and the future UI consume.

**DFM-first policy:** the eager candidate (DFM minimums + secondary
objectives) is *the answer* if verification passes — no loop ever spins up.
Failed obligations seed the loop's direction via their constraint-graph
variable sets and sensitivities.

## Part V — Build System and Developer Experience

### 20. Build tiers

| command | runs | latency target |
|---|---|---|
| `dcad check` | parse, types/units, queries, borrows, monomorphized checks, symmetry, DOF ledger, capability/fit lookups, budget arithmetic, *analytical-model obligations* | ms–s |
| `dcad build` | + geometry, post-geometry verification, conformance T2, FEA verification of the eager candidate | s–min |
| `dcad optimize` | + the orchestrator loop | as needed |
| `dcad build --release` | + refuse `todo!`/`assume!` (per-item waivers), full evidence ledger | — |

### 21. Lockfile (`dcad.lock`)

Pins everything resolution produced, making builds reproducible and diffs
reviewable: resolved free variables (with resolving constraint: DFM rule /
obligation / topology boundary), allocated tolerances (with cause: local |
budget-name), fit expansions, `any` canonical choices, evidence hashes,
kernel version, model-registry version, waivers. A tolerance or dimension
change in review *names its cause*.

### 22. Diagnostics

Rust-style, geometric, constructive; stable error codes. Established
families: `E03xx` selection/topology (`E0301` ambiguity, `E0302` borrow
conflict, `E0304` topology-class change), `E04xx` contracts (`E0410`
capability, `E0420` DOF, `E0432` budget), `E05xx` instances/symmetry
(`E0501` index vs. domain, `E0502` broken-orbit `any`). Errors are
**batch-emitted with cross-references** (edit blast radius shown at once via
`dcad check --explain`), never first-error-stops.

### 25. Roadmap

**Phase A — spec hardening (no code):** 5–10 real parts + 2 assemblies in
target syntax (stress: scopes, queries, stages, impls, matings); quantity
core schema; entity DB schema; query grammar (EBNF); obligation/evidence
schema (serialization format).

**Phase B — static core (first shippable: a linter):** constraint graph +
entity DB; parser for a minimal subset; query validation, ownership/borrow,
monomorphized checks, DOF ledger, capability/fit lookups — all geometry-free.

**Phase C — geometry vertical slice:** Feature IR → CadQuery/Build123d →
STEP; post-geometry verification; one DFM pack (sheet metal).

**Phase D — contracts:** interfaces, impl conformance T1/T2, matings, one
assembly end-to-end (ledger + statics + a bolted joint's analytical state
checks via a hand-rolled mini-harness with 3–4 model nodes).

**Phase E — the boundary, for real:** obligation/evidence serialization;
harness as a separate process; orchestrator with caching + lockfile; eager
DFM resolution; lazy loop with adjoint hooks stubbed.

**Later:** Rust migration of hot paths; sketch language; mechanisms; UI.
