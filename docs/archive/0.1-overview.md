# Declarative CAD Language — Overview Specification (Draft 0.1)

> Working name: **dcad** (`.dcad` files). Naming TBD.
> Status: design-phase synthesis. Nothing here is implemented; this document
> captures the agreed design direction and flags open decisions.

---

## 1. Vision

A declarative, goal-oriented CAD language for mechanical and electrical design,
structured like a modern compiled language: multiple intermediate
representations (IRs), lowering passes between them, and lint/verify/optimize
stages at each level.

The core inversion versus traditional CAD:

```
Traditional CAD:   Geometry → (manual analysis) → "does it work?"
dcad:              Requirements → (solvers) → Geometry
```

The designer declares **what a part must do** and **how it will be made**.
Dimensions may be left as bounded free variables; DFM rules and FEA-driven
optimization resolve them. The text representation is the single source of
truth — diffable, reviewable, version-controllable, and tractable for LLM
generation because correctness is statically checkable without rendering.

### Goals

1. **Intent-first design.** Function and manufacturing process are first-class;
   geometry is derived.
2. **Goal-oriented dimensions.** Any dimension can be a free variable with a
   domain, resolved by DFM constraint propagation (eager) or FEA optimization
   (lazy).
3. **Unambiguous by construction.** Every geometric selection has exactly one
   valid interpretation, enforced at compile time (the ownership/borrow model).
4. **Manufacturing-aware from the start.** Features are imported from process
   modules (`from sheet import Bend`) that carry both geometric meaning and DFM
   knowledge.
5. **Testable designs.** `require` blocks are the unit tests of hardware:
   named, checkable claims about structure, manufacturability, and function.
6. **Open and extensible.** Process libraries, DFM rules, and physics models
   are user-extensible modules, not built-in magic.

### Non-goals (for now)

- Replacing interactive surfacing/industrial-design workflows (Class-A
  surfaces, sculpting).
- Building a geometry kernel. We drive OpenCASCADE (via CadQuery/Build123d
  initially).
- A GUI (Part 3 of the larger project; CLI-first for a long time).

## 2. Design Principles

1. **Name intent, not topology.** No edge indices, no topology hashes in source.
   All references are semantic queries resolved against the current model state.
2. **Ambiguity is a compile error.** A query that matches more or fewer
   entities than its consumer expects fails the build with a geometric,
   actionable error message. `.all` is the explicit escape hatch.
3. **Ownership is single and explicit.** Every topological entity has exactly
   one owning feature at any point in the feature graph. Cross-feature
   selections require explicit join syntax.
4. **Eager where cheap, lazy where expensive.** DFM constraints propagate
   without geometry evaluation; FEA-driven constraints resolve via an
   optimization loop (with surrogate models where possible).
5. **Everything is a module.** Processes, materials, DFM rule packs, and
   physics models are imported libraries with versioned interfaces.
6. **Readable errors are a feature.** Rust-style diagnostics that show matched
   entities, why they matched, and concrete suggested fixes.

## 3. Architecture: Compilation Pipeline

```
[.dcad source]
      │  parse
      ▼
[AST] ──────────────── syntax/type/unit checks
      │  lower
      ▼
[Intent IR] ─────────── functional layer: loads, envelopes, requirements
      │  lower            • requirement well-formedness checks
      ▼                   • assembly-level constraint checks
[Feature IR] ────────── feature graph + entity DB + constraint graph
      │  lower            • ownership / borrow checking (static, pre-geometry)
      │                   • query validation (typed, pre-geometry)
      │                   • DFM lint passes (eager constraint propagation)
      ▼
[Geometry IR] ───────── resolved parametric model, B-rep operations
      │                   • FEA-driven optimization loop (lazy resolution)
      │                   • final DFM verification on concrete geometry
      ▼
[Backends] ──────────── STEP, STL, drawings, G-code, Gerber/drill, BOM
```

Key property: **the borrow check and query validation run on the Feature IR,
before any geometry kernel is invoked.** A large class of errors is caught
without computing a single boolean operation.

### The layered intent model

Variables live at a layer; constraints flow downward:

| Layer | Declares | Resolved by |
|---|---|---|
| Functional | loads, safety factors, environment, envelope | FEA optimizer |
| Parametric | dimensions, domains, derived relations | constraint solver |
| Process | stock, process params, tool access | DFM propagator |
| Geometry | concrete B-rep | geometry kernel |

## 4. Language Overview

### 4.1 Parts, processes, materials

```
part GearboxHousing:
    material: AlSi10Mg
    process: cnc_mill(axes=3)

    features:
        body = Extrude(rect(80mm, 60mm), height=40mm)
        bore = Pocket(circle(d=var(30mm, 50mm)), depth=through,
                      on=body.front, center=true)
        ...

    require Structural:
        fea.bearing_load(face=bore.inner, load=500N) < material.yield / 2

    require Manufacturable:
        dfm.cnc.tool_access(all_features, tool_diameter=6mm)
```

- `process:` binds the part to a process module; features must come from that
  process's vocabulary (or generic features valid for it).
- Process modules are imports: `from processes.sheet_metal import Blank, Bend,
  Pierce`.

### 4.2 Variables and the constraint graph

A dimension is a typed node in a constraint graph, not a bare number:

```
var wall_thickness: length in [1mm, 6mm] minimize
var fillet_r: length = 0.5 * wall_thickness        # derived, not free
slot = Pierce(shape=rect(6mm, var(15mm, 25mm)), from_edge=free)
```

- **Types:** `length`, `angle`, `force`, `pressure`, `mass`, `ratio`, … with
  full dimensional analysis at parse time. Unit errors never reach a solver.
- **Domains:** hard bounds; violating them is unsatisfiable, not clamped.
- **`free`:** "let DFM decide the minimum/legal value."
- **Optimization direction:** `minimize` / `maximize` as secondary objectives
  after all requirements are satisfied.
- **Resolution policy:** DFM-driven constraints are **eager** (closed-form
  propagation on the Feature IR). FEA-driven constraints are **lazy**
  (outer optimization loop proposing values, inner FEA evaluation; surrogate
  models / adjoint sensitivities planned to keep evaluation counts low).

### 4.3 Requirements (`require` blocks)

The hardware analog of tests. Named, grouped, individually reportable:

```
require Structural:
    fea.max_von_mises(part=Body) < material.yield_strength / 2.5
    fea.max_displacement(face=MountFace) < 0.1mm

require Manufacturable:
    dfm.sheet_metal.bend_relief(all_bends)
    dfm.sheet_metal.min_flange(all_flanges) > 4 * stock_thickness

assembly LinearActuator:
    require kinematics:
        travel_range >= 150mm
    require envelope:
        fits_within(box(50mm, 50mm, 250mm))
```

Build output reports pass/fail per requirement, like a test runner.

## 5. Features and the Entity Database

### 5.1 Topology as a database

After each feature executes, the body's topology is an **entity database**:

```
faces:  id | origin_feature | current_owner | type    | normal | area | tags
edges:  id | origin_feature | current_owner | type    | dir    | len  | adjacent_faces
verts:  id | origin_feature | current_owner | ...
```

Entity IDs are internal only — never written in source. All source-level
references are **queries** against this database.

### 5.2 Features as queries + transformations

A feature (a) queries entities, (b) performs a geometric transformation,
(c) inserts/updates entities and **ownership**, (d) exports **named views**:

```
boss = Extrude(profile=circle(d=10mm), height=5mm, on=shell.top)
# exports: boss.top, boss.side, boss.base_edge, boss.axis (virtual)
```

Exports are filtered views into the database (entities this feature created or
modified), not raw topology references. This is the persistent-naming
("toponaming") solution: names are defined by origin and intent and re-resolve
against current topology.

### 5.3 Semantic selectors

Queries compose from geometric predicates:

```
shell.edges.where(parallel_to=Z)
shell.edges.where(adjacent_to=shell.top)
shell.edges.where(between=(shell.top, shell.front))
edges.where(from_feature=cutout)
shell.edges.where(parallel_to=Z).longest
body.top_edges.current - slot.all_edges          # set algebra
edges.at_intersection(slot, base.top)            # explicit join
```

Sugar form: `body.edges[adjacent=top, dir=Z]`.

**Query validation is static:** predicate names, entity kinds, and operand
types are checked at parse time, before any geometry exists.

### 5.4 `.original` vs `.current`

Feature exports are snapshotted and live:

- `f.top_edges.original` — entities as they existed when `f` first evaluated.
- `f.top_edges.current` — entities currently satisfying the semantic query,
  after all intervening features.

Where the distinction matters (the "fillet extension" problem), the source must
say which is meant; defaulting silently is forbidden in ambiguous cases.

## 6. Ownership and Borrow Model

The novel core of the language. Inspired by Rust: ambiguity and aliasing are
the source of bugs; eliminate aliasing in geometric selection statically.

### Rules

1. **Single ownership.** Every entity has exactly one owner: the feature that
   most recently created or modified it. Modification transfers ownership.
2. **Unambiguous queries.** A query must resolve to a deterministic set whose
   cardinality matches consumer expectations. Over/under-matching is a compile
   error; `.all` explicitly accepts "everything that matches."
3. **No implicit cross-ownership selection.** Selecting entities at the
   interface of two features requires explicit join syntax
   (`body.top ∩ boss.bottom_face`, `edges.at_intersection(a, b)`).
4. **Borrows.** A feature consuming a selection holds an immutable borrow on
   those entities. A later feature that would modify borrowed entities is a
   **borrow conflict** — a compile error with suggested resolutions
   (reorder features, `rebind()` after the modifier, or narrow the original
   query's scope).

### Scoping: `with` blocks

```
with body.top as mounting_face:
    boss  = Extrude(circle(d=20mm), height=8mm, on=mounting_face)
    holes = HolePattern(circle(d=5mm), pattern=bolt_circle(d=35mm, n=4),
                        on=mounting_face, depth=through)

with boss:
    taper = Draft(faces=boss.side, angle=2deg, from=boss.bottom_edge)
    blend = Fillet(where=boss.top_edge, radius=1mm)
```

A `with` block (a) declares the region a feature group operates on,
(b) makes out-of-scope selection a static error, (c) documents intent.

### Error model

Diagnostics are Rust-style and geometric: show the query, the matched entities
with origin and dimensions, and 2–3 concrete fixes (narrowed query, reorder,
`.all`, `rebind()`). Error codes are stable (`E0301` ambiguous selection,
`E0302` borrow conflict, …).

### Implementation note

The full borrow check runs on the **Feature IR** using predicted topology
effects of each feature class — before the geometry kernel runs. A second
verification pass after geometry evaluation confirms predictions (features
whose topological effect cannot be statically predicted must declare it).

> **Open question:** how to handle features whose topological outcome is
> data-dependent (e.g., a fillet radius large enough to consume an adjacent
> face). Likely answer: conservative static prediction + mandatory post-geometry
> verification, with failures reported against the source query.

## 7. Geometry Expression Tiers

1. **Parametric primitives** (always available): `rect`, `circle`, `polygon`,
   `slot`, …
2. **Constraint-based sketch language** (opt-in): SolveSpace-style constraints
   (symmetry, tangency, dimensions), not coordinate soup. Coordinates allowed
   but discouraged for anything parametric.
3. **Import escape hatch:** `geometry: import("casting.step")` plus semantic
   annotation of imported faces so the query/ownership system can operate on
   foreign geometry.

## 8. Electrical Track (sketch — needs its own spec pass)

Same architecture, different vocabulary:

```
schematic PowerBoard:
    process: pcb_2layer
    stackup: jlcpcb_2l_standard

    net VCC_3V3:
        source: U1.pin3
        sinks: [C1.p1, C2.p1, U2.VDD]

    constraint trace_width(VCC_3V3): IPC2221.current(500mA, 1oz_copper)
```

- Nets/components/footprints take the role of features/entities.
- DFM ⇒ design-rule packs (IPC-2221, fab capability files).
- Backends ⇒ Gerber/drill, pick-and-place, netlists.
- **Open question:** how much of the ownership model maps onto layout
  (zones, courtyards, keepouts are plausibly "owned" regions).

## 9. Solver Architecture (summary)

1. **Geometric constraint solver** — sketch-level (drive an existing engine,
   e.g., SolveSpace's solver).
2. **DFM constraint propagator** — eager, mostly closed-form inequalities from
   process modules (min bend radius ≥ k·stock, edge distances, tool access).
3. **FEA-driven optimizer** — lazy outer loop over free variables; inner
   mesh/solve/check loop; termination = all requirements satisfied, then
   secondary objectives. Surrogate models and adjoint sensitivities are the
   planned path to tractability. (Detailed in the Part 2 / FEA harness spec.)

## 10. Open Design Decisions

| # | Question | Leading candidate |
|---|---|---|
| 1 | Language/file naming | `dcad` / `.dcad` |
| 2 | Statically unpredictable topology changes (large fillets consuming faces) | conservative prediction + post-geometry verification |
| 3 | Default for bare feature exports: `.original` or `.current`? | error when they differ; force explicit choice |
| 4 | Ownership granularity: per-entity vs per-region (face areas) | per-entity first; regions later for split faces |
| 5 | Assemblies × borrow model: mates, cross-part constraints | parts are ownership islands; mates are explicit joins |
| 6 | Determinism/reproducible builds: solver outputs pinned in a lockfile? | yes — `dcad.lock` pinning resolved variable values |
| 7 | Query grammar: method chaining vs bracket sugar vs both | both, one canonical form for formatting tools |
| 8 | Electrical ownership semantics | unresolved |
| 9 | Embedding host language for logic (loops, functions) | restricted built-in constructs first; escape to Python plugin API |

## 11. Implementation Roadmap

**Phase A — Spec hardening (no code)**
1. Write 5–10 real parts in target syntax (sheet metal bracket, machined
   housing, PCB, weldment, molded clip). Awkwardness = design bugs.
2. Define the variable/unit type system precisely.
3. Define the entity DB schema and the query grammar (EBNF).

**Phase B — Static core**
4. Constraint-graph + entity-DB data structures (Python).
5. Parser (lark or tree-sitter) for a minimal subset: one part, primitives,
   `Extrude`/`Pocket`/`Fillet`, queries.
6. Query validation + ownership/borrow checking on the Feature IR.
   **This works before any geometry exists** — it's the first shippable
   artifact (a linter for `.dcad` files).

**Phase C — Geometry vertical slice**
7. Drive CadQuery/Build123d from the Feature IR → STEP export.
8. Post-geometry verification pass (confirm static topology predictions).
9. One eager DFM rule pack: sheet metal (closed-form, well-documented).

**Phase D — Goal-oriented resolution**
10. Eager DFM propagation resolving `free` variables.
11. Lazy FEA loop with a trivial physics model (hand-rolled 1D/2D first),
    then integrate the Part 2 harness.

**Later:** Rust migration of hot paths, electrical track, UI.
