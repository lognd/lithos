## D243. Injections are declared inputs, never evidence (AD-40; charter 42; WO-129)

Owner directive: graphite must let a high-skill engineer inject
decisions at the intermediate steps and move things around, with
optimization removal permitted but overridable -- and the allowances
must land in lithos FIRST (graphite consumes only public surfaces,
D234). Design authority delegated to the coordinator; the rulings
are charter 42 secs. 1-5.

The load-bearing rule, from which the rest follows: an injection
changes an INPUT or a CHOICE; the pipeline re-derives every
obligation from it. There is NO path from an override to an evidence
value, a margin, a verdict, or a waiver. An override can make a
design fail -- it can never make a failing design pass (INV-33).
That property is precisely what makes it safe to hand this channel
to a GUI, and it is why the answer to "let the engineer override the
optimizer" is yes without weakening D206/D220.1: superseding the
search changes the input, and the claims are re-proved against it.

Optimization removal is allowed, explicit, and never inferred:
`mode = "pin"` takes the variable out of the search (recorded, with
author and reason, in the parity ledger, the acceptance ledger, and
the calc-book audit index); `mode = "seed"` keeps it searchable from
that starting point. Unexplained overrides (no author, no reason)
are refused -- an audit trail that a GUI can silently write to is not
an audit trail.

   not by the channel behaving itself.

   LOWER factor would weaken a claim without touching any verdict

for the right reason: the language ALREADY defines a hint as
droppable and never load-bearing, so a hint injection cannot change

evidence-carrying, audited path -- the waiver machinery (WO-98's
acceptance ledger, D207 memos). Never the injection channel.

than by review: an override cannot name an evidence-ladder or

## D250. Safety honesty in power: no typical values, no compliance claims, no uncertified arc flash

Second, and this is the cost of honesty, stated plainly: refusal 1

## F154. WO-131's exemption list is keyed by LINE NUMBER, so it rots on any edit above it

The bare-string sweep WO-131 landed (D247.4, the check that makes the
one-code-space rule able to FAIL) records its 37 deferred exemptions
as `(relpath, lineno)` pairs. Merging WO-127 and parking the injection
channel shifted lines in the swept files, and the sweep went red
asserting `'kind="' in lines[lineno - 1]` against
`return Err(` -- the exemption pointed at the wrong line.

The check is right; its KEY is wrong. A line number is not a stable
identity for a call site: any edit above it moves it, so the
exemption list is guaranteed to rot, and a check that goes red for
reasons unrelated to what it guards is a check somebody eventually
disables. That is how a gate dies -- not by being wrong, but by being
annoying.

Fix: rekey the exemptions on something stable -- the (file, raised
kind string) pair, or file + enclosing symbol. The count still shrinks
monotonically (F-WO131-2's worklist), but a refactor above a call site
no longer breaks the build.

Worth stating as a general rule, because this is the second
line-number-brittleness in two cycles (the golden-diff review rule has
the same shape): NEVER key a gate on a line number. Keys must survive
a refactor, or the gate is a tax on unrelated work.

## D254. The aesthetic layer: graphite's one write surface (charter 42 sec. 10)

The owner parked the injection channel and ruled: all configuration
through graphite is AESTHETIC ONLY. Design authority delegated; this
is the resulting design (charter 42 sec. 10 is the normative text).

THE TEST THAT DECIDES EVERY CASE, and it is the whole design:

> Would a competent engineer need to RE-CHECK anything because of
> this edit?

Yes -> SEMANTIC, a GUI may not write it. No -> AESTHETIC, a GUI may.

Moving a component placement moves DRC clearances, thermal coupling,
EMC behavior. Moving a BDF block moves ink. The test asks about
CONSEQUENCES rather than file types, which is why it stays correct as
new artifact families arrive -- a rule keyed on "is this a diagram?"
would rot the first time someone invents a diagram that carries
geometry.

The aesthetic list is CLOSED by design (diagram layout; sheet
annotation/view ANCHORS -- not the note's text; style/theme
selection; view state). An open-ended "aesthetic" category is exactly
how a GUI eventually writes a dimension.

MECHANISM: aesthetic config is a hash-pinned RECORD (`view.toml`),
reusing the style-pack precedent (charter 41 sec. 5) rather than
inventing a channel. It is an input to RENDERERS only. There is no
code path from it to the payload, an obligation, an evidence value, a
verdict, or the DESIGN hash. So:

- the design hash is INVARIANT under aesthetic config, by
  construction -- two packages with identical designs and differently
  arranged diagrams carry the SAME design hash, DIFFERENT artifact
  hashes, and a manifest that says which view pin produced which
  picture;
- verdicts are untouchable by UNREACHABILITY, not by discipline --
  the same argument D246 makes for the evidence ladder, and the
  reason both are safe to hand to a GUI;
- determinism holds per view pin (INV-10).

The write verb is `regolith view set|list|reset [--json]` -- graphite
never writes the file directly, and a target outside the closed list
is refused with a code.

NOT ENABLED, deliberately: an engineer who wants to pin a dimension,
choose a component, or supersede the optimizer edits the SOURCE and
gets a diff, a review, and git history. That is not friction to be
smoothed away -- it IS the audit trail. The parked channel exists on
`experimental/injection-channel` if that judgment is ever revisited.

Note the correction this forced: sec. 7's EDIT models went with the
injection channel (they described placements and poses -- semantic
entities that existed to be written back). The aesthetic layer needs
a different, narrower object: a VIEW model that describes ink. Naming
them the same thing would have smuggled the semantic surface back in
under an aesthetic label, which is precisely the failure this ruling
exists to prevent.

## D255. Standard families are declared and checked: mixing IEC apparatus with NEC sizing is a diagnostic, not a silent smoothing

Raised by WO-134B: every transformer row carries
`standard_family = "IEC"` (10kV/20kV classes, Dyn11/Yyn0, IP20/IP21,
metric), while std.power's conductor ampacity, grounding, and motor
families are NEC 2023 / ANSI-NEMA. Today nothing stops a design from
combining an IEC transformer with NEC-derived conductor sizing, and
no diagnostic fires.

This is not pedantry. The two standards bodies carry different
assumptions -- ambient references, insulation classes, derating
conventions, fault-calculation methods -- and a design that silently
straddles them has a correctness hole that looks exactly like a
correct design. It is the same failure shape as a guessed fault
current (D250.3): the arithmetic is right and the answer is wrong.

Ruling:

1. Every std.power record DECLARES its `standard_family` (IEC | NEC |
   ANSI/NEMA | ...). WO-134/134B already do this -- it becomes law.
2. A design whose power net mixes standard families in a way that
   affects a claim's result emits a DIAGNOSTIC naming both families,
   both records, and the claim at stake. Not a warning: the author
   must see it and either declare the crossing deliberately (an
   explicit, reasoned `assume!` with basis -- the existing honest-
   deferral ladder, which is exactly what it is for) or fix it.
3. Mixing is NOT forbidden -- real plants do it, and a blanket ban
   would be a lie about engineering practice. What is forbidden is
   mixing it SILENTLY.
4. Cross-standard conversion tables are a NON-GOAL. The toolchain
   does not translate an IEC %Z into an ANSI assumption; it names the
   crossing and lets a qualified engineer decide. Inventing the
   conversion is precisely the "correctly-computed, lethally-wrong"
   move D250 forbids.

Machinery: folded into WO-133 (the power lowering + claim routing),
which is where a claim first sees both its apparatus record and its
conductor record. The diagnostic joins the E09xx/E11xx-era code space
(WO-131) -- a real code with explain content, never a bare string.

## F156. The unit token is thrown away in Rust lowering -- one line, 14 call sites, three workstreams blocked
