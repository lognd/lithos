## F135. Artifact-presentation recon: the shipped sheets fail a professional bar in four ways (visual audit, evidence on disk)

The coordinator rendered representative shipped artifacts to PNG
and inspected them by eye (the D238 proof discipline, applied first

1. MECH DRAWING (`demo3_removal_ribbed_panel/ribbed_panel_drawing
   .pdf`): dimensions are floating raw text (`bbox.depth=80.0000mm`

## D237. Debug instrumentation + hardware bring-up doctrine (AD-38; charter 40-debug-and-bring-up.md; WO-125..127)

Four rulings:

1. DEBUG IS AN EMISSION PROFILE, NOT A DIALECT. `regolith build
   --profile debug` / `ship --profile debug` augments EMITTED
   artifacts (board tap header + test points, firmware trace
   hooks, HDL tap taps) without touching design source, semantics,
   claims, or verdicts. The default (release) artifact set stays
   byte-identical when the profile is off -- determinism (AD-6/
   INV-10) holds per profile.
2. TAPS DERIVE FROM THE SAME TRUTH THE CENSUS READS. Every
   net/signal a claim names is a tap candidate; explicit taps are
   declared in the ship spec block's `"debug"` entry (the WO-102
   spec-block idiom -- no grammar change). Grammar-level `tap`
   statements are DEFERRED with a reopen criterion: a tap need
   that cannot be named by net path from the spec block.
3. THE BRING-UP HARNESS PACK IS AN ARTIFACT FAMILY (`dist/<proj>/
   harness/`): tap map, bring-up procedure document, capture
   configs for the analyzer, and an expected-signal manifest
   sourced ONLY from discharged claims / calc-book sheets /
   declared records (D224: never a fabricated expectation; an
   unverifiable expectation is a named absence with a reason).
4. THE TEST HARNESS HARDWARE IS ITSELF A LITHOS DESIGN. The
   logic-analyzer-class jig is authored in cuprite, ships through
   the same pipeline (census, calc book, gerbers, firmware), and
   mates the tap header via ONE shared pinout record (single home,
   AD-37 spirit) -- dogfooding is the acceptance test. New
   invariant INV-32 (tap-map/artifact agreement: every tap row
   exists in the emitted artifacts and every emitted tap is in the
   map) lands WITH its proof argument in WO-125.

## F136. WO-124 close-out: no pad-stack/courtyard/polarity data in the realized surface (escalated, no schema bump taken)

`RealizedLayout.placements` (`Placement`) carries `reference`,
`footprint` (a name string), `position_mm`, `rotation_deg`, `side` --
enough to place a refdes label on silkscreen, not enough to derive
pad-stack apertures (soldermask/paste openings), drill holes,
courtyard outlines, or polarity/pin-1 marks: none of those facts are
footprint-registry data this repo resolves anywhere yet (no footprint
library resolution exists -- `kicad_wrapper.py`'s own longstanding
documented cut). WO-124's fab-set exporter (both legs) therefore
emits those layers/files as honestly EMPTY-but-valid (D224: never a
fabricated pad or hole) rather than inventing geometry.

This is NOT the D239 schema-bump trigger: the gap is footprint-
registry-shaped (a future pad-stack record + resolution pass), not a
missing `RealizedLayout` field -- `Placement`'s existing fields are
sufficient for everything WO-124 could honestly deliver (refdes
labeling seam, board identity). No agent touched `SCHEMA_VERSION`.
Reopen criterion: a footprint/pad-stack registry resolution pass
lands (mech's stdlib registry precedent, or a WO-24-remainder
dispatch) -- at that point mask/paste/courtyard/drill genuinely have
data to draw from, and this finding's scope becomes real content
instead of an honest absence.

## F137. WO-124 close-out: no design-revision concept anywhere in the realized surface

Charter 41 sec. 3's board-identity block names three fields: name,
rev, design short-hash. `RealizedLayout`/`Lockfile`/`BackendInputs`
carry a design's netlist-hash-derived short-hash and its subject name,
but no "revision" concept exists anywhere in the schema, the lockfile,
or project metadata -- there is no field a `REV:` line could honestly
read. WO-124 prints `REV: N/A` (an honest absence label, matching the
project's existing `status="unrouted"` convention -- never a
fabricated value like `REV: A` or `REV: 1`). Reopen criterion: a
project-level revision/version concept is designed and lands as
project data (likely `magnetite.toml` metadata, cross-repo with any
future release-tagging work) -- at that point the identity block reads
a real value instead of the absence label.

## F139. Cycle-36 fuzz campaign record (D240): one real parser panic found, fixed, and the bound fixed a silent truncation bug

Campaign (15 min per target, seeded from `examples/`, cargo-fuzz
0.13.2 on the pinned-workspace-detached nightly):

1. `fuzz_lexer`: 900s, ~5.0M execs, cov 660, CLEAN.
2. `fuzz_cbor_decode`: 900s, 4.60M execs, cov 662, CLEAN.
3. `fuzz_parser`: CRASH at ~13 min -- "attempt to exponentiate
   with overflow", minimized to 18 bytes (`ex\n a:3V - 90mV9,b`):
   `Unit::parse_atom`'s trailing-exponent path ran
   `scale.pow(exp)` with any source-controlled i32 (`mV9` ->
   (1/1000)^9 -> 10^27 > i64), a parser-stack panic violating
   AD-3. FIX (merged 41b2b3d): `MAX_UNIT_EXPONENT = 12` (largest
   real corpus exponent is `s8`; 1.5x headroom) checked BEFORE any
   pow, plus a checked i64 pow behind it, both returning the new
   `UnitError::ExponentOutOfRange` -- error as data. Regressions
   committed at the qty level (mV9/mm2147483647/mV99/TV12 all Err;
   m12 round-trips; m13 rejected) and the syntax level (the
   byte-exact reproducer parses without panic). RE-RUN post-fix:
   900s, 735k execs, cov 4365, CLEAN.
4. BONUS REAL BUG the bound exposed (WO-122 class): `W40` used to
   parse as watt^40 (scale 1), so the claim-text canonicalizer
   silently truncated the identifier `SAE_10W40` to `SAE_10` in
   dune_buggy's `film` claim obligation. With `W40` rejected the
   identifier survives; the one-key dune_buggy golden swap was
   attribution-proven (base-commit wheel passes old golden, fixed
