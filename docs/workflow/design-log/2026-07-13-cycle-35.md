## D220. The rigor doctrine (law of the cycle)

Ruling: "engineering-firm rigor" means every claim whose physical
content is modelable DISCHARGES through a real registered model
over real declared inputs, and the waiver set shrinks to a CLOSED
set of named classes. Concretely:

1. Verdict math, trust floors, INV-2/12/13/24/26/28, D195, and
   D206 are all untouchable. Rigor is achieved by ADDING models,
   routing, inputs, and lowering surface -- never by weakening any
   gate or fabricating any window.
2. Permitted terminal waiver classes (everything else burns down):
   (a) structural conformance edges (Class A); (b) D195-gated
   conformance windows (owner queue); (c) named machinery
   exclusions carrying a design-log F-number and a reopen
   criterion; (d) author-intent exclusions (e.g. the two tracks
   corpus intended-behavior files).
3. The fleet census golden gains PER-CLASS accounting (discharged /
   waived-by-class / deferred), so the debt is visible, and the
   health gate fails on regression: any obligation moving from
   discharged back to waived, or any new waiver outside the closed
   classes, is a health failure.
4. No fabricated inputs: every datum added to the corpus follows
   D224. A claim that cannot honestly discharge stays honestly
   deferred-and-waived under 2(c) -- a smaller, named residue.

## D221. The audit trail is a first-class artifact: the calc package

Ruling: every shipped `dist/` package gains a CALCULATION REPORT
family (`calc/`), the artifact an engineering firm calls the calc
book, produced through the D208 registry seam (its first

1. One calc sheet per DISCHARGED obligation: the claim (source text

   zero unexplained rows. The index cross-links the WO-98

3. Forms: canonical JSON (hashed, deterministic) + rendered PDF
   through the existing style-pack renderer seam + a row in the
   package index. Byte-determinism rules unchanged (AD-6/INV-10).
4. No schema bump expected (payload-side JSON + package files); if
   an agent believes otherwise it escalates per D225.

## D224. Corpus data enrichment is authoring, not fabrication

Ruling: adding the missing inputs (Class D) to corpus designs is
legitimate design authoring under three rules:

1. PROVENANCE: every added value (a) comes from a std.*/registry
   record (preferred -- add the record if the part is real, e.g.
   bearing dynamic ratings from a manufacturer table), (b) is
   derived from already-declared design data with the derivation
   recorded in-file, or (c) cites an external datasheet/source
   in-file. NEVER reverse-engineered from the model to force a
   pass; a value chosen to make a margin positive without a source
   is fabrication and refused.
2. SAME-CHANGE BURN-DOWN: when a claim discharges for real, its
   waive block is DELETED in the same change and the project memo
   regenerated; a waiver shadowing a real discharge is debt.
3. HONEST FAILURES ARE FINDINGS: if real declared inputs produce a
   VIOLATED verdict, the design is wrong -- fix the DESIGN like an
   engineer (resize the member, pick the bigger bearing), recording
   the change; never touch the model or the window.
