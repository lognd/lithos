## D141. FOPEN-2 closed: compressible delivery is a discharge tier, not language surface

**The question** (fluorite/04): does compressible network solving
need new claim FORMS beyond `choked(edge)`?

**The decision (owner closure directive; argued against the
mantras).** NO new claim forms. `fluids.dp`/`fluids.pressure`/
`fluids.mdot` claims over gas subnets are already the right
vocabulary; what changes when line friction matters is WHO can
discharge them -- a compressible-capable model (Fanno-line network
tier, already enumerated in feldspar's `fluids` catalog) registering
under the SAME claim kinds and reporting its regime tags through the
D97 channel. The incompressible tier's validity is regime-guarded
(`fluids.mach(edge) <= 0.3` screening, the existing tag-shaped
form), so an out-of-regime margin honestly routes to the
compressible tier or an honest indeterminate -- exactly the fidelity
ladder every other domain rides. Unambiguous (the claim states the
requirement, not the method); intent-based (the margin picks the
tier); user-friendly (nothing new to learn). fluorite/01's v1
scope line is amended from "compressible network solving out" to
"compressible network solving is pack-tier content"; the corpus
gains a GN2 purge/blowdown fixture exercising the regime route
(WO-52). Feldspar's `fluids` compressible entries are scheduled in
its Phase 2 WO.

## D142. FOPEN-1's reopen question answered: mixing is declared-outlet components over mixture RECORDS

**The question** (fluorite/04, to be answered IF mixing ever
entered): mixture property records vs state-carrying nodes.

**The decision.** Records, never state-carrying nodes. The v1
single-medium-per-subnet rule STANDS (WO-49 implements its check);
what this closes is the design question the deferral held open, so
that admitting mixing later is scheduled work rather than a reopen:

- A mixing/junction component (`Mixer(outlet=<medium>)`, fluorite/02
  sec. 3) is a MEDIUM BOUNDARY: each terminal belongs to its own
  single-medium subnet; the outlet's medium is DECLARED and its
  property records are ordinary registry content. `Coolant60` (60/40
  EGW) has been a mixture-as-record since the first corpus file --
  the precedent was already load-bearing.
- The ullage/pressurant case (FOPEN-1's named expected first case)
  is a `Mixer`-shaped interface component on the tank: GN2 subnet on
  one side, propellant subnet on the other, the interface state
  carried by the component's declared parameters and records.
- State-carrying nodes are REJECTED: they would make node identity
  depend on solve results (composition as network state), breaking
  the static single-medium payload property and the corner
  discipline in one stroke.

Nodes stay stateless; media stay records; the checker stays
structural. fluorite/02 gains the `Mixer` row; fluorite/04's FOPEN-1
entry now records the answered design question and points its
implementation at WO-52 (parse + boundary treatment in the medium
consistency check), which lands beside WO-49's binding enforcement.

## D144. Pattern and mechanism libraries: reuse is stdlib content plus advisory recommendation (AD-28)

**The directive (owner, 2026-07-08).** "There's a ton of mechanical
mechanisms (and electrical patterns and so forth) -- make sure we
have a plan to easily include them, recommend them in place of other
things, and everything to make the design cycle easier on the
engineer."

**The decision.** Three legs, none of them new machinery -- the
whole point is that the regolith already carries this weight;
chartered in `docs/spec/toolchain/26-pattern-libraries.md`, ledger
rule **AD-28** (patterns are packages and advice; the compiler never
hard-codes a pattern, and a recommendation is never load-bearing):

1. **Content: pattern packs are ordinary packages** (regolith/11
   kinds `parts` + `interfaces` + `matings` + `models`, co-versioned
   halves). `std.mech.mechanisms` (four-bar/slider-crank linkages,
   lead/ball screws, belt/chain/gear drives, bearing arrangements,
   flexures, detents), `std.elec.patterns` (converter topologies,
   level shifters, sensor frontends, protection networks -- abstract
   `block` + `spec:` contracts with reference impls),
   `std.fluid.circuits` (relief legs, filter loops, accumulator
   stations), `std.civil.assemblies` (braced bays, wall/roof/floor
   type families). Every pattern is a contract with promises + a
   `spec:` behavioral law + model-half nodes, so consuming one buys
   verification content, not just geometry/topology. D64's
   kinematics closure said "v2 is a models-and-stdlib project" --
   this is that project, scheduled.
2. **Recommendation: `advise:`-severity recognition rules.** Pattern
   packs ship rules (the WO-28 engine, unchanged) whose match domain
   recognizes hand-rolled shapes ("a pivot pair whose `couples:` law
   matches a four-bar; a discrete transistor arrangement matching a
   level-shifter pattern") and whose `advise:` names the library
   pattern, its contract, and what the engineer gains (models,
   evidence, DFM coverage). INV-3 discipline: advice is
   verdict-inert, never an obligation -- the engineer stays
   sovereign. No new severity, no new engine.
3. **Ergonomics: the existing tooling surfaces carry it.**
   `magnetite new --template <pattern>` scaffolds (WO-41 machinery);
   `regolith doc` renders pattern datasheets from their contracts
   (the "connection datasheet" watchlist item finally lands
   somewhere); the LSP completes pattern contracts from imported
   packs (AD-24, no new channel). Machinery: WO-53 (charter
   ratification + the first `std.mech.mechanisms` slice +
   the recognition-rule seam proven end-to-end on one mech and one
   elec pattern).

Feldspar's side: the mechanism model halves consume its `dynamics` /
`mech.design` catalog entries (four-bar kinematics, cam dynamics,
power screws, belt/chain capacity); the demand signal is recorded in
its capability map and rides its phase WOs.
