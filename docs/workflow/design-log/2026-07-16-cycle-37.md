## F-TBD / D-TBD. The D256 hash window: resolve_unit_suffix preserves the unit token

### The fix

`crates/regolith-lower/src/claims/common.rs::resolve_unit_suffix`
parsed a declared bound's unit through `regolith-qty`'s `Unit::
parse_expr` for its SI scale factor and then discarded the token
(`45ohm` -> `45`). It is the ONE helper all 14 call sites route
through (comparison bounds, window halves, sweep domains, temporal
window durations, `floor(...)` masks), so the fix lands once.

`Unit` gained one new method, `base_symbol()`
(`crates/regolith-qty/src/unit.rs`): the canonical unprefixed SI
symbol for a unit's dimension, found by matching the unit's dimension
against the fixed unit table's unprefixed, non-offset rows (`mV` ->
`V`, `kN` -> `N`, `ohm` -> `ohm` unchanged; a compound dimension the
table names no single entry for, e.g. `N/m`, returns `""` -- the
caller omits the suffix rather than fabricate a spelling the crate
would not itself accept back as input). `resolve_unit_suffix` now
appends this symbol after the SI-reduced magnitude instead of nothing:
`45ohm` stays `45ohm`; `20mV` reduces to `0.02V` (the BASE symbol,
never the original prefixed spelling, which would mislabel the
reduced value); `degC` resolves through its additive offset and
carries `K` (its dimension's base unit); a suffix `regolith-qty` does
not recognize (`rpm`, `dB`) is untouched exactly as before.

### Representation choice (documented per the dispatch brief)

Two shapes were on the table for the lowered obligation text:

1. **Attach the unit directly to the magnitude** (`45ohm`, chosen).
   Every existing Python-side bound reader already treats a trailing
   unit token as its PRIMARY shape: `_resolve_bound`/`_split_bound_term`
   (WO-122, `translate.py`) already parse an attached-or-detached unit
   suffix off a bound's text, because a unit `Unit::parse_expr` does
   not recognize (`rpm`, `%`, `dB`) was ALREADY passing through this
   way pre-D256. `unit_from_claim` (`backends/calc.py`) and this WO's
   own `_split_expected_magnitude_and_unit` (`backends/harness_pack.
   py`) are the same shape. Choosing this representation meant ZERO
   Python-side parsing changes for the bound-resolution path itself --
   only the two closed-vocabulary FALLBACKS (now dead code paths since
   the primary path always succeeds) needed deletion.
2. A separate structured unit field on the obligation/claim schema
   (an L3 payload bump) -- REJECTED: out of this window's scope by the
   dispatch brief's own instruction (a schema bump requires escalation
   and D256.3 already ruled the cycle-36 D239 schema window stays
   closed and unspent; nothing in this window needed a new field
   because the existing text-based readers already tolerated this
   shape).

No schema bump was needed or taken. The L3 payload schema is
byte-identical before and after this window.

### A real bug this exposed and fixed in the same commit

With the unit now attached to the claim's own `rhs`, the evidence
surface's `expected` (bare-value) and `units` (unit-only) fields would
otherwise both carry the unit, doubling it in `bringup.md`'s
`"expect <value> <unit>"` line (`45ohm ohm`). `harness_pack.py`'s
`_expected_text`/`_units_of` pair (WO-128) is replaced by one function,
`_expected_magnitude_and_units`, that splits the claim's declared
threshold into magnitude and unit ONCE -- `expected` stays the bare
magnitude the L3 schema already commits to, `units` stays separate.
Caught by the existing unit test fixture (`_rail_obligation`'s
`rhs="3.465V"`), not discovered live -- the pre-existing `elec.
impedance` corpus claim happened to have its unit stripped by the bug
this window fixes, so the duplication was latent until now.

### D256.4: the interim closed-SI-vocabulary channel, deleted

WO-128 (cycle 36) landed a fallback -- `translate.si_output_unit` +
`_SI_TERMINATION_UNIT`, a hand-maintained `(scheme, selector) -> unit`
table for `elec.impedance`/`elec.termination` -- because the Rust bug
this window fixes made the claim's own unit token unreachable. D256.4
required its deletion once the root fix landed, since two live unit
channels is exactly the "no duplication" violation the project treats
as a bug waiting to desync.

Deleted: `si_output_unit`, `_SI_TERMINATION_UNIT`,
`si_sheet_fields`'s `"unit"` dict key (translate.py);
`calc.py::_si_unit_from_obligation` and its position in `_build_sheet`'s
unit-fallback chain; `harness_pack.py`'s `si_sheet_fields`-based
fallback branch inside the old `_units_of`.
