## D257. DigiKey is discovery-only; committed parametrics come from manufacturer datasheets under structured citations

The directive asked for a "digikey extension" carrying every value in
the parts' datasheets. The DigiKey API User Agreement forecloses the
literal reading: sec. 5.1(e) forbids using the API or DigiKey Data to
"update or create your own database of information", 5.1(d) forbids
bulk download, and 3.2(iv) forbids redistributing DigiKey Data. A
committed stdlib corpus populated from the API is exactly the
prohibited database, in a public repo. The owner confirmed the split
(2026-07-16): DigiKey serves DISCOVERY ONLY, at runtime, through a
live loader whose output is attributed to DigiKey (sec. 3.1.4) and
never committed; ALL committed parametric data is transcribed from
MANUFACTURER DATASHEETS, which are not DigiKey Data, under per-value
citations. The two loaders stay separate BY CONSTRUCTION so no API
response can reach the repo. This is recorded as law: any future
design that caches DigiKey API responses into the tree is a violation,
not an optimization.

Rulings from the processors recon (scratch_recon_digikey_processors.md,
2026-07-16), adopted:

1. Processor catalogs are vendor-named record packages (ti.mcu,
   microchip.mcu, ...) beside the existing ti.logic / st.mcu /
   microchip.cpld -- the ordinary magnetite mechanism, never a new
   plugin kind, never the AD-26 seam (which is behavior, not data).
2. Citations become STRUCTURED: the one justified divergence from the
   evidence = { method, trust_tier, reference } house shape is
   decomposing the prose reference string into fields (document,
   revision, date, page, table, url), so the charter 39 sec. 5.4
   health check can tighten from "string non-empty" to "page cited,
   doc/rev known". The per-field precedent already exists:
   transformer_dry_type's xr_ratio_evidence (D224). Uncited datasheet
   values become unrepresentable at the model level (no constructor
   without a citation).
3. Datasheet ingestion is HUMAN-IN-THE-LOOP transcription with a
   machine assist, confirmed against the RENDERED page (the WO-134B
   lesson, independently reproduced by the recon's own prototype
   misreading a wrapped VCC+0.3 abs-max bound). No automated corpus
   factory; a mandatory confirmed gate rides every generated row.
4. First slice: the citation models + health tightening, then ONE
   package (ti.mcu), ONE family (MSP430FR5, TI SLASE54D Rev. D),
   uniform core only (~30-60 cited values per part), tier=community.
   Assigned WO-145 (after the fluids block, D258 ruling 4).

Owner items parked: ST.com bot-wall (browser fetch likely clears it;

## D259. Graphite may author source; it may never edit semantics

The owner directive (scan -> size via reference points -> trace ->
basic CAD -> pipeline) lands against D253's law that semantic edits
stay source-only forever. The ruling that reconciles them, made
before the design recon reports so the boundary is fixed while the
design is drawn:

D253 forbids the GUI from EDITING an existing design's semantics
through an override or injection channel that bypasses source. It
does not forbid the GUI from AUTHORING NEW SOURCE. A trace tool whose
entire output is a declarative source artifact -- written to disk,
reviewed by a human, committed, compiled through the ordinary
pipeline -- IS the source-only path; the GUI is a source emitter, a
keyboard with better ergonomics, never a runtime overrider. The
boundary is structural, not procedural: graphite's only contacts with
the pipeline are (a) reading the D244 artifact index and (b) saving
source text that the user then compiles. Any design element that
needs a runtime channel from GUI into the compiler is a violation and
an escalation, not a trade-off.

A traced profile is CITED GEOMETRY: its provenance record carries the
scan (file hash, capture date), the calibration (reference points,
known distance, residual error), and the tracer (human, date), in the
same structured-citation family as D257 ruling 2. A trace without its
scan provenance is as unrepresentable as a datasheet value without
its page. The design recon (in flight) fills in the source
representation, the CAD v1 closed set, and the WO decomposition under
this ruling; the declarative non-GUI path remains first-class -- every
GUI action must have a readable source-text equivalent, and the GUI
must be able to reopen its own emitted source (round-trip as a design
requirement, not a feature).

Owner addition (same day, mid-window): calibration must climb past
two reference points to a full ladder -- uniform scale (2 points),
projective homography (4 points, closed-form), and an N-point grid of
known pitch that back-calculates perspective PLUS non-linear lens
distortion (the Zhang planar-target formulation). The provenance
record carries which model was fitted, the grid spec, the parameters,
and the reprojection residual in mm at the object plane; the source
artifact stores image-space points and the calibration transform
SEPARATELY so a trace is auditable and re-calibratable without
re-tracing. A trace whose calibration rung cannot support a consuming
claim's accuracy is a diagnosable condition, not a surprise.

## D260. Authoring surfaces: where hand-writing source is awkward though possible, graphite may grow a front-end that emits it

Owner directive (2026-07-16): signal design should get a front-end
tool, and more generally "any point where it would be awkward --
ALTHOUGH POSSIBLE -- to hand-write" the source may earn one. The
ruling that generalizes D259 into a family:

1. An AUTHORING SURFACE is a graphite tool that emits declarative
   source a human could have typed. The D259 boundary is inherited
   verbatim and is the admission test: the tool writes source files
   and reads the artifact index; nothing else touches the pipeline.
   The emitted source is the unambiguous artifact (the mantra order
   is preserved: the GUI adds user-friendliness ON TOP of an
   unambiguous text form, never a second meaning); round-trip --
   the tool reopens its own emitted source -- is a requirement on
   every surface, not a feature.
2. NO BLIND FEATURES (owner's phrasing, adopted as process): each
   surface gets a design recon arguing every interaction against the
   mantras before implementation, and each surface's WO ends with a
   POST-COMPLETION FEATURE AUDIT -- a recorded design-log pass over
   what small adjacent affordances the finished tool now makes
   obvious, each argued in, deferred with a reopen criterion, or
   argued out. Feature ideas discovered mid-build are logged, not
   slipped in.
3. EVIDENCE HONESTY (the D246 interaction, load-bearing for
   signals): an authoring surface can only emit source whose
   evidence posture is AUTHORED. A hand-drawn expected waveform is
   design intent -- trust-tier authored/asserted -- and can never be
   dressed as a model-backed or measured expectation; the bring-up
   pack's expected_signals carry D224 provenance precisely so a
   drawn curve cannot forge it. The GUI cannot mint evidence it does
   not have, by unreachability, same as D246's "cannot forge a
   pass".
4. First instances: profile/scan tracing (D259, recon in flight) and
   SIGNAL DESIGN (waveforms, envelopes, masks, stimulus/expectation
   pairs -- the concrete owner ask). A census of further candidates
   (where the corpus shows hand-written source straining) belongs to
   the signal-design recon, prioritized by demo value, each with an
   honest "is text actually awkward here, or merely unfamiliar"
   check so surfaces are not built for their own sake.

## D262. No bare dimensioned values in artifacts -- enforcement by unreachability, not by sweep alone

Owner directive (2026-07-16, mid-window): "we had bare values in
artifacts; make sure that we enforce units to stop any dumb mistakes
... no dumb mistakes should EVER be possible to make."

The observed bare values are F156's family -- resolve_unit_suffix
discarding the unit token -- and the ROOT FIX is already executing in
the D256 hash window. This ruling is about the second half of the
ask: after the window closes, bare-ness becomes IMPOSSIBLE rather
than currently-absent. By D256's own law nothing rides in the window,
so this is the first post-window slice, assigned WO-150:

1. STRUCTURAL HALF (the real enforcement): every artifact-rendering
   interface that accepts a dimensioned value accepts a unit-carrying
   quantity type (regolith-qty Qty/Interval or the schema mirror),
   never a bare float-plus-hope. The renderer signature refuses the
   mistake at the type level, the same unreachability doctrine as
   D246 ("cannot forge a pass") and D257 ruling 2 (uncited values
   unrepresentable). Genuinely dimensionless values pass an EXPLICIT
   dimensionless unit, not an absent one -- absence stops being
   expressible.
2. SWEEP HALF (the rot guard): a health-consistency sweep over
   emitted artifacts (calc sheets, bring-up packs, drawings, BOM
   views) that flags dimensioned-looking bare numerals, as a
   regression tripwire for surfaces the type system cannot reach
   (e.g. prose strings). Report-only at first landing, promoted to
   error once the fleet is clean, per the F154 lesson applied in
   reverse: a gate promoted before it is satisfiable is a gate that
   gets waived.
3. The guarantee enters the invariant ledger with its proof argument
   in the SAME change (INV number coordinator-assigned at
   integration; the proof argument is ruling 1's unreachability plus
   ruling 2's sweep evidence).
4. Seam notes: the signal-design recon (D260) already carries the
   axis-unit discipline for authored waveforms; WO-145's Cited
   models carry units inside cited values; WO-147's .rgp schema
   declares mm explicitly. New surfaces inherit this ruling at
   design time -- a reviewer may reject any new interface that
   accepts a dimensioned bare float as a matter of law, citing this
   entry.

WO-150 depends on the D256 window (it builds on the repaired unit
channel) and precedes WO-143 (the Moody sheet renders quantities
through the enforced interfaces rather than being migrated later).
