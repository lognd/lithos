## D268 -- the capability extension program

Owner directive (2026-07-19): extend generation capabilities so the
toolchain can, in an automated pass, produce (1) hardened tool-steel
parts profiled on wire EDM, bolted to a mild-steel plate as a
stamping die set; (2) PCBs (already substantially real -- gerber fab
sets both tiers); (3) perf-board routing (fixed-grid substrate,
jumper/wire assignment, human-followable wiring map + cut list);
(4) dwelling/house wiring (circuits, panels, cable schedules) on top
of the WO-132..137 power track.

Rulings:

1. Every program lands through the AD-47 capability registry -- no
   capability ships without its process records, DFM checks, tool
   adapters (with AD-45 provenance), and claim vocabulary.
2. Prerequisites before the new programs' realizer halves:
   `RealizedLayout` put seam (A7) for anything board-shaped; the
   AD-44/AD-45/AD-46 charter tickets for everything.
3. Wire-EDM program needs: material STATE modeling (heat-treatment /
   hardness as a first-class material state, absent today -- language
   + stdlib work), a profile-cut program kind (2D contour + lead-ins,
   distinct from milling removal ops), die-set assembly composition
   (bolted joints exist, WO-72 precedent), and EDM process records +
   DFM (kerf, corner radii, taper, skim passes).
4. House wiring stays REPRESENTATION-first under the D250/D258
   licensing posture: rule MODELS with cited public-domain/owner-
   verifiable sources; no transcription of NEC tables beyond what
   the existing std.power citations already cover; refusals stay
   named. Breaker/panel content remains gated on verifiable offline
   sourcing (D250 sec. 3) unless the owner supplies sources.
5. Sequencing: charter tickets -> RealizedLayout seam -> perf-board
   (smallest board-shaped win, exercises the new seams end to end)
   -> EDM program (largest language surface) -> house wiring (rides
   power track WO-132..137 completion).

## D269 -- the stdlib process/DFM expansion program

Owner directive (2026-07-19): stdlib gains DFM + process records for
every process family we can honestly source. Rulings:

1. ONE process-record doctrine before population: a `std.process`
   record schema (capabilities envelope: materials, size limits,
   tolerance grades, surface finish, min features, cost drivers,
   lead-class) + a DFM check-set contract per family, both under the
   AD-37 naming/citation law and the AD-47 registry's
   `process_records`/`dfm_checks` fields. Schema first, data second.
2. Provenance regime (the D266 lesson, owner-flagged 2026-07-19):
   US-government/public-domain sources and general-engineering-
   knowledge values with explicit posture markers; citations
   render-confirmed where used (WO-145 machinery); NAMED REFUSALS
   instead of transcribing copyrighted handbook tables. Counsel
   items from D266 remain open and are not re-litigated here.
3. Taxonomy for population waves (each family = records + DFM checks
   + at least one demo-exercised check): subtractive (milling,
   turning, drilling, grinding, honing, lapping, sawing, wire EDM,
   sinker EDM, waterjet, laser, plasma, oxy); sheet (shear, bend,
   punch, stamp, deep-draw, roll-form, spin); casting (sand,
   investment, die, permanent-mold); molding (injection, blow,
   rotational, thermoform, compression); additive (FDM, SLA, SLS,
   DMLS, binder-jet); joining (TIG/MIG/spot/stick, brazing,
   soldering, adhesive, threaded fasteners, rivets, press fits);
   bulk forming (forging, extrusion, rolling, drawing); heat
   treatment (anneal, normalize, quench+temper, case-harden,
   nitride, stress-relieve -- ties into D268's material states);
   surface (anodize, plate, passivate, paint, powder-coat, black
   oxide); PCB (fab, assembly, reflow, conformal); electrical
   installation practice (the D268 house-wiring companion).
4. Population order follows D268 demo value: EDM + heat-treat +
   stamping first (the die-set program consumes them), then PCB
   assembly, then the long tail.

### Amendment to the process program (same day): exhaustive-research results folded in

The frontier-loop research pass (100 processes, 12 families, coverage
proven against DIN 8580's six Hauptgruppen; dossiers in the session
scratchpad, to be converted into the schema ticket's reference pack)
amends the sec. 3 taxonomy: POWDER METALLURGY (press+sinter, MIM,
isostatic) was omitted and is added as a family; the universe also
gains ECM, gear cutting, hydroforming, hemming, incremental forming,
cold heading, swaging, FSW, laser/ultrasonic welding, electroless/
PVD/CVD coating, shot peening, induction hardening, austempering/
martempering, and solution-treat-age (aluminum/titanium hardening --
the steel-centric framing missed it). Provenance reality check: the
population will be predominantly GEK-with-named-refusals (~78% GEK,
~20% carrying at least one named refusal, ~8% with spot-verified
PD-GOV anchors like MIL-H-6875) -- the record schema must therefore
make the GEK posture a first-class, owner-visible marker, not an
apology. Cross-family finding adopted into D268 sequencing: shot
peening enters the EDM die-set chain as recast-layer remediation,
and press-brake/stamping DFM lands with the die-set wave.

## D270 -- materials science program (owner directive, 2026-07-19)

Owner: feldspar should carry everything needed to pick a material or
justify a selection -- phase equilibria (eutectics, alloy systems),
crystal structure, TTT/CCT-class transformation behavior, thermo,
and price. Rulings:

1. MODEL/RECORD split per AD-37: feldspar gets the MODELS (phase-
   diagram thermodynamics, TTT/CCT prediction via published closed-
   form kinetics families -- Kirkaldy/Li-type diffusional onset,
   Koistinen-Marburger martensite, Grossmann ideal critical
   diameter, Jominy correlations, Hollomon-Jaffe tempering --
   each cited, calibrated against independently published oracle
   points); lithos stdlib gets the RECORDS (compositions, crystal
   structures, price classes) as cited community-tier data.
2. Licensing: NEVER transcribe ASM-handbook chart curves; predict
   from published equations instead. Price data from public-domain
   sources (USGS Mineral Commodity Summaries class) as cost CLASSES,
   never scraped vendor quotes. D266 counsel items stay open.
3. Selection/justification surface: a solver route from requirements
   (hardness, toughness class, section size, quench severity, cost
   class) to candidate materials with the full calc chain as
   evidence -- the material-selection justification artifact.
4. Tickets: feldspar T-0018 (filed + pushed 2026-07-19) = model
   half + record schema; the lithos stdlib record-population ticket
   is its companion (filed after the closeout lane releases
   tickets.md). The D268 EDM program's heat-treat states consume
   this program's outputs.

## D271 -- WO-146 ratifies the `.rgp` traced-profile format (T-0017)

Records, as the ratification act (not a re-argument), the D259 sec.
5 structural-boundary ruling this format complies with, verbatim
from `scratch_recon_graphite_cad.md` sec. 5:

> D253 (design log 2026-07-15-cycle-36.md:873-926) forbids the GUI
> from EDITING an existing design's semantics through an override/
> injection channel that bypasses source -- "SEMANTIC (source-only,
> never a GUI): ... dimensions ..." is about a GUI writing semantics
> into DERIVED or RUNTIME state so that the picture and the source
> disagree. It does not forbid the GUI from AUTHORING NEW SOURCE. A
> scan-trace tool whose output is a declarative source artifact --
> written to disk, reviewed by a human as a git diff, committed,
> compiled through the ordinary pipeline with full provenance -- IS
> the source-only path. The GUI is a source EMITTER, like a
> keyboard; it is never a runtime overrider.
>
> THE STRUCTURAL BOUNDARY (each clause is enforceable in code review
> and by tests, not by discipline):
>
> 1. graphite's studio writes SOURCE TEXT ONLY, under the project's
>    source tree, visible to git: the traced-profile file, the
>    pinned scan copy, and a `.hema` snippet. It never writes
>    derived artifacts, ledgers, lockfiles, waivers, overrides,
>    aesthetic view pins, or anything the pipeline reads other than
>    as source.
> 2. graphite talks to the build pipeline in exactly the two ways it
>    already does, and no third: it READS the D244 artifact index,
>    and it drives ordinary CLI builds through the existing run
>    console. Saving a trace does NOT trigger a compile; the user
>    reviews, then builds.
> 3. No new lithos surface accepts data from graphite at runtime.
>    The lithos-side work is a file format + its elaboration in
>    ordinary lowering -- the compiler cannot tell whether a
>    traced-profile file was written by graphite, by hand, or by a
>    vendor. (That indistinguishability is the proof the boundary is
>    structural.)
> 4. v1 writes NEW files and re-writes ITS OWN emitted traced-profile
>    files (round-trip); it never rewrites hand-authored `.hema`
>    text. Even the re-write is a source diff under git --
>    source-only holds.
> 5. D246 untouched: claims, evidence ladder, trust floors, `model=`,
>    `sf=` remain source-only and are not writable from the studio;
>    the studio emits geometry + provenance, never claims.
> 6. Calibration (including the multi-point/grid ladder and the
>    distortion fit) is part of AUTHORING PROVENANCE: the fit runs
>    in the studio before any source exists, its outputs are fields
>    of the source artifact, and the compiler only ever APPLIES
>    recorded parameters deterministically. No runtime channel is
>    created or needed.
