## A. Findings

**F97. `.calc` misreads; the naming ask is now demand-backed.** The
owner flagged `.calc` as reading "calculation package". Independently,
COPEN-1's ratification demand grew from one fixture to four (feldspar
G39: coolant loop, fuel feed, brake hydraulics + the original
feed_lines reproduction). Both the name question and the adoption
question are ripe at once.

**F98. The draft and its own demand fixtures drifted in five places**
(adversarial read of `docs/calcite/` v0 against
`../feldspar/examples/lithos/dune_buggy/*.calc`):
1. `brake_hydraulics` uses an `Imposer(...)` component class the
   draft's component list (02 sec. 3) never declares -- and it is the
   load-bearing one (a pressure imposer whose VALUE is another
   track's mechanical promise).
2. `from=` is overloaded: geometry source on `Pipe(from=part.role)`
   vs promise source on `Imposer(from=PedalBox...)` -- one word, two
   ideas.
3. `fluids.pressure.min(x, within ...)` is invented vocabulary; the
   settled quantity-core temporal forms are `peak/settles/rms/
   stays_within` (regolith/02 sec. 5), where `peak` is the worst
   excursion in the claim's ADVERSE direction (sense-driven), which
   the cooling fixture already uses correctly for a lower bound.
4. `circuit_f in {intact, failed}` declares a NET-LEVEL config
   variable inside `states:`, but the draft only specifies
   edge-parameter state domains (`mv.position in {...}`).
5. `forall circuit in {circuit_f, circuit_r}: ... given circuit =
   failed` quantifies over state VARIABLES (not values); the draft
   has no semantics for it.

**F99. Compliance extraction is load-bearing, not future work.**
COPEN-5 (wall compliance for water hammer) looked like a transient-
tier nicety; the brake circuit's `Compliance` budget (hose swell +
caliper flex + mount flex vs master displacement) makes the
extraction rule a v1 requirement of the circuit that most needs the
track.

**F100. The four schema asks are one coupled change.** Coverage
encoding (sec. 7 item 2), the payload-ref channel (item 3), given
resolution + regime tags (item 4), and claim-kind competition
(item 1) all touch `regolith-oblig`/`DischargeRequest` and the
harness selection path. Landing them as separate WOs means three or
four consecutive `SCHEMA_VERSION` bumps and three re-goldens; they
are one work order.

**F101. Cuprite's spec promises derived pin assignment; no WO builds
it.** cuprite/04 sec. 1 item 2 specifies pin-mux as a monomorphized
matching problem (alternate-function tables x flow demands ->
lockfile-caused assignment, constructive contention errors). WO-24's
deliverables cover binding, netlist, and the KiCad layout seam --
pin-mux assignment appears in NO work order. Same class of gap as
F96: a spec-promised derived output with no producer. The
"done-for-you" audit result otherwise: intent->block allocation
(cuprite/02 sec. 5), component binding + E-series snapping,
placement/routing/DRC via the realizer, and budget allocation
(`allocate: cost_optimal`) are all specified AND work-ordered.

**F102. Hose extraction and wire-run extraction are the same seam.**
Fluid parameter extraction (calcite-draft 03 sec. 1: area, length,
bends, roughness from realized geometry) and G42's wiring-harness
needs (conductor length, bundle membership, connector environment
from a routed path) are one mechanism: read a routed-path geometry
record, emit lowered givens cited to the geometry snapshot hash. Two
implementations would desync (the known failure mode).

## B. Decisions

**D93 (F97, F98, F99; closes COPEN-1). The fluid track is ratified as
`fluorite`, extension `.fluo`.** Name: fluorite is the mineral named
for flowing (Latin `fluere`; it named flux and fluorine) -- the one
mineral whose NAME is the track's semantics; it stays in the ore-
mineral scheme (hematite/iron/mech, cuprite/copper/elec). Extension
`.fluo` follows the `.cupr` first-four pattern and collides with
nothing. `calcite`/`.calc` join the dead-names list (legitimate only
in design-log history and the feldspar G-ledger's verbatim entries).
Ratification folds the F98 fixes into the normative spec
(`docs/fluorite/`):
- `Imposer(p=<expr>, driven_by=<promise ref>)` joins the component
  set: a pressure imposer whose value is an ordinary quantity-core
  derivation; `driven_by=` names a cross-track promise chain.
  Imposers count toward the one-imposer-per-subnet net check.
- `from=` means realized-geometry source ONLY (`Pipe(from=...)`,
  `Hose(from=...)`); promise-driven parameters use `driven_by=`.
- No `fluids.pressure.min`: the temporal vocabulary is regolith/02
  sec. 5 verbatim, and `peak` is normatively the worst excursion in
  the claim's adverse direction (sense-driven).
- Net-level config variables are declared: `state <name> in {...}`
  inside the flownet; `states:` holds edge-parameter domains and
  declared net states, both ordinary config domains.
- `forall <var> in {<state refs>}` over a finite set of declared
  state variables is admitted, lowering to ONE swept obligation with
  a discrete axis whose points are (state-variable, value) bindings
  -- carried by D95's coverage encoding, never enumerated into
  obligation copies.
- Compliance is a first-class edge parameter: record-bound
  (`Hose(compliance=registry(...))`) or EXTRACTED in lowering from
  the implementing part's wall record (E, thickness -> compliance and
  wave speed), cited to the geometry snapshot hash (closes COPEN-5:
  the extraction rule's home is lowering, same as every hydraulic
  parameter). `fluids.volume_consumed(<edge set>, at=<p>)` joins the
  claim vocabulary as the compliance-budget form.
- Leak budgets (closes COPEN-6): the circuit-level leak budget lives
  in fluorite (`fluids.leak_total(<subnet>) < x` as a budget over
  fitting/seal contributors); a component's seal leakage stays a mech
  interface promise consumed through the ordinary promise chain. One
  quantity kind, one budget owner (the flownet), two claim sites.
- COPEN-3 (medium mixing) and COPEN-4 (compressible networks) stay
  out of v1 as deferred-with-reopen-criteria (F90 style): reopen on
  the first corpus fixture that cannot be written as single-medium
  subnets (pressurant/ullage), resp. the first GN2 blowdown fixture
  needing Fanno solving; `choked(edge)` IS admitted now as a
  tag-shaped screening claim (reynolds precedent).
- COPEN-7 closes by reference into D95 (discrete axes in coverage).

**D94 (sec. 7 item 1; feldspar OPEN-6). Claim kinds are owned by the
claim vocabulary; models never mint kinds.** `mech.fea.static_stress`
was a bootstrap error: the kind names WHAT is claimed
(`mech.static_stress`), never the method or tier (model attributes).
Rulings: (a) packs register under vocabulary kinds; a kind string
containing a method/tool name is a registration lint error; (b) one
model MAY register under multiple kinds -- `ModelRegistry` keys
(claim_kind, model_id); the duplicate-model-id error is per-kind, not
global. feldspar flips its constructor-override default to the shared
kinds when this lands (its 06 already records the intent).

**D95 (sec. 7 item 2; feldspar OPEN-8, G29, G43; COPEN-7). Coverage
becomes structured and per-axis.** `regolith-oblig` replaces the bare
coverage float with `Coverage { axes: [CoverageAxis], fraction: f64 }`
where `CoverageAxis { axis: str, domain: Interval | Enum[str],
method: corners | grid(k) | enumerated | analytic | monotone }`.
Discrete axes (line-ups, failure states, range selections) carry
`enumerated` with the point list; `fraction` is the conservative
scalar collapse kept for the margin note and old consumers.
`Prediction`, `Evidence`, and `SolverResponse` carry it;
`SCHEMA_VERSION` bumps once (F100: together with D96/D97).

**D96 (sec. 7 item 3; feldspar OPEN-2/11/12). One payload-ref channel,
feldspar's kind strings adopted verbatim.** `DischargeRequest` gains
`payloads: Mapping[str, PayloadRef]`, `PayloadRef { kind, digest,
origin }` -- hash-pinned refs only, never inline bytes. The kind
vocabulary is feldspar 09 sec. 4's list verbatim
(`geometry.parametric`, `geometry.realized`, `mesh`, `table`,
`spectrum`, `profile`, `mask`, `field`, `flownet`, `plan`) so
descriptor and signature stay the same strings on both sides.
`ModelSignature` gains required payload kinds (match = scalar ports
AND payload kinds). Digest->bytes resolution is an orchestrator-owned
content-addressed store handle passed to models at discharge; packs
never do their own IO discovery.

**D97 (sec. 7 item 4; feldspar OPEN-13, A-10, friction G2). Given
resolution is an orchestrator pass with a normative rule set; regime
tags get a real channel.** Lowering stays IO-free (AD-17), so the
step that turns obligation NAMES into `DischargeRequest` scalars is
Python-side, between obligation intake and model selection:
(a) property records evaluate over the environment box using the
record's declared per-axis monotonicity to pick the worst corner,
else the full-domain hull (conservative, honest);
(b) interface-envelope loads extract to named load components via the
WO-12 contract IR vocabulary;
(c) the port-name vocabulary is SHARED and single-homed: feldspar
06's strings (`mech.geom.<family>.<param>`, `mech.material.*`,
`mech.load.<case>`) are adopted as the registry, documented in the
pack contract, and packs reject unresolved names (their half,
already pinned);
(d) `DischargeRequest` gains `regimes: [str]`, asserted by LOWERING
from claim-kind construction and net discipline (`linear_elastic`,
`static`, `incompressible`, ...); a model whose validity domain
REQUIRES a tag the request does not carry is a non-match (honest
no_model), never an assumption. The v1 interim (tags guaranteed by
kind construction, feldspar 06) becomes the degenerate case of the
real channel.

**D98 (sec. 7 item 7; feldspar G23/G36, OPEN-14). Computed indexed
fields: one design for zone- and config-indexed.** New claim form:
`compute <name>: <quantity kind> over <index domain>` where the index
domain is a zone set (regolith/02 sec. 4) or a declared config
domain (`travel in [-80mm, 120mm]`). Lowering: one obligation whose
evidence carries a `field` payload (D96 channel) plus the produced
name entering the datum ledger (borrow-exempt, like events).
Consumption: sibling claims use ordinary projections -- `max(name)`,
`name at zone(tip)`, `slope(name, <axis>) in [...]` -- lowering to
derived quantities whose givens carry the producing obligation's
field payload ref (the promise-chain mechanism, dissipation
precedent). Until a discharging model exists, the honest interim
stays worst-point scalar reduction (feldspar OPEN-14's extremal
ports). Full shape in WO-33.

**D99 (sec. 7 item 8; feldspar G42; F102). Routed runs: one
extraction seam, cuprite gets `harness:` runs.** A `run` is a routed
path along realized structure with per-segment length, environment
class, and grouping. cuprite grows a `harness:` block (`run <name>:
from <endpoint> to <endpoint> along <waypoint refs>, bundle <group>`)
whose lowering extracts conductor lengths, bundle membership, and
connector environment as lowered givens -- hand-asserted lengths
become a lint once a run exists. Fluid `Pipe(from=...)` extraction
and wire-run extraction share ONE realized-geometry extraction module
(the seam lands with fluorite lowering, WO-32; WO-34 consumes it).
Path AUTO-ROUTING (choosing the waypoints) is planner/realizer
territory later; the language admits `route: free` so nothing is ever
hand-asserted in source, only lockfile-caused.

**D100 (COPEN-2). AD-23: one net core, per-discipline plugins.** The
net ledger machinery (terminal ledger, reference reachability,
imposer counting, per-subnet consistency) is generalized once in
`regolith-sem`, parameterized by a discipline: elec (single-driver,
voltage imposer, supply shorts) and fluid (pressure imposer, medium
consistency) are data/plugins over the same core. cuprite's existing
checks refit onto the core in the same change that adds fluid
discipline (WO-31); duplication of the ledger logic is the named
failure mode. Recorded as AD-23 in `00-architecture.md`.

**D101 (F100, F101; sequencing). The dispatch plan.** WO-30 (parser
promotions + typed emission, the WO-29 remainder) gates WO-28's
engine, WO-22/24 end-to-end halves -- it goes first and alone on
`regolith-lower`. WO-30 (pack contract v2 = D94+D95+D96+D97, one
schema bump) is the feldspar seam and touches none of WO-30's files
-- it may run concurrently. WO-31 (fluorite front end + AD-23 core),
then WO-32 (fluorite lowering + flownet payload + the extraction
seam; needs WO-30's channel). WO-33 (computed fields; needs WO-30)
and WO-35 (pin-mux assignment; Python realizer, independent) run in
the gaps; WO-34 (routed runs) last (needs WO-32's seam). WO-27
(feldspar conformance) unblocks progressively as WO-30 lands.
