## F109. What "the compiler wins" actually requires (gap audit)

The compiler-beats-assembly argument works because a compiler
exhaustively applies MECHANICAL transformations under an explicit
cost model and never forgets a constraint. The toolchain already has
the analogue for every decision it can reach: DFM eager resolution
(`free`), budget arithmetic, the cycle-30 optimizer (`in [lo,hi]`,
`by select`, traces, causes), margin-driven model selection, and the
no-unattributed-number drawing schema. The audit of where it CANNOT
yet reach, each with its recorded evidence:

1. **Geometry realization coverage.** The declarative-to-STEP chain
   is proven (coolant_gallery, D152) but narrow: WO-22's named
   residue (close-edge closure-solve in `regolith-ir::sketch`,
   sheet-gauge thickness source) still skips sheet_bracket, and no
   coverage LEDGER exists mapping which feature classes the
   interpreter honestly supports vs skips.
2. **Assemblies never realize.** Matings verify at L2/L3
   (contracts, ledgers, symmetry) but nothing solves mate
   constraints into placed transforms, emits an assembly-level
   realized artifact, or extracts assembly facts (mass/COM,
   interference) -- machinery is exactly this. A machine that
   cannot be REALIZED as an assembly cannot claim the bar.
3. **Structural demand extraction.** The five-design section-search
   flip stayed deferred (F108) on tributary-transfer load paths --
   feldspar WO-21's PARTIAL close-out names the same gap from the
   solver side. Cross-repo, now in scope by owner directive.
4. **The bar itself is not checkable.** "Could not have done
   better" must be a REPORT, not a slogan, or it will be claimed
   dishonestly.

## D170. The parity bar, made checkable (AD-33)

"A hand designer could not have done better" becomes a machine-
checkable PROVENANCE CLAIM, never an optimality claim (AD-30's
honesty non-goal stands: evidence never says "optimal"):

A design is **parity-grade** when the parity report proves, per
subject: (1) ZERO unattributed values -- every resolved number
carries a cause class (`optimize` / `dfm|drc` / `budget` /
`planner` / `derived` / asserted-with-source-position); (2) every
DECISION-shaped value (free variable, select, allocation) is either
engine-pinned with its trace or expert-asserted (ladder rung named);
(3) all demands discharged, or deviation/waive-recorded with basis
(`--release` semantics); (4) the counts render as a ledger in
`regolith ship --explain` (extending the WO-50 audit surface, one
report, no new mechanism). An "arbitrary number" -- a literal a
reviewer cannot distinguish from a guess -- is what hand design
tolerates and this bar forbids: literals are legal but COUNTED and
attributable to a source position, so the report makes guessing
visible. Machinery: WO-63.

## D181 (addendum, post WO-65 audit). Discrete catalog domains are `in registry(...)`; the forall-combo emission gap is a compiler bug-class fix

WO-65's mandatory per-member audit surfaced two real gaps and
correctly refused to paper over either (2026-07-10):

1. **The search domain was never declared.** `section: free` names
   no candidate family, and nothing principled can infer one
   (materials carry only numbers). RULING: the value-source grammar
   (regolith/03) generalizes its bounded-freedom form to DISCRETE
   RECORD DOMAINS: `in registry(<family-ref>)` -- "optimizer
   decides inside this catalog family" -- the exact `in [lo, hi]`
   semantics with a record family as the hard domain. `free` alone
   keeps its existing meaning (process-rule minimum; honest
   indeterminate where no rule resolves it) -- no reinterpretation.
   The five calcite corpus designs update their searchable members
   to `section: in registry(std.civil.<family>)`; `FrameMember`
   gains the lowered `section_domain` field (one serialized bump
   24->25, owned by WO-68 -- the D168 train rule).
2. **`forall combo in ...:` nested named claims never emit
   obligations** -- verified live (footbridge emits 4 obligations,
   zero `strength`). This is not new surface; it is an
   obligation-emission completeness BUG in `regolith-lower`'s sweep
   handling (INV-1's spirit: a claim in source that silently
   produces no obligation is the worst failure class this project
   has -- worse than a wrong verdict, because nothing ever reports
   it). Fix is Rust, WO-68; a regression test must assert every
   corpus `forall` claim form emits per-axis obligations.

WO-65 re-gates on WO-68 (its reopen criterion updated in place);
the WO-56 flagship acceptance chain is unchanged, one gate deeper.
Mantra check for (1): Unambiguous (a closed, hash-pinned catalog
family as domain), Intent-based (the author declares WHERE to
search, never WHAT to pick), User-friendly (one existing keyword
pair reused; no new block form).

**LANDED (WO-68, 2026-07-10).** Both fixes shipped in the same
SCHEMA_VERSION 24->25 bump (D168 train rule). Root cause of (2), found
live: `forall <var> in <domain>:` as a BLOCK claim (header line, no
inline predicate, a nested indented body of named claims) was never a
recognized shape in ANY statement context outside a rule pack -- the
parser's generic dispatch saw a bare `Ident` follower (not `Colon`),
classified it `StmtShape::Opaque`, and `parse_opaque_stmt` swallows an
opaque statement's ENTIRE nested indented body into the same
`OpaqueIsland` (structure-at-the-boundary, AD-3) -- so every named
claim inside the block was invisible past the parser, not just to
`regolith-lower`. Fix: a new CST node, `SyntaxKind::ForallSweepClaim`
(`crates/regolith-syntax`), recognized in `StmtCtx::Generic` (the
context every `require`-group body already parses under) whenever a
header line ending in `:` (checked BEFORE any token is consumed, so
independent of how loosely the D105a domain grammar's tail sweeps)
follows `forall <ident>` -- it opens a real nested statement block via
the same `enter_body_block` machinery a `Field`'s own nested body
already uses, so nested claims land as ordinary structured `Field`
nodes. `RequireClaim`/`RequireDecl` gained `all_claims()` (direct
claims interleaved with each sweep's claims, sweep-tagged), and every
require-group lowering path in `regolith-lower::claims` (decl claims,
calcite frame claims, fluid claims, top-level cost claims) now uses it,
attaching the block's `SweepDomain` to the emitted obligation (INV-1).
Proved red-when-reverted once (the exact live repro: footbridge drops
from 5 back to 4 obligations, `strength` missing) and left as two
permanent regression tests in `regolith-lower::claims::tests`
(`forall_sweep_block_nested_named_claim_emits_an_obligation`,
`forall_sweep_block_over_calcite_frame_claims_flips_the_live_repro`).
Corpus-wide: EVERY track uses this block form (not calcite-only --
cuprite `spec:`/`require:` sweeps, fluorite `states:`-indexed
sweeps), so the fix's obligation-count deltas land wherever a nested
NAMED claim existed: the five calcite designs (footbridge/bus_shelter/
pole_barn +1 `strength` each; small_office +1; retaining_wall
untouched, no `forall combo` in its require group) plus
`fluorite_dual_brake_circuit` (+1, its `margin:` claim under `forall
circuit in {front_circuit, rear_circuit}:`). Every other corpus golden
churned digests only (the SCHEMA_VERSION bump propagates through
content addressing everywhere, AD-18) with unchanged `obligation_count`
-- verified file by file, not assumed.

`in registry(<family-ref>)` (finding 1) needed NO grammar/parser
change at all: it already parses through the existing generic `in
<expr>` value-source arm plus the ordinary `CallExpr` expression
grammar (verified with a zero-diagnostic parse of `section: in
registry(std.civil.hss_square)` before writing any code). The only
real work was the schema (`FrameMember.section_domain: Option<String>`)
and lowering (`frame_lower::section_domain_ref`, leaving `section` at
its `free` placeholder since the domain is declared, not resolved).
The five ratified designs' searchable members now declare families
matching their FIXED siblings' material class (steel `astm_a992`
members -> `std.civil.w_shape`: footbridge G1/G2, bus_shelter G1,
small_office G2_AB/GR_AB; timber `spf_no1` member -> `std.civil.
timber_sawn`: pole_barn T1, the design's only catalog family for a
timber roof truss). `frame_resolve.resolve_member` now defers a
declared-domain free section as `frame_section_domain_unsearched`
(naming the family), distinct from the pre-existing `frame_
section_free` for a section with no declared domain -- WO-65's
concrete reopen target. No new diagnostic code exists yet for a
malformed `in registry(...)` (empty ref; non-`registry` callee); both
degrade to `section_domain: None` honestly (never a guess, never a
panic) -- recorded as a named gap (unit tests in `frame_lower.rs`
cover both shapes), not a silently invented one, since inventing an
E0xxx code was outside this WO's named deliverables.

## D182 (renumbered +1 at integration; WO-66 follow-up dispatch addendum). std.fluid polymer-melt medium record shape ratified (WO-64 wall W1)

WO-64 phase A wall W1 (its own dispatch ledger, `WO-64`'s walls
list) asked for a `std.fluid` filament-melt property family (PLA/
PETG classes minimum) so a future flownet-solve pass has a real
medium to bind the hotend melt path to. The record shape, ratified
before generation per the sourcing law (32-stdlib-depth.md sec. 1
rule 5):

A new `[[polymer_melt]]` array-of-tables in
`stdlib/std.fluid/records/polymer_melt.toml` (loaded generically by
the SAME `load_toml_records` seam every other stdlib kind uses --
no schema/Rust change, AD-34 content-only). Fields:

- `polymer_class` -- "PLA" / "PETG" (class label, not a vendor SKU).
- `rho_solid_kg_m3` -- room-temperature solid density from a
  manufacturer TDS (ASTM D1505 in both records landed here).
- `melt_temp_min_c` / `melt_temp_max_c` -- the TDS's own melting
  range (ASTM D3418), or a single DSC endotherm reading repeated in
  both fields for an amorphous polymer (PETG) where the source
  states only one number.
- `process_temp_min_c` / `process_temp_max_c` -- the TDS's stated
  processing/extrusion or print-temperature window (a proxy for the
  hotend's own operating band, cited separately from the melting
  range since the two are not the same figure).
- `rho_melt` -- an array of `{t_k, rho_kg_m3}` POINTS ONLY, never a
  fitted curve this package invents. Where a cited source states its
  OWN closed-form relation (e.g. a linear thermal-expansion formula
  from a published patent document), points MAY be evaluated
  directly from that stated relation -- this is using the source's
  own model, not fitting a new one -- and the evidence clause must
  say so explicitly, naming the formula and its cited constants.
  Absent any such source, this array is OMITTED with a note.
- `mu` -- an array of `{t_k, mu_pa_s, shear_rate_1_s, note}` POINTS
  (never a fitted curve), each point citing its own source
  independently; a polymer melt is non-Newtonian, so a point without
  a stated shear rate is a zero-shear/Newtonian-plateau figure and
  MUST say so in its `note`. Absent any independently verified
  point, this array is OMITTED with a note (sourcing-law rule 1).
- one `evidence` table per record (method/trust_tier/reference),
  matching the `std.fluid` `components.toml` precedent (D166): the
  `reference` string itemizes which field came from which document,
  edition/revision, and fetch context, rather than one bare citation
  standing in for every field.

This mirrors `media.toml`'s existing `[[medium]]` shape (T_K/P_Pa/
rho/cp/mu single-state-point fields) widened with named point ARRAYS
instead of one T for `rho`/`mu`, since a melt's whole reason for a
new family is that its properties are NOT state-point-invariant
across the hotend's operating range the way water/air/N2 are treated
