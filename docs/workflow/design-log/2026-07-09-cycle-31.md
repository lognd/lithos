## F109. What "the compiler wins" actually requires (gap audit)

The compiler-beats-assembly argument works because a compiler
exhaustively applies MECHANICAL transformations under an explicit
cost model and never forgets a constraint. The toolchain already has
the analogue for every decision it can reach: DFM eager resolution
(`free`), budget arithmetic, the cycle-30 optimizer (`in [lo,hi]`,
`by select`, traces, causes), margin-driven model selection, and the
no-unattributed-number drawing schema. The audit of where it CANNOT
yet reach, each with its recorded evidence:

1. **Geometry realization coverage.** The declarative-to-STEP chain
   is proven (coolant_gallery, D152) but narrow: WO-22's named
   residue (close-edge closure-solve in `regolith-ir::sketch`,
   sheet-gauge thickness source) still skips sheet_bracket, and no
   coverage LEDGER exists mapping which feature classes the
   interpreter honestly supports vs skips.
2. **Assemblies never realize.** Matings verify at L2/L3
   (contracts, ledgers, symmetry) but nothing solves mate
   constraints into placed transforms, emits an assembly-level
   realized artifact, or extracts assembly facts (mass/COM,
   interference) -- machinery is exactly this. A machine that
   cannot be REALIZED as an assembly cannot claim the bar.
3. **Structural demand extraction.** The five-design section-search
   flip stayed deferred (F108) on tributary-transfer load paths --
   feldspar WO-21's PARTIAL close-out names the same gap from the
   solver side. Cross-repo, now in scope by owner directive.
4. **The bar itself is not checkable.** "Could not have done
   better" must be a REPORT, not a slogan, or it will be claimed
   dishonestly.

## D169. The authoring doctrine is the product thesis (restated, not new)

Agents (or humans) AUTHOR declarative sources; the toolchain lowers,
resolves, optimizes, verifies, and ships; the expert ladder
(regolith/12) plus manual lowering/extern linkage (regolith/08
sec. 4) are the nitty-gritty escape AT EVERY LEVEL -- the default
path needs no intervention, and no path forbids it. Nothing new
lands with this entry; it is recorded because it is now the stated
product goal, and because it resolves the sovereignty question for
agent-driven design: the AGENT is the author, so AD-28/AD-30
sovereignty rules (humans declare candidate sets; advice never
auto-substitutes) apply to the authoring agent unchanged.

## D170. The parity bar, made checkable (AD-33)

"A hand designer could not have done better" becomes a machine-
checkable PROVENANCE CLAIM, never an optimality claim (AD-30's
honesty non-goal stands: evidence never says "optimal"):

A design is **parity-grade** when the parity report proves, per
subject: (1) ZERO unattributed values -- every resolved number
carries a cause class (`optimize` / `dfm|drc` / `budget` /
`planner` / `derived` / asserted-with-source-position); (2) every
DECISION-shaped value (free variable, select, allocation) is either
engine-pinned with its trace or expert-asserted (ladder rung named);
(3) all demands discharged, or deviation/waive-recorded with basis
(`--release` semantics); (4) the counts render as a ledger in
`regolith ship --explain` (extending the WO-50 audit surface, one
report, no new mechanism). An "arbitrary number" -- a literal a
reviewer cannot distinguish from a guess -- is what hand design
tolerates and this bar forbids: literals are legal but COUNTED and
attributable to a source position, so the report makes guessing
visible. Machinery: WO-63.

## D171. Geometry lowering deepens: coverage ledger + assembly realization (AD-32)

Charter `30-geometry-lowering.md`; machinery WO-62. Load-bearing:

- **Close the WO-22 residue**: the close-edge closure-solve
  (sketch profiles with free lengths solve to closed loops or emit
  a constructive diagnostic) and the sheet-gauge thickness source
  (`process=laser_cut(sheet=...)` -> blank thickness);
  sheet_bracket realizes to STEP.
- **A feature-coverage LEDGER, honest by construction**: the
  realizer publishes (as data, drift-checked against the
  interpreter) which FeatureProgram ops it supports; an op outside
  the ledger is a named skip diagnostic, never a silent partial
  solid. The corpus + flagship demand list drives coverage growth;
  the ledger is the record of what "declarative mech" can claim.
- **Assemblies realize (the new capability)**: mate constraints
  (align/coincident/distance/angle, the existing mating vocabulary)
  solve to placed part transforms at L4 -- a deterministic
  sequential constraint solve over the mating graph (loop closure
  residuals become diagnostics, not silent least-squares fudges);
  the result is a `RealizedAssembly` IR (parts by RealizedGeometry
  digest + transforms + solved DOF states) with STEP assembly
  export, extracted assembly mass/COM, and pairwise interference
  facts feeding realized-fact rules (an interference is an ordinary
  release-gated diagnostic). One more first-class L4 IR by AD-25's
  own growth rule.
- Kinematic MOTION (joint envelopes, sweep volumes) is deliberately
  staged BEHIND the flagship's evidence: reopen listed in the
  charter, not built speculatively.

## D172. The flagship program (charter 31-flagships.md)

(`examples/flagships/printer_k1/`) -- chosen because it exercises

- **Phase A (now, contract-first)**: full architecture at L0->L2 --

  machine scale).

Planes and motherboards are flagship-2/3, GATED on flagship-1's
close-out findings (each will be chartered from evidence, not
ambition). Flagships are corpus-enrolled: their diagnostics stay at
zero and their goldens pin.

## D176 (WO-66 dispatch addendum). tools/stdlib generation framework ratified

WO-66 execution. `tools/stdlib/` is the ONE home for D174's
generation machinery: `render.py` (shared deterministic TOML
rendering -- sorted keys, fixed float repr, no per-script
reimplementation), one `gen_<family>.py` module per generated family,
`generate_all.py` as the single `make stdlib-gen` entry point, and
`tools/stdlib/data/*.toml` as the committed input tables (citation
headers in-file, per the sourcing law). No live network fetcher is
introduced this dispatch (see `tools/stdlib/SOURCES.md`'s own note:
the standards bodies this WO's families cite do not publish open,
scrapeable tables under a permissive TOS this dispatch could verify;
generation therefore runs over committed, manually-transcribed input
tables, which stays inside the sourcing law's "official/open sources"
and "unverifiable fields omitted" rules -- it does not invent a
fetcher that would violate rule 2). Idempotence + committed-output
drift are enforced by `tests/tools/test_stdlib_gen_drift.py`, reusing
the SAME `generate_all()` collection `make stdlib-gen` calls (one
generation surface, not a second copy) -- this satisfies the
acceptance criterion's drift-check requirement without a second
Makefile target (this WO's surface allowance is ONE additive target).

## D177 (WO-66 dispatch addendum). std.fasteners record shape ratified

New package `std.fasteners`, `components` kind. One record per
(standard, nominal diameter, length, property class) for screws/
bolts (GENERATED, `tools/stdlib/gen_fasteners.py`) and per (standard,
nominal diameter[, bolt property class]) for nuts/washers. Thread
tolerancing/engagement MATH stays a `std.models`/feldspar concern per
charter `32-stdlib-depth.md` sec. 2 ("Thread math stays a MODEL");
these records carry dimensions + rated mechanical properties only,
never a computed thread-engagement formula. Fields the cited standard
does not state for a given row (the full per-diameter min/max length
range; the ISO 898-2 nut proof-load table) are represented instead by
an explicit, honestly-noted approximation (a common-stock length
subset) or omitted outright -- never silently presented as the real
range.

## D179 (WO-66 dispatch addendum). std.bearings/std.motion/std.machines/std.tooling package shapes ratified

Four new vendor-neutral `components` packages, following the
`ti.logic`/`microchip.cpld`/`st.mcu` precedent (WO-60 close-out: raw
component/class dimension records belong in a `components`-kind
package, not force-fit into a pattern-pack `block`/`impl` shape) but
WITHOUT a vendor name (regolith/11 sec. 2's `std.` prefix is
appropriate here: these are class-level engineering-commons records
-- bearing boundary dimensions, motion form factors, machine
envelope/kinematics classes, tool geometry classes -- not one named
vendor's SKU list):

- `std.bearings`: `deep_groove_ball` (60xx/62xx/608 boundary dims per
  ISO 15, load ratings per generic manufacturer general-catalog
  figures) + `linear_bushing` (LM_UU class, dims only -- load ratings
  omitted, no governing standard fixes them and none were
  independently verified).
- `std.motion`: `stepper` (NEMA17/23 rated-point classes),
  `leadscrew` (Tr8 class per DIN 103), `belt_pulley` (GT2 class),
  `linear_rail` (MGN12/15 class, dims only -- load ratings omitted).
- `std.machines` (feeds WO-67/AD-35): `mill_3axis`, `printer_fdm`,
  `laser_co2` -- CLASS records (representative envelope/kinematics/
  spindle-or-nozzle for a common machine category across many vendor
  catalogs), each evidence clause says so explicitly so a reader
  never mistakes a class record for a specific product spec.
- `std.tooling` (feeds WO-67/AD-35): `end_mill` (solid-carbide
  diameter x flute x stickout classes) + `drill` (ISO 235/DIN 338
  jobber-length series) -- same class-record discipline.

All four: `tier=community` (D58), no signed evidence, no
schema/Rust surface touched.
