# Design cycle 1 -- 2026-07-03

Method: read the full tree cold, write/repair examples in target syntax,
treat every awkwardness or internal contradiction as a design bug, decide
or escalate each one. Findings are numbered F-nn; decisions D-nn (the
D-numbers match `TODO.md`). Status tags follow the spec legend.

## A. Vocabulary bugs (violations of "one word, one idea")

**F1 / D1. `process` keyword collision.** [FIXED]
`process` is [S] substrate vocabulary (manufacturing process module,
`process=` stage arg) AND the elec behavioral construct
(`process on <event>:`). Both occur in one language: a board writes
`stage bare: process=pcb_fab(...)` while its blocks write
`process on clk.rise:`. Flagrant violation. Resolution: the behavioral
construct is now spelled `on <event>:` (body of statements with
non-blocking `<=`). The `on` overload is registered: `on` = the context
guarding a body -- a region (scope guards, feature placement) or a
triggering event (discrete behavior). Alternatives considered:
`always` (Verilog baggage, not a human word), `react on` (longer, no
added meaning), `at` (already used for spatial datums in boundary loads).

**F2 / D2. Budget `pins:` collides with electrical pins.** [FIXED]
`pins:` inside `budget` is fatal vocabulary in a language whose domain
uses "pin" for package terminals (pin-mux, pinout, pin assignment).
Also the old form `pins: x: locked(v)` double-marked the same idea.
Resolution: budgets use `locked:` entries (`locked: decouple.value:
100nF`); `locked()` as a wrapper is retired; the substrate "pin family"
is renamed the **lock family** (`locked:`, `use`, `sequence:`,
`merge()`). Prose may still say "pinned decision" outside elec docs, but
elec docs prefer "locked".

**F3 / D3. `free` double meaning in `dof:`.** [FIXED]
`free` the value source (process-rule minimum decides) vs `dof:
free=[...]` (kept freedoms). Resolution: `dof: removed=[...] kept=[...]`.
OPEN-3's constraint "the syntax must not need to change" is exactly why
this is fixed now, pre-implementation.

**F4 / D5+D6. Undefined spellings: two-sided comparator literal; range
vs interval.** [FIXED]
- `within [lo, hi]` is the two-sided comparator literal (a demanded
  window), distinct from `[lo, hi]` the interval value (a scatter/range
  I assert). `within` is a registered overload: containment, in all
  three positions (`within(a,b,tol)`, `within d after e`,
  `within [lo, hi]`).
- `[a, b]` (comma) is a closed interval over a continuous quantity or a
  closed discrete domain (`in [2, 6]` monomorphizes 2,3,4,5,6).
  `[i .. j]` (dots) is a half-open positional range: bus bits
  (`sel[0 .. 3]` = bits 0,1,2), memory addresses (`flash[0 .. 32kB]`).
  Substrate 07's `sweep: n in [2..6]` was wrong on this rule; fixed to
  `[2, 6]`.

**F5 / D22. Bus bit addressing vs the no-positional-index rule.** [FIXED]
`sel[0 .. 3]` looks like the banned positional indexing. It is not:
a bus bit's position is *semantic content* (binary weight, protocol
lane), not creation-order accident -- addressing by it is semantic
addressing, the same reason `flatten(...)` may say "layout order".
Blessed for buses/width'd ports only; slicing splits the orbit like any
special-role assignment. A width'd port's per-bit set is `.bits`
(the mux example's ad-hoc `.lanes` is retired).

**F6 / D21. Block-impl naming had a second mechanism.** [FIXED]
`impl Mux by composing DecoderTree:` invented postfix naming. The
language already has `as` for naming (`impl X for self as front_seat`).
Now: `impl Mux by composing as DecoderTree:`.

**F7 / D12. `import` was never in any vocabulary table.** [FIXED]
Documented: `import <package> (<names>)` at file top. Registered
overload with the import stage (`stage src: import(path)`): both mean
"bring foreign content into this compilation" -- source-level vs
data-level. Example headers unified to package paths (`std.mech.cnc`,
not `processes.cnc`).

**F8 / D13. The elec demands block had no spelling.** [FIXED]
It is `demands:` (the substrate's generic name). Mech keeps its binding
`tolerances:`; elec has no reason to rename.

**F9 / D18. Cross-track homonyms (`setup`/`hold`).** [POLICY ADDED]
Mech keywords `setup`/`hold:` are immovable machining jargon; in elec,
setup/hold are timing terms of art (but only appear as field names in
`timing:` promises, never as keywords). New vocabulary principle:
keywords are per-track; [S] words are global; a track must not adopt a
keyword that is domain jargon *meaning something else in that track*.
`setup`/`hold` pass (each is jargon only in its own track).

**F10 / D2-adjacent.** `pins:`-style redundancy audit found one more:
`preload_at:` appeared once in the Compression mating sketch (mech 03)
-- an undeclared keyword. It is `preload:` with a config-domain
qualifier; the doc example was wrong (see F15).

## B. Doc-internal contradictions (spotless-language pass)

**F11.** `mating Compression(seat_a, seat_b)` used positional sides,
violating FIX-2 ("named sides, always"). Fixed to `between:` form.

**F12.** TorchIgniter assembly: `pivot: Revolute(body.boss, nozzle.hub)`
positional AND references `nozzle`, which is not in `parts:`;
`seal:` had a `lubrication:` state but no mating. All fixed.

**F13 / D19.** `size(text + ro) <= partitions.app` compared a size to a
region. Coercion rejected (defaults test: too magical); explicit
`partitions.app.size`.

**F14 / D9.** Fatigue claim spelling harmonized: bare `life(...)` ->
`mech.life(...)`; claim forms are namespaced like every other quantity
claim. OPEN-4's "design claim syntax now" is hereby done:
`mech.life(<subject>, under=<spectrum>) >= <life>, scatter_factor=k`
and `mech.damage(miner, under=...) <= d`.

**F15.** thermostat example + intent-layer doc both wrote
`actuate: switch(load: 24V_ac, 1A, isolated)` -- `switch` is not a verb
in the table; the line parses as intent named `actuate` with unknown
verb. Fixed to `switch_heat: actuate(ac_load(24V, 1A), isolated)`.

**F16 / D15.** pillow_block used `ExtrudeProfile` inside a
`cnc_mill` stage -- additive CSG vocabulary in a subtractive process,
violating "constructs must come from the stage's process vocabulary".
The milling vocabulary word for "remove everything outside a profile"
is `Contour`. The same bug class was then found in mech/02's own lead
example (`Revolve` inside a `cnc_lathe` stage; now `Turn`/`BoreTurn`).
Rule of thumb (sharpened in cycle 2): process features are named in
the trade's vocabulary -- form nouns and trade verbs (Contour, Pocket,
Boss, Turn, Bore, Blank, Bend). CSG construction verbs (Extrude,
Revolve, Loft) are wrong everywhere in machining stages; "milling a
boss" is trade speech and fine. The example also had `body` and `bore` as siblings in one
scope with `bore` depending on `body`'s placement, laundered through a
profile-value datum (F17), and bound `bore.floor` on a *through* bore
(no floor exists). All rewritten.

**F17 / D14. Profile export anchoring.** [SETTLED]
A `profile` is a placeless value; its exported datums acquire 3D
existence only through an instantiating feature, and are addressed
feature-first (`body.mid_plane`, never `BodyOutline.mid_plane` after
placement). Referencing the profile's export where a placed datum is
needed is a compile error naming the instantiating feature(s). This
also closes the snapshot-rule loophole in F16.

## C. Design gaps the examples forced open

**F18 / D8. Multi-piece parts (weldments).** [LEANING, spec'd]
A weldment is one part (one BOM line; machined as a unit post-weld)
built from several pieces with their own stock and material. Neither
`part` (single material, single stock) nor `assembly` (no stages) could
express it. Resolution shape: a part may declare `pieces:` (named
stocks, part refs, or imports, each with its own material); a joining
stage (`process=mig_weld(...)`, `joins=[...]`) declares piece placement
with the *same `align:` vocabulary as matings* ("a weldment is an
assembly frozen into a part") and weld features create joint entities.
Weld feature/claim vocabulary is new OPEN-13.

**F19 / D4. Claims over derived structure at intent altitude.**
[SETTLED in shape]
The thermostat's `droop` claim named `vdd_rf` and `radio.tx_burst` --
interior entities that do not exist in source. Rule: interior structure
realized *for* an intent is addressable through the intent's namespace
(`report.supply`, `report.tx_burst`, `sense_temp.chain.out`). These
handles resolve at allocation; an unresolvable handle is a constructive
error. This is the flow-realization lockfile philosophy applied to
naming.

**F20 / D7. Two parameter mechanisms, no stated rule.** [SETTLED]
`<Params>` (angle brackets) are caller-chosen: part of contract
identity, monomorphized when discrete. `params:` are impl-chosen:
internal design variables, existentially quantified in `spec:`
(any impl picks values; resolution is lockfile-caused). NtcFrontend's
`r_fixed` is the canonical example.

**F21 / D10. Config-variable exposers.** [SETTLED, generalizes FIX-7]
Config variables are namespaced by the construct that *exposes* them:
a connection (`pivot.theta`), a block instance (`pm.mode`), or the
system itself (`exposing op: {sleep, run, fault}`). EOPEN-8 stays open
only for whether a statechart is ever needed; the motor-drive example
worked on config variables alone.

**F22. `impl by circuit` body shape.** [LEANING, spec'd]
Never defined. Now: circuit bodies instantiate *component classes*
(`Inductor(l=in [...], i_sat >= 3A)`) in construction scopes and join
terminals in a `nets:` block (`swn: (u1.sw, l1.a)`). Component classes
resolve to catalog parts at L4 (E-series snapping). The driver ledger
for analog/power nets needs its own discipline (KCL-relational, not
single-driver) -- new EOPEN-16.

**F23. Supply ports need direction.** [FIXED]
`supply(v, i)` could not express a converter's output rail. Port kind
now `supply(in|out, v, i)`; `reference` unchanged.

**F24 / D11. Global optimization objectives.** [new SOPEN-4]
`minimize(count_distinct)` (BOM consolidation) attached a *global*
objective to a *local* variable -- architecturally wrong (optimization
direction is per-variable and secondary). Removed from the example.
The real need (minimize distinct part values across a design) is an
allocation-policy concern; SOPEN-4 opened in substrate 03.

**F25. Intent -> workload seam.** [new EOPEN-15]
The flight controller declared workloads out of thin air; the docs say
peripheral demands "arrive from the system's flows", but no construct
ties a `compute` intent to a `computer` workload. Sketch adopted in the
motor-drive example: `foc: loop(...) realizes decide`. Needs a full
design pass (rates/latencies must reconcile between the intent's
demands and the workload's declaration).

**F26 / D20. Zones and image partitions are one substrate concept.**
Named partition of an owned extent, with the `remainder` rule and
boundary datums. Two domain spellings (`zones over` / `partitions:`) --
legitimate per the "different vocabularies, same machinery" charter,
but the binding table now has the row so nobody reinvents it.

**F27 / D17. Boundary loads carry `at=<datum>`** (application point);
used by pillow_block, never documented. Now in the mech vocabulary.

**F28. OPEN-1 variants: shape drafted.** [LEANING, spec'd]
`variant <name>: {a, b}` declares an externally-chosen discrete config
axis (chosen by the orderer, not the optimizer -- so *all* variants
must verify; contrast `in {..}` which the optimizer resolves).
Construction may branch with `when <variant> = <value>` statement
guards; instantiation sites choose (`clip: CableClip(hand=left)`);
lockfile carries per-variant sections. Cross-variant evidence sharing
stays open under OPEN-1.

**F29. Pins legitimately live in registry records.** The rule "pins
never appear in source" means *design* source; `component` records are
exactly where package pin maps and alternate-function tables belong
(EOPEN-12). Clarified in elec 04; schema exercised in
`examples/registry/stm32g0.loom`.

**F30 / SOPEN-2 pressure test.** The sensor-pod example (mech enclosure
+ elec board) worked the substrate 10 sec. 3 sketch: the interface lives
in one home language; foreign-domain roles bind through quantity-core
vocabulary; the thermal handoff rides the existing `effects:` routing.
Two things the example could not express cleanly (left open under
SOPEN-2): the cross-language artifact reference form
(`artifact("imu_board.loom", ImuBoard)` used as a placeholder), and
which build graph owns the joint obligations.

## C2. Findings surfaced while writing the corpus

**F31. Impl-supplied promise values = inline refinement.** [SETTLED]
The IMU board needed to state its own dissipation/mass, tighter than
the PodMount interface's ceilings. No new mechanism: an impl may carry
a `promises:` block that *narrows* the interface's promises --
`refines` semantics spelled inline. Consumers binding that impl see the
refined values; consumers of the bare interface see the base. Widening
is rejected (same one-way rule as `refines`).

**F32. Where do a standalone part's loads come from?** [SETTLED]
Parts have no `boundary:` (that stays assembly-only). A part claim may
reference `boundary.` names, which resolve against *each enclosing
assembly* (the part re-verifies per context); a part designed with no
assembly routes loads through interface envelopes
(`under=interface_envelope(<Interface>)`) whose literal promise values
are the asserted truth. The sheet-bracket example is the canonical
form.

**F33. Assembly-level datums.** Assemblies may declare `datum`,
including captures from measured import geometry by query
(pillow_block's `plate_ref`). Vocabulary row extended from "in part"
to "in part / assembly".

**F34. Count syntax `n x thing`.** Used by boundary supplies
(`battery(2 x AA_alkaline)`), peripheral demand vectors
(`pwm x 4`), and component `functions:`. Documented as substrate-shared
quantity syntax (a count is a discrete quantity; `x` is its
constructor), not ad-hoc notation.
