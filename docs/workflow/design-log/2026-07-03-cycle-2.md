## A. Decisions

**D23. Cross-language artifact reference = ordinary `import`.**
[SETTLED; was SOPEN-2 part 1]
The cycle-1 placeholder `artifact("path", Name)` is retired. A
`.mill`/`.loom` path in an `import` statement imports that file's
top-level declarations; the extension selects the front-end; everything
contract-level (interfaces, impls, promises, matings) is substrate IR,
so it composes across languages with no bridge syntax.
`import "servo_drive.loom" (ServoDrivePcb)` and
`import std.mech.cnc (...)` are one mechanism.
- *Unambiguous:* one reference form; a second mechanism would be two
  spellings of "bring declarations into scope".
- *Intent-based:* the referencing design names the artifact, not the
  toolchain plumbing.
- *User-friendly:* nothing new to learn; the L1 error surface (unknown
  name, wrong kind) is the existing one.

**D24. Joint obligations belong to the declaring system.**
[SETTLED in shape; was SOPEN-2 part 2]
The artifact that declares a connection owns the obligations that
connection generates (this was already implicitly true within one
language). Cross-domain: a mech-home assembly that connects an elec
board owns the joint thermal/mass obligations; each imported artifact
keeps its own artifact-level obligations in its own build graph;
evidence is content-addressed, so shared subjects never re-verify.
- *Unambiguous:* single ownership, applied to obligations.
- *User-friendly:* incremental verification falls out -- editing the
  board re-runs board obligations and exactly the joint ones whose
  subject changed.

**D25. EOPEN-15 resolved: intent -> workload reconciliation.**
Three rules, all L2 arithmetic:
1. `realizes <intent>[, <intent>...]` on a workload claims it serves
   those compute intents. A workload may realize several intents;
   every compute intent must be realized by **exactly one** workload
   across the system's computers -- a ledger, like flows (Unambiguous).
2. The workload's declaration must *imply* each realized intent's
   demands: rate >= intent rate, latency <= intent latency, state >=
   intent state. Failure is an E04xx naming both sides (Intent-based:
   demand flows down; the workload cannot weaken it).
3. An unrealized compute intent is completed by allocation: a
   **derived workload** with the intent's demands copied verbatim
   (conservative), lockfile row `cause: derived(intent <name>)`.
   Passes the defaults test: conservative (verbatim demands can only
   over-ask), local, lockfile-materialized.

**D26. EOPEN-16 resolved (v1): the analog net discipline.**
Four static checks at L3 on `nets:` bodies (cheap tier first --
User-friendly), replacing the digital single-driver ledger which does
not fit relational (KCL) nets:
1. **Terminal ledger.** Every terminal of every instantiated component
   appears in exactly one net; intentionally unconnected terminals are
   `discard`ed (`discard u1.dnc` -- the existing word; registered
   overload: explicit sink for an intentionally unconsumed
   output/terminal).
2. **Reference reachability.** Every net must be conductively
   reachable to a `reference` port (catches isolated islands).
3. **At most one voltage-imposing terminal** (`supply(out)`) per net;
   paralleling supplies requires a declared arbitration
   (`arbitrate ... parallel(<share discipline>)`).
4. **Supply-short detection:** two distinct `supply(out)` nets joined
   is an error (special case of 3, reported in supply vocabulary).
Full KCL/DAE well-formedness stays with the behavioral semantics
decision (EOPEN-7).

**D27. Panelization is a planner concern, not a language artifact.**
Production multiplication -- PCB panels, sheet-metal nesting, casting
trees, 3D-print build plates -- is manufacturing tooling: the designer
declares the *product* artifact; the process pack + planner decide the
multiplication; the panel/nest/tree is **plan evidence**
(content-addressed, lockfile-caused), serialized by backends like any
plan. (Intent-based: paneling is mechanism, not product truth; the
defaults test holds: conservative, local, lockfile-visible.)
Consequently the multi-piece `pieces:` row binds in elec to
**module-on-carrier / hybrid assemblies** (castellated module soldered
to a carrier = pieces joined into one product artifact), NOT to
panelization.

**D28. OPEN-13 first pass: weld vocabulary.**
[SETTLED in shape; models remain open]
- Taxonomy: `FilletWeld`, `GrooveWeld(prep=...)`, `SpotWeld`,
  `PlugWeld`, declared per weld process module like any feature class.
- Each weld feature creates a **`weld` joint entity** with measures:
  kind, leg/throat_eff, effective length, process record ref --
  queryable (`welded.welds.all`, `.where(kind=fillet)`).
- Distortion enters the entity DB as position scatter on the joined
  pieces, from the weld process capability table (it then flows into
  stackups with zero new machinery).
- Claim form `mech.weld_stress(w, under=...)`; the cheap tier is the
  line-weld (treating-welds-as-lines) method, expensive tier
  shell/solid FEA. Harness models remain the open half of OPEN-13.

**D29. Boundary subsumption.** [SETTLED in shape; new, surfaced by the
servo example]
An imported artifact arrives verified under its *own* declared
boundary (its system's `boundary:`). The enclosing system's context
must be **contained** in that assumption for the evidence to transfer:
for every boundary quantity both declare (ambient, supply window), the
enclosing interval must be a subset of the imported one, else an L2
error naming both declarations. This is promises-not-actuals applied
to environments, and it is what makes cross-context reuse of evidence
sound (Unambiguous: no silent revalidation gap; User-friendly: caught
at contract tier, constructively).

## B. Findings

**F35. Retire the ad-hoc `@ <value>` claim condition.** The doc
example `leak(seal) < 1e-3 scc/s @ 25bar` used an undocumented
condition spelling. Corner discipline already owns this: the claim is
evaluated at its worst corner of `boundary.chamber_pressure`
([0, 25bar] -> 25bar), which is the model's job, not the author's.
The `@` form is deleted; `@` remains only in `@hint`.

**F36. Mixed-domain vendor records.** The servo motor
(`vendor(maxon_dcx22)`) carries mech interfaces (flange, shaft) AND
elec interfaces (winding ports, thermal promise) in one `parts`-kind
registry record. No new mechanism: contracts are substrate IR, so a
record's interface bundle may span quantity namespaces; each consumer
binds the roles its own language understands (this is substrate/10
sec. 3's rule exercised in the purchased direction).

**F37. New [EOPEN-17]: host binding for synthesized blocks.** The
frame grabber needed "PixelPipe (RTL) is hosted by u_fpga". Which
programmable part hosts which `by spec` impl is an allocation decision
(lockfile `cause: planner(host pix -> u_fpga)`), `use`-pinnable.
Multi-FPGA floorplanning and partial reconfiguration are explicitly
out of v1 scope.

**F38. EOPEN-7 observations (leaning strengthened, NOT settled).**
The FPGA example needed: synchronous-reactive semantics *within* one
clock domain (all `on clk.rise:` bodies see coherent state); ALL
cross-domain interaction through declared crossing matings
(`async_fifo`); continuous/discrete contact only at declared converter
ports. It did NOT exercise continuous-discrete feedback (a switching
regulator's comparator loop would); EOPEN-7 stays open until an
analog-feedback example is written, but the three properties above are
now recorded as constraints any chosen semantics must satisfy.

**F39. Cross-track matings in one system.** The servo assembly
instantiates `ConnectorMate` (elec pack) next to `FlangeMount` (mech
pack). Legal by construction: matings are substrate contracts; ledgers
are pluggable domain packs keyed by the mating's declared kind, so the
driver/domain ledgers run over the elec connections of a mech-home
assembly and the DOF ledger over its mech ones. Stated explicitly in
substrate/10 now.

**F40. Two loads in one `programmed` stage** (firmware + bitstream)
work under existing rules: independent construction statements in one
scope, each hash-pinned. No new vocabulary.

**F41. The rules caught their own author.** The torch igniter's first
draft contained three bugs the language's own checks would flag:
(1) the body's flange face bound at `turned` while `milled` cut
counterbores through it -- the SEAM-1 E0302, fix is the one-token
requalification to `milled.envelope.back`; (2) one tapped-hole orbit
bound into two bolted joints -- double use of a whole-orbit permanent
borrow (and physically wrong); (3) a duplicated `boundary:` key where
a zone-indexed value was meant. Good evidence the check placement is
right: all three are static-tier catches, and all three were realistic
authoring mistakes, not contrived ones.
