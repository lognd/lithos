## D150. Walk segments gain an optional leading name label (WO-51 escalation, blocker 1)

WO-51's Walk -> SketchClosure promotion was blocked by a recorded
scope note (`regolith-ir` sketch.rs): corpus walks name segments
only in COMMENTS (`line right  # a: base edge`) while `constraints:`
blocks already reference those names syntactically (`a.length =
80mm`) -- the binding does not exist at the syntax level, and WO-51
forbade itself a grammar change. Resolution: the walk surface opens
minimally. A walk step MAY carry a leading name label:

```
walk:
    from left_edge
    a: line right
    b: line up
    c: line left
    d: close
```

- Grammar: `walk-step = [ ident ":" ] rest-of-line` inside
  walk-block only; `hole` keeps its existing labeled-block form
  (the precedent this follows). Unlabeled steps stay legal.
- A `constraints:` line referencing a segment name that no label
  binds is a compile diagnostic (constructive: name the step lines
  and the label syntax) -- comment-only naming stops silently
  producing unbindable constraints.
- Argued against the mantras: Unambiguous (the name is now syntax,
  not a comment convention); Intent-based (labels read as the
  drafting callouts they already were); User-friendly (the corpus
  comments prove authors already want this spelling).
- Corpus sweep rides WO-51: every fixture whose `constraints:`
  reference segment names gets its comment names promoted to
  labels. hematite/02's walk section + grammar.ebnf update + track
  header version bump land in the same change.

## D151. flow_paths derive from the feature-op chain, honestly (WO-51 escalation, blockers 2-3)

D130 gives flow_paths a schema but no derivation. Resolution: the
FeatureProgram producer derives `flow_paths` from the feature-op
chain the `.cavity(inlet=..., outlet=...)` query touches -- never
from realized geometry, never by guessing:

- Segment order: the op-graph walk from the op carrying the inlet
  face to the op carrying the outlet face.
- Per-segment fields, each from a DECLARED source fact: role from
  the op kind; flow_area from the op's minimum section;
  elevation_change from the op axis against the part datum (0 with
  cited provenance when no datum orients gravity); roughness_class
  from the material/finish record the op cites; wall from the op's
  wall-thickness derivation where one exists.
- Any field with no declared source derives as HONESTLY
  INDETERMINATE (the AD-25 GeomExtract rule, verbatim) -- the
  producer never substitutes a plausible value. This extends D130's
  no-research-grade-guessing rule from the realizer to the
  producer.
- Fixture correction: WO-51 named `regen_chamber.hema` as the
  cavity exemplar; that fixture uses `Turn(profile=...)` + zones
  and no cavity query. The exemplars are the fixtures that DO use
  `.cavity(...)`: `examples/systems/regen_engine/injector.hema` and
  `examples/systems/dune_buggy/exhaust_intake.hema` (plus the
  xdomain pair). The WO file is amended, not the fixture bent to
  the WO.
