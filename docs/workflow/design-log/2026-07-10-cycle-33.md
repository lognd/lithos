## F114. Environment + release-build sweep (the owner's "debug my installations" pass)

Host facts (arm64 Ubuntu 22.04 jammy, KiCad 10 PPA active):

- `apt install ngspice` collides with the PPA's `libngspice-kicad`
  over `/usr/lib/aarch64-linux-gnu/ngspice/*.cm` (both package the
  XSPICE code models). Resolution: `-o Dpkg::Options::=
  "--force-overwrite"`. Caveat recorded: the CLI's v36 code models
  then shadow the PPA lib's v43 ones; if KiCad's internal simulator
  ever complains, `apt install --reinstall libngspice-kicad` with
  the same flag restores them.
- `calculix-ccx` has NO OCCT dependency; it only failed because it
  was requested in the same transaction as `gmsh` (whose jammy
  build needs OCCT 7.5, permanently broken by the PPA's 7.6).
  Installable alone, as-is.
- `gmsh` has no workable apt candidate on this host, no arm64 pip
  wheel, and no upstream ARM64 binary. conda-forge ships a
  linux-aarch64 build with its own OCCT; installed via micromamba
  (`~/micromamba/envs/fea`), exposed globally by symlink.
- `ghdl` 1.0.0 was already present and healthy.
- feldspar was ABSENT from the lithos venv (the sibling checkout
  exists but was never `uv pip install -e`-ed after an environment
  rebuild) -- every harness discharge silently deferred. Editable
  install restored it. This is a host-env fact, not a repo bug,
  but it motivates the `regolith doctor` support item (D191).

The repo bug the sweep found: `cargo build --release --workspace`
is CLEAN (zero warnings), but `regolith build --release` over all
16 flagship/system projects returned rc=1 with discharged=0 for
every one. Root cause chain (verified): the CLI `build` verb calls
`staged_build` with NO record search paths; `staged_build` does not
even accept frame/plan record paths (only `cost_record_paths`,
also never passed); the committed tests discharge the same claims
only because they pass `("stdlib",)` explicitly from the repo-root
CWD. So `[depends] "std.civil"` in a project manifest is dead
letter for CLI builds: timber_pavilion's `timber_sawn` family
(11 records, present in stdlib/) reports
`frame_section_family_not_landed`, its cost profile reports
`cost_record_unresolved`. -> WO-84.

Process finding, recurred: dispatched-agent worktrees were cut
from a stale base (a7dd660, one day and three cycles behind
master's 6b040e4). Standing rule reaffirmed: every dispatch
verifies `git merge-base HEAD master == git rev-parse master` as
step 0 and hard-resets to master before starting; the coordinator
re-verifies bases before trusting any diff.

## D192 (WO-84). Record-path resolution for CLI builds

Design: dependency resolution is magnetite's domain, so ONE
resolver in `python/regolith/magnetite/` maps a project's
`[depends] std.*` entries to record search-path roots, with
precedence (a) explicit config key (`regolith config` doctrine,
WO-59), (b) the project's vendored copies (vendor pins win),
(c) the development fallback (walk up from project root / package
location to a `stdlib/` containing `std.quantities`). A dependency
that resolves nowhere is NOT a resolve-time error: the existing
honest per-obligation deferrals already name the missing record.
`staged_build` gains `frame_record_paths`/`plan_record_paths`
mirroring `cost_record_paths`; every discharge-running CLI verb
(`build`, `ship`, `optimize`, `test`) threads all three. Lockfile
pin semantics (INV-22) unchanged. No SCHEMA_VERSION bump.
