## D194 (WO-85 escalations resolved). Load vocabulary: source grammar, aggregate expansion, embedment

The WO-85 dispatch verified (correctly, per its tripwires) that
three surfaces were undeclared and stopped. Rulings, argued
against the mantras (Unambiguous > Intent-based > User-friendly):

1. POINT/LINE LOAD SOURCE GRAMMAR (calcite/02 sec. 7): the one
   `loads:` row form stays `<case>: <qty> on [<targets>] by <ev>`
   -- NO new `at` keyword. Load KIND is derived from the quantity's
   unit dimension, which is unambiguous by construction: pressure
   (kPa) -> area; force/length (kN/m, N/m) -> line; force (kN) ->
   point; moment (kN-m) -> moment. A point load needs a location:
   the target grammar gains an optional station refinement,
   `member@<fraction>` (normalized 0..1 along the member axis), or
   the load targets a joint (no station). A force-unit load on a
   bare member target is a CONSTRUCTIVE diagnostic naming both
   spellings ("G1@0.5 or target joint J2") -- never inferred.
   Rationale: unit-dimension dispatch cannot collide (dimensions
   partition), reuses the landed row form, and the payload's
   `kind:` discriminator (calcite/03 sec. 4) stays the IR twin of
   a source fact rather than a new source keyword.
2. SCHEMA (the one 26->27 bump, D168): FrameLoad gains
   `LoadKind::Line` and `station: Option<f64>` (normalized;
   None for area/line/joint-point); the embedment obligation
   payload rides the same bump. Nothing else boards this train.
3. `.members.all` AGGREGATE (calcite/03 sec. 5): per-member
   OBLIGATION EXPANSION at Rust claims lowering -- one obligation
   per member with the member pinned in the subject, exactly the
   WO-68 forall-combo sweep precedent and consistent with sec. 5's
   singular "member id" obligation shape. The group spelling is
   sugar for the sweep; a group with one indeterminate member
   yields N-1 real verdicts plus one honest per-member deferral,
   never a wholesale defer and never a fabricated aggregate pass.
   Obligation-key/golden churn is accepted and regenerated.
4. `civil.embedment` (calcite/03 sec. 5): declared shape follows
   the `civil.bearing_pressure` reaction-based closed-form
   pattern: subject = the embedded member (post), comparator over
   required embedment depth from lateral/moment demand at grade
   (nonconstrained-earth closed form, IBC 1807.3-family cite in
   the model record) vs the declared depth. Vocabulary word
   already reserved in calcite/02 sec. 9.

The re-dispatched WO-85 applies these as spec-text deltas to
calcite/02 sec. 7 and calcite/03 sec. 4-5 in the SAME change as
the implementation (track headers version-bumped), per the
house rule that spec and machinery land together.

## D195 (answers F116). Conformance windows: generic instantiation resolves the SPEC side only

Ruling on F116's owner question (generic-parameter instantiation
as a conformance window):

1. `impl HeaterDrive<watts=50W>` against `power: <= watts`
   RESOLVES THE SPEC SIDE: the interface's parametric bound plus
   the instantiation pin yields `spec_bound <= 50W`. It supplies
   NOTHING for the impl side -- an impl body that declares no
   bound has asserted nothing, and fabricating `impl_bound = 50W`
   would discharge `50 <= 50` vacuously, masking real
   indeterminacy (exactly the INV-13/26 violation the WO-92 agent
   refused). The both-sides law stands.
2. The impl side arrives ONLY via (a) an explicit re-declared
   bound in the impl body (landed, WO-26 D104) or (b) the
   realized-fact channel when a realizer emits the quantity (the
   recorded WO-22/24-family growth; unchanged by this ruling).
3. What lands NOW (WO-92 deliverable 1', the honest sharpening):
   when the spec side resolves but the impl side does not, the
   Rust extraction still emits `conformance_sense` + `spec_bound`
   (never a fabricated `impl_bound`), and translate defers with a
   NEW, distinct reason `conformance_impl_bound_missing` whose
   detail teaches: the resolved spec bound, the field name, and
   the two honest paths to discharge (declare an impl-body bound,
   or realize the quantity). Bindings with no scalar window on
   either side (module imports, geometric/role promises) keep the
   existing `conformance_windows_unresolved` reason. The census
   then separates "impl owes a bound" from "nothing scalar to
   compare" -- which is the information the fleet actually needs.
   No schema bump (text channel; a reason string is data).

## F116. WO-92 execution: conformance-window census refuted; the
##       comparator-after-call fix is fluorite-local

Executing WO-92 refuted its own deliverable-1 premise (and the F115
census that drafted it). The census assumed "the core never threads
the spec/impl comparator windows into `given.loads`, so every one
defers." That is false: the refinement-window extraction ALREADY
landed via WO-26 D104 (commit 3d96812, two days before this WO was
drafted). `regolith-lower::claims::conformance_windows` emits
`conformance_sense`/`spec_bound`/`impl_bound` end-to-end, and
`translate._translate_conformance` consumes them -- both unit-tested
and firing on the D104 "Seat" pattern (interface `y: <= 20`, impl body
re-declares `y: <= 14`).

The extraction fires on ZERO real corpus bindings because the corpus
NEVER uses that pattern. Every flagship conformance edge is one of
three genuinely-unbounded shapes, none of which is a both-scalar-bound
refinement pair:

1. Generic-parameter instantiation against a parametric promise:
   `interface HeaterDrive<watts: power>: promises: power: <= watts`
   bound by `impl HeaterDrive<watts=50W> for self = todo!`. The spec
   ceiling resolves (50W) but the impl body declares NO scalar field
   -- there is no second (realized) bound to compare. One-sided;
   honest defer.
2. Geometric/role promises: `RailMount` promises
   `rail: planar, rect(>= len x 8mm)` -- a geometric role bound, not a
   scalar comparator field.
3. Whole-module `import` and `select` edges: no scalar refinement
   window exists for a module-vs-module or choice-among-candidates
   conformance.

Baseline == post-change: printer_k1 54, uav_talon 20
`conformance_windows_unresolved` -- all genuinely unbounded. Threading
a window here (e.g. treating a generic instantiation `watts=50W` as
BOTH the spec ceiling and the impl bound, yielding a vacuous 50<=50
discharge) would fabricate a verdict the source never stated and mask
the real indeterminacy (the realization's actual power is unmeasured).
That violates INV-13/26 and the WO's own "never invent a window" law,
so deliverable 1 is ESCALATED, not coded: the honest fleet-discharge
path for these obligations is the realized-fact channel (WO-42) once a
realizer measures the lower side, plus corpus authors adding explicit
scalar refinement fields where a real narrowing exists -- NOT a
compiler-invented window. OPEN for owner: should generic-parameter
instantiation against a parametric promise count as a conformance
window at all, and if so with what second bound? No reopen without

   net/instance/port extraction that is WO-24/WO-35 realizer-grade and
   unbuilt. This is not a leaf inside WO-87's stated Rust-lower scope;
   it is a prerequisite structural layer.

2. **The "un-block jlc_2l/decoupling, no special-casing" criterion
   forces `EntityKind::Net`, not a board-domain shortcut.**
   `stdlib/std.elec.patterns/decoupling.cupr` (the pack jlc_2l's gap

3. **The rule-eval registry dereference (deliverable 2) needs
   registry records inside the Rust rule engine, and no seam feeds

   payload shape) the WO does not make. Classifying an instance into

   sec. 3) -- so records-in-Rust is load-bearing for deliverable 1

- Python (magnetite `RecordStore`, the elec realizer bridge already
  imports it) resolves the board's `[depends]` record slice the packs
  cite and feeds it as `RealizedInput{ kind: "registry.records",

  check gains the domains' field lists in the same table.
