## D192 (WO-84). Record-path resolution for CLI builds

Design: dependency resolution is magnetite's domain, so ONE
resolver in `python/regolith/magnetite/` maps a project's
`[depends] std.*` entries to record search-path roots, with
precedence (a) explicit config key (`regolith config` doctrine,
WO-59), (b) the project's vendored copies (vendor pins win),
(c) the development fallback (walk up from project root / package
location to a `stdlib/` containing `std.quantities`). A dependency
that resolves nowhere is NOT a resolve-time error: the existing
honest per-obligation deferrals already name the missing record.
`staged_build` gains `frame_record_paths`/`plan_record_paths`
mirroring `cost_record_paths`; every discharge-running CLI verb
(`build`, `ship`, `optimize`, `test`) threads all three. Lockfile
pin semantics (INV-22) unchanged. No SCHEMA_VERSION bump.

## D193. magnetite stays in-wheel; a future index SERVER is an apps/ application

Owner floated moving the "package index" magnetite into apps/
beside graphite. Assessed and decided AGAINST for the client, with
a recorded home for the server half:

- magnetite-the-CLIENT (`regolith.magnetite`: manifest parsing,
  dependency/record resolution, vendoring, trust floors, scaffold)
  is load-bearing INSIDE the wheel: thirteen modules import it
  across orchestrator (costing/discharge/frame_resolve/loop/
  orchestrate/plan_staging), backends (ship/manifest/drawings
  attest), harness attest, the elec realizer bridge, and the CLI.
  Lockfile pins (INV-22) and evidence trust (INV-28) flow through
  it. Moving it to apps/ would invert the AD-31 seam (apps consume
  regolith's public surface; the wheel never imports apps/) and
  would make `regolith build` depend on an out-of-wheel install --
  breaking the two-command demo and the corpus tests. Argued
  against the mantras: Unambiguous loses nothing by the status quo;
  the move adds a cyclic-boundary ambiguity.
- magnetite-the-INDEX-SERVER (registry hosting) remains out of
  client scope (regolith/11 sec. 10 stands). WHEN it materializes
  it lands as an apps/ application (apps/ is exactly the graphite
  precedent: out-of-wheel, consumes public surfaces, own install
  target). That is the half of the owner's instinct this decision
  adopts.
- Reopen criterion: a real second consumer of the registry-client
  protocol outside the wheel (e.g. a CI mirror tool) would justify
  extracting a shared client PACKAGE (own wheel, not apps/) -- a
  different move than apps/, recorded here so it is not confused
  with this decision.

## D194 (WO-85 escalations resolved). Load vocabulary: source grammar, aggregate expansion, embedment

The WO-85 dispatch verified (correctly, per its tripwires) that
three surfaces were undeclared and stopped. Rulings, argued
against the mantras (Unambiguous > Intent-based > User-friendly):

1. POINT/LINE LOAD SOURCE GRAMMAR (calcite/02 sec. 7): the one
   `loads:` row form stays `<case>: <qty> on [<targets>] by <ev>`
   -- NO new `at` keyword. Load KIND is derived from the quantity's
   unit dimension, which is unambiguous by construction: pressure
   (kPa) -> area; force/length (kN/m, N/m) -> line; force (kN) ->
   point; moment (kN-m) -> moment. A point load needs a location:
   the target grammar gains an optional station refinement,
   `member@<fraction>` (normalized 0..1 along the member axis), or
   the load targets a joint (no station). A force-unit load on a
   bare member target is a CONSTRUCTIVE diagnostic naming both
   spellings ("G1@0.5 or target joint J2") -- never inferred.
   Rationale: unit-dimension dispatch cannot collide (dimensions
   partition), reuses the landed row form, and the payload's
   `kind:` discriminator (calcite/03 sec. 4) stays the IR twin of
   a source fact rather than a new source keyword.
2. SCHEMA (the one 26->27 bump, D168): FrameLoad gains
   `LoadKind::Line` and `station: Option<f64>` (normalized;
   None for area/line/joint-point); the embedment obligation
   payload rides the same bump. Nothing else boards this train.
3. `.members.all` AGGREGATE (calcite/03 sec. 5): per-member
   OBLIGATION EXPANSION at Rust claims lowering -- one obligation
   per member with the member pinned in the subject, exactly the
   WO-68 forall-combo sweep precedent and consistent with sec. 5's
   singular "member id" obligation shape. The group spelling is
   sugar for the sweep; a group with one indeterminate member
   yields N-1 real verdicts plus one honest per-member deferral,
   never a wholesale defer and never a fabricated aggregate pass.
   Obligation-key/golden churn is accepted and regenerated.
4. `civil.embedment` (calcite/03 sec. 5): declared shape follows
   the `civil.bearing_pressure` reaction-based closed-form
   pattern: subject = the embedded member (post), comparator over
   required embedment depth from lateral/moment demand at grade
   (nonconstrained-earth closed form, IBC 1807.3-family cite in
   the model record) vs the declared depth. Vocabulary word
   already reserved in calcite/02 sec. 9.

The re-dispatched WO-85 applies these as spec-text deltas to
calcite/02 sec. 7 and calcite/03 sec. 4-5 in the SAME change as
the implementation (track headers version-bumped), per the
house rule that spec and machinery land together.
